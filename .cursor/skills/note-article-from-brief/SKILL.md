# note 記事化（ブリーフ → 本稿）

`reports/note-briefs/` の**報告資料**（`note-deep-dive-report` 系テンプレに沿った Markdown）を入力として、**note 掲載想定の本稿**（`note/20######/` 配下の Markdown）を書く。ブリーフは編集者向けの箇条書き・表が多いので、**そのまま貼り替えず**、読者向けの語り口・見出し階層に再構成する。

## いつ使うか

- ユーザーが **`reports/note-briefs/YYYY-MM-DD-*.md` を @ 参照**し、「このブリーフから note 記事を書いて」「本文化して」と依頼したとき
- 深掘りコマンド（例: `overseas-it-news-deep-dive`）の**直後**に、同じトピックを note に載せるとき

## 入力

1. **必須:** ブリーフファイルのリポジトリ相対パス（ユーザーが添付または明示）。
2. **任意:** 保存先パス、シリーズ名（既定: `ITニュース`）、想定文字量、既存 note とのリンク（「前提稿」など）。

ブリーフに無い主張は**足さない**。不足は本稿では「未確認」「執筆時点の一次ソースでは明記されていない」など**読者向けの中立表現**にし、**ブリーフという語や `reports/note-briefs/` パスは出さない**。

## 一次確認（任意・短時間）

ブリーフの **「4. note 執筆メモ」** に **検証が必要な一点**がある場合:

- **ユーザーが明示したとき**、または **日付・型番・API 名など硬い事実**が本稿の要になるときだけ、Web で**該当点のみ**照合する。
- 照合で変わった場合は本稿に反映し、必要なら「ドキュメントの更新後に要確認」など**中立の注記**にとどめる（**ブリーフ名・パス・「ブリーフと異なる」等のメタは本稿に書かない**。リポジトリ側のブリーフファイルは自動では書き換えない）。

## 出力（保存先）

| 項目 | 既定 |
|------|------|
| **フォルダ** | ブリーフ先頭の **調査日**（`- **調査日**:`）の `YYYY-MM-DD` から `note/YYYYMMDD/`（ゼロ埋め8桁） |
| **ファイル名** | `YYYY-MM-DD-<英語スラッグ>.md`。スラッグはブリーフファイル名の日付以降を踏襲してよい（例: `2026-04-15-openai-gpt-5-4-cyber-tac.md` → `2026-04-15-openai-gpt-5-4-cyber-tac.md` または短縮スラッグ）。**読みやすい副題**を付ける場合は `2026-04-15-slug｜日本語副題.md`（リポジトリ既存例に合わせる） |
| **ユーザー指定パス** | 指定があればそこに保存 |

フォルダが無ければ作成してよい。

## 本稿の構成（推奨）

既存の `note/` 本編（例: 政策・報道の切り分け稿）に寄せる。

1. **タイトル行（H1）**  
   `# 「〈シリーズ名〉」｜〈記事独自のキャッチ〉`  
   ブリーフの H1 題名をそのまま使わず、**読者向けに短く言い換える**。

2. **はじめに（リード）**  
   - ブリーフ **要約（1文）** を骨に、**誰の発表／どの媒体の何日**を最初の段落で明示。  
   - **報道と一次（公式）**を混ぜない。報道ベースなら「〜と報じられています」「〜とも書かれています」など**帰属**を付ける。  
   - 本稿の**読みどころ**（1〜2点）を自然な文で示す。  
   - **略語・英語が3つ以上続く見込み**のときは、リードの直後に **「この記事での用語」** として小さな表（語数 **3〜8**、各1行説明）を置く。以降の本文では **和語または短い説明を先に、括弧で正式名・略語**（例: 試験導入（いわゆる PoC））とする。

### 平易さ（用語・アクセス）

「誰でも」より現実的な既定読者は **ITニュースは読むが、セキュリティ職・調達の専門家ではない人**。次を満たすこと。

| ルール | 内容 |
|--------|------|
| **初出** | 専門語・略語は **初回だけ** 平易語＋括弧で正式名。2回目以降は略語のみ可。 |
| **1文1概念** | 1文に **固有名＋略語＋英単語**を詰め込まない。長いら分割するか表に逃がす。 |
| **和名優先** | 「レイヤ」→「**情報の出どころが違う**（報道／公式など）」のように、**読み替えを一度は文中に出す**。以降「レイヤ」単独に依存しすぎない。 |
| **英語のまま残す** | 公式・報道の引用として英語が必要なときは残し、その**直後か括弧内**で日本語を添える。 |
| **表の列** | 想定読者向けの列名は **現場語より「何をする人」** に寄せてよい（例: 「本稿からの実務メモ」）。 |
| **置き換えない** | 契約・法務・一次の断定に効く語は **ぼかさず**、前後に**一文の説明**を足す。 |

難易度を下げることと、**一次と報道の切り分け**は両立する（平易さのために帰属を削らない）。

3. **本編（H2 / H3）**  
   ブリーフの **1. 背景** → 時系列・「なぜ今」を読者向けに段落化。**2. 目的** の表は、必要なら **表のまま** または **箇条書き＋解説** に変換。**3. 効果** は時間軸の定義を冒頭で再掲し、各期の「予想／不確実性／効果が出ない条件」を**冗長に繰り返さず**整理する。

4. **混同しやすい点・チェックリスト（任意）**  
   ブリーフやトピックに応じて、2列表「論点 / 注意」を入れると既存 note と整合しやすい。

5. **まとめ（短い H2）**  
   行動指針は**押し付けず**、ブリーフの実務示唆を一文ずつ程度に圧縮。

6. **免責・注記**  
   ブリーフ **4. note 執筆メモ** の免責に加え、投資・医療・**無許可の侵入テスト**など該当する注意を**はじめに直後**または**末尾**に置く（既存稿の置き方に合わせる）。

7. **参照（任意・簡潔）**  
   ブリーフ **5. 参照一覧** をすべて複写しなくてよい。本稿で用いたリンクを本文に埋め込み、残りは短い「主な参照」リストでよい。

## 本稿に書かないこと（必須）

**読者向けの note 本稿には、ブリーフや作成プロセスへの言及を一切入れない。**

含めない例（列挙は代表的なもの）:

- 「ブリーフ」「報告資料」「執筆前ブリーフ」「下調べメモ」「ブリーフに基づく」「ブリーフ時点」「ブリーフでは整理」「ブリーフ注釈」などの語句
- リポジトリパス **`reports/note-briefs/`** やブリーフファイル名へのリンク・脚注・更新履歴での言及
- *企画・下調べメモ: …* のような、調査用ファイルを指すフッター（読者には不要なメタ）

事実の出典は **Reuters / 公式ブログ / 当局** など、**公開一次・準一次の URL と日付**に帰属させる。執筆プロセスはチャットやリポジトリのコミットログ側に留める。

## 文体・表記

- **日本語**本体。英語固有名は初出で必要なら短い説明。
- ユーザー向け本文に **「§」記号は使わない**（見出し番号は `## 1.` の形式でよい）。
- 推測は「〜の可能性」「〜と読める」に限定し、**根拠のない動機の断定**をしない（ブリーフ原稿の「補足・推測」と**同じ厳しさ**を本稿の内容に反映するが、**ブリーフという語は本稿に出さない**）。
- 数値・日付は**入力ブリーフで既に出典付きのもの**（＝検証済みの事実）を本稿に引き継ぐ。新たな数値を Web で足す場合は必ず出典を付ける。

## 品質チェック（出力前）

- [ ] ブリーフの事実・出典レイヤを崩していない（報道と公式を混同していない）
- [ ] H1 が読者向けキャッチになっている（「報告資料:」のままになっていない）
- [ ] **ブリーフ・報告資料・`reports/note-briefs/`・「〜に基づく初稿」等が本稿に1語も残っていない**
- [ ] 背景・目的・効果のうち、**読者に不要なメタ**（執筆工程の説明）は削っているか置き換えている
- [ ] **略語だらけの段落が続いていない**（「この記事での用語」表または初出ルールで手当て済み）
- [ ] 免責・検証保留が適切
- [ ] 保存パスが `note/YYYYMMDD/` 規則に沿っている（またはユーザー指定どおり）

# note 向け深掘り・報告資料スキル

調査結果（例: `reports/overseas-it-news/news-log.md`、チャット要約、単発トピック）を入力として、**note 記事執筆前の報告資料**を作る。必ず **背景 / 目的 / 効果** の3軸で整理し、推測と事実を分離する。

## いつ使うか

- 海外ITニュースやログの**1トピック／複数トピック**を note 化したい
- 編集者・自分用の**ブリーフ（企画根拠）**が欲しい
- 「なぜ今」「誰に何をしてほしい」「何が起きうるか」を**執筆前に言語化**したい

## 作業フロー

1. **入力の特定**  
   ユーザーが指すファイル・URL・箇条書きを読む。未指定ならリポジトリ内の最新調査ログを提案してから進める。

2. **深掘りのモード選択**  
   - **ライト**: 単一エージェントで Web 検索のみ（トピックが単純、一次が1〜2本で足りる）。  
   - **フル**: 下記 **サブエージェント** を使う（複合トピック、規制・市場・技術が同時に絡む、ユーザーが「出典以外も」と言った場合、高リスク題材）。

3. **事実の再確認（深掘り）**  
   ライトの場合: Web 検索で公式を優先し、少なくとも次を埋める:  
   - **いつ**: 公表日・施行日・締切・インシデント検知日  
   - **誰が**: 発表主体（企業 / 政府 / 研究者 / メディア独自）  
   - **何を**: 技術的・政策的に変わる点（1〜3行で要約可能か）  
   - **出典**: 主たるURL（可能なら一次）  

   フルの場合: **サブエージェント**の結果をマージしてから同じ項目を埋める（下記）。

4. **3軸で記述**（下記テンプレに沿う。推測は「予想」「可能性」と明記）

5. **出力**  
   ユーザーが指定したらそのパスに Markdown を保存。未指定ならチャットに全文＋保存先パスを提案（例: `reports/note-briefs/YYYY-MM-DD-slug.md`）。フル調査時は **各サブエージェントの出力見出し**を「参照・根拠」に要約して残すか、別ファイル `*-sources.md` に付録として保存してよい（ユーザー希望に従う）。

## サブエージェント（調査レーン）

定義ファイルは [subagents/README.md](subagents/README.md) と各レーンの `.md`。親エージェントは **Task ツール**で並列起動する（`subagent_type`: `explore` または `generalPurpose`、`readonly: true` 推奨）。

| ID | レーン | 定義ファイル |
|----|--------|----------------|
| A | 一次・公式 | [subagents/primary-official.md](subagents/primary-official.md) |
| B | 規制・政策 | [subagents/regulatory-policy.md](subagents/regulatory-policy.md) |
| C | 市場・資本 | [subagents/market-capital.md](subagents/market-capital.md) |
| D | 技術・サプライ | [subagents/technical-chain.md](subagents/technical-chain.md) |
| E | 文脈・履歴 | [subagents/context-narrative.md](subagents/context-narrative.md) |

### 起動手順（親）

1. 対象 README の **「いつどれを起動するか」** に従い、起動するレーンを決める。**A は原則必須**。  
2. 各レーンの **「Task 用プロンプト」** コードブロックを開き、トピック・URL・法域など空欄を埋める。  
3. **同じユーザーターンで** Task を複数回呼び出し、**並列**で走らせる（可能なら）。  
4. 返却された `### 調査結果: ...` ブロックを収集し、README の **マージ順**（A→B/C/D→E）で矛盾を解消。  
5. マージ結果をもとに **背景 / 目的 / 効果** の報告資料を書く。

### フォールバック

Task が使えない環境では、親が **各レーンの Task 用プロンプトを自分で順に実行**（検索をセルフで行い、同じ出力見出しでまとめる）。

## 背景（なぜ今この情報が出たか）

次の観点から**該当するものだけ**簡潔に書く。該当なしは「明示的な動機は公開情報から読み取れず」と書く。

| 観点 | 例 |
|------|-----|
| 期限・施行 | 法施行日、パッチ締切、四半期・決算 |
| インシデント | 悪用実績、漏えい、サプライチェーン侵害 |
| 競合・市場 | 他社発表への対抗、シェア、価格 |
| 政治・規制 | 制裁、輸出管理、認証制度、選挙・議会 |
| 技術的成熟 | 標準化、RFC、製品GA、論文・カンファレンス |
| メディア要因 | リーク、FOIA、大型カンファレンス集中 |

**禁止**: 根拠のない陰謀論・動機の断定。推測は「〜の可能性」とし根拠を併記。

## 目的（誰に、何をしてほしいか）

| 項目 | 書く内容 |
|------|-----------|
| **想定読者セグメント** | 例: セキュリティ担当、経営、開発者、一般ユーザー、投資家、政策関係者 |
| **発信側の意図（推定）** | 公式声明・アドバイザリから読み取れる行動喚起 |
| **具体的に望まれる行動** | パッチ適用、設定変更、調達見直し、バックアップ、開示・報告、製品導入 など |
| **読者が取るとよい行動（執筆者提案）** | note 読者向けに中立的・実務的に1〜3個 |

発信者が複数（ベンダ＋CISA 等）の場合は**行ごとに主体を分ける**。

## 効果（短期・中期・長期の予想）

**必ず時間軸を定義**してから書く（定義を資料に明記）。

- **短期**: 目安 **0〜3か月** — パッチ率、株価・信用イベント、ヘルプデスク負荷、誤情報の拡散 など
- **中期**: **3か月〜1年** — 調達・製品ラインナップ、クラウドコスト、人材スキル、中小の追随
- **長期**: **1年以上** — 規制の定着、標準の変化、産業構造、地政学

各項目で:

- **予想される効果**（箇条書き）
- **不確実性**（データ不足、政治・訴訟、技術転換）
- **効果が出ない条件**（対策が普及しない、別CVEが優先 等）

数値は**出典付き**のみ。推定はレンジと前提を書く。

## 報告資料テンプレート（そのまま出力に使う）

```markdown
# 報告資料: [トピック短題]

- **調査日**: YYYY-MM-DD
- **主出典**: [媒体・機関名](URL)
- **要約（1文）**:

---

## 1. 背景 — なぜ今この情報が出ているか

- **事実（時系列）**:
  - …
- **公開情報から読み取れる動機**:
  - …
- **補足・推測（根拠付き）**:
  - …

## 2. 目的 — 誰に、何をしてほしいか

| 想定読者 | 発信側が望む行動（推定） | note 読者への実務的示唆 |
|----------|---------------------------|-------------------------|
| … | … | … |

## 3. 効果 — 短期・中期・長期の予想

- **短期（0〜3か月）**: …  
  - 不確実性: …
- **中期（3か月〜1年）**: …  
  - 不確実性: …
- **長期（1年以上）**: …  
  - 不確実性: …

## 4. note 執筆メモ（任意）

- **フック案**（1〜2文）
- **検証が必要な一点**（事実・数値・日付）
- **免責・注意**（投資・医療・法務では専門家相談など、該当すれば）

## 5. 参照一覧

- …
```

## 品質チェック（出力前に自己確認）

- [ ] 背景に「なぜ今」が**事実と推測**に分かれている
- [ ] 目的に**行動**が具体的に書いてある（曖昧な「注意喚起のみ」で終わっていない）
- [ ] 効果に**時間軸の定義**と**不確実性**がある
- [ ] note 向けに**誤解しやすい点**が1つ以上指摘されている

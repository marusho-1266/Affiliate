# 実務ITアラート調査・報告（期限・国内公式・行動優先）

**海外速報（`overseas-it-news`）と別ストリーム。** 読者の**確認・計画・優先順位**に直結するトピックを調査し、日本語で `alerts-log.md` に追記する。情報源は**国内公式・ベンダ日本語ページを含めてよい**（英語圏のみに限定しない）。

## 出力ファイル（リポジトリ内）

| ファイル | 用途 |
|----------|------|
| `reports/practical-it-alerts/alerts-log.md` | **累積レポート**（時系列で追記） |
| `reports/practical-it-alerts/seen-urls.json` | **既出の主出典URL**（重複排除） |
| `reports/practical-it-alerts/alerts-policy.md` | **採用ポリシー**（実行前に読む） |

`seen-urls.json` は `{"urls": ["https://...", ...]}` 形式（配列はユニーク・昇順推奨）。

## `overseas-it-news` との違い（必読）

- **鮮度:** 本コマンドは **`news-recency-policy.md` の D−1 ルールを適用しない**。代わりに **`alerts-policy.md`**（期限の近さ・公式更新の実務価値）に従う。
- **ログ:** `news-log.md` には書かない。必ず **`alerts-log.md`** に追記する。

## note/ 底稿とのフォロー（任意だが推奨）

リポジトリ内 **`note/**`（`note/Druft/` 除く）**で、**期限・手順・確認コマンド**を主題にした底稿がある場合、**公式の新しい更新や期限明確化**がないか調べる。

- **該当があれば** `### note底稿のフォロー` を **`### トピック` の直後**に置く（1件以上あるときのみ見出しを書く）。
- **行フォーマット:** `**底稿:** \`note/...\`。**フォロー出典:** [名](URL)（**いつ・何が更新されたか**を括弧で短く）`

### 対象範囲

- ユーザー指定がなければ **直近 8 週間分の `note/20######/`** を目安にする。負荷で狭めたら `### メモ（任意）` に1行。

## 重複排除（必ず実行前に）

1. **`seen-urls.json`** の `urls` を読む。
2. **URL 正規化**（比較用）:
   - `http` → `https`（可能なら）
   - 末尾 `/` を除去
   - フラグメント `#...` と追跡用クエリ（`utm_*` 等）を除去してよい
   - ホスト名は小文字
3. 採用する各トピックの**主出典 URL 1 本**を決め、正規化後が `urls` に**既に含まれる場合はスキップ**（同一事件の別URLのみなら内容で判断し、既出ならスキップ）。
4. 残りを `alerts-log.md` に追記し、使った正規化後 URL をすべて `seen-urls.json` に**マージ**（重複なし・昇順推奨）。

## 調査手順

1. **`alerts-policy.md`・`seen-urls.json`・`alerts-log.md`（末尾）**を読む。
2. **（任意）** `note/` の期限・実務系底稿を列挙し、フォロー調査を実施。採用行と URL を確定（この時点では `seen-urls.json` 未保存でもよい。最後にまとめてマージ）。
3. **Web検索**で候補を集める。例（組み合わせて実行）:
   - 日本語: `セキュアブート 2026` / `UEFI 証明書 期限` / `Windows サポート終了` / `IPA 注意喚起`
   - 英語: `certificate expiry UEFI` / `product lifecycle end of support` / `vendor security advisory deadline`
   - ベンダ: Microsoft Learn / Google Chrome Enterprise / Apple サポート等の**公式**を優先
4. **`alerts-policy.md` に照らし**採用を確定。重複排除を通す。
5. **`alerts-log.md` に1ブロック追記**（下記フォーマット）。**`seen-urls.json` を保存**（新規トピック＋ noteフォローで採用した主出典 URL をマージ）。

**トピック0件・フォローも0件**のときも、`## 調査 …` を1つ追加し、「該当なし（既出のみ／フォローなし）」と書く。

## ログ追記フォーマット（`alerts-log.md` の末尾に追加）

```markdown
## 調査 YYYY-MM-DD HH:MM （JST）

調査の前提（任意・1〜2行）: 今回のフォーカス（例: 証明書期限、特定ベンダの EOS 周知）を短く。

### トピック

- **件名** — 要約（日本語）。**読者アクション**（確認／更新／計画）を一文で。**期限・根拠日:** …。出典: [表示名](正規化前URL)

（カテゴリ分けする場合は `#### OS・ファーム` などの下に箇条書き）

### note底稿のフォロー

（**1件以上あるときのみ**この見出しを置く）

- **〈件名〉** — 要約。**底稿:** `note/YYYYMMDD/…md`。**フォロー出典:** [名](URL)（更新日など）

### メモ（任意）

- 調査対象の `note/` 範囲、見送り理由、次回フォロー予定など
```

## チャットでの返答

- **`### トピック`** の要約（ログと整合）
- **`### note底稿のフォロー`** を書いた場合は底稿パスと URL を明示
- **保存先:** `reports/practical-it-alerts/alerts-log.md` と `seen-urls.json`

## ツール

- **Web検索**を主に使う。根拠が要るページは取得ツールで確認する。
- **`note/`** は Glob／Read で列挙・参照する。

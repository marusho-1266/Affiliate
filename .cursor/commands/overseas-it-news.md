# 海外・日本 ITニュース調査・報告（実行時点）

実行時点の最新の **IT・テックニュース**を、**海外（主に英語圏）と日本国内の両方**から調査し、日本語で報告する。国内は **官公庁・独立行政法人の公式発表**、**大手報道**、**業界・セキュリティ専門メディア**、**上場企業の適時開示・ニュースリリース（IR）** など、公開ウェブで確認できるものを対象とする。あわせてリポジトリ内 **`note/` 底稿**ごとに**続報「有」**がないか調べ、該当がある場合は同一調査ブロックの **`### note底稿の続報`** として `news-log.md` に記載する。結果は **Markdown ファイルに追記**し、**過去実行とURL単位で重複しない**ようにする。

**別ストリームとの分界:** **期限・EOS・更新期限・国内実務アラート**を「一覧運用」する用途は引き続き **`practical-it-alerts`**（`reports/practical-it-alerts/*`）。本コマンドでは **ニュース・政策・企業動向・インシデント開示**として採るかどうかを **`news-recency-policy.md`** と同日誌の **`### メモ（任意）`** で整理する。

## 出力ファイル（リポジトリ内）

| ファイル | 用途 |
|----------|------|
| `reports/overseas-it-news/news-log.md` | 本コマンドの**累積レポート**（時系列で追記） |
| `reports/overseas-it-news/seen-urls.json` | **既出の出典URL**（重複排除用。本コマンドと brief 共通） |
| `reports/overseas-it-news/news-recency-policy.md` | **掲載日（鮮度）ルール**（新規トピック採用時に**必ず遵守**。実行前に読む） |
| `reports/practical-it-alerts/*` | **別ストリーム:** 期限・EOS・実務アラートの**一覧運用**は `practical-it-alerts`（**ニュース採用・鮮度ルールは本コマンドと共通しうる**が、混線しないよう用途で切り分ける） |
| `reports/note-articles-follow-up-*.md`（任意） | **note 続報の整理表**（有／薄／無の一覧。無くても本コマンドは実行可。追記・別名でもよい） |

初回のみ、上記が無ければ作成する。`seen-urls.json` は `{"urls": ["https://...", ...]}` 形式（配列はユニーク・昇順推奨）。

## note/ 底稿の続報調査（本コマンドと**併せて**実施）

リポジトリ内 **`note/`** に置いた note 用 Markdown（**底稿**）について、各記事が扱うトピックに**執筆日以降の続報**がないか調べる。**続報が「有」に相当するものだけ**を `news-log.md` に載せる（下記判定）。

### 対象範囲

- 原則: `note/**/*.md` のうち、**`note/Druft/` を除く**（ユーザーが Druft も含めよと指示した場合のみ含める）。
- **日付範囲**はユーザー指定を優先（例: `2026-03-27` 以降のみ）。未指定のときは **直近 8 週間分の `note/20######/` フォルダ**を対象とし、負荷の都合で範囲を狭めた場合はチャットと `news-log.md` のメモにその旨を1行書く。
- **ラジオ台本**（`*ラジオ台本.md`）と**本編が対**のときは**本編のみ**を調査対象にしてよい（重複調査を避ける）。

### 「有」「薄」「無」（`news-log.md` への載せ分け）

| 判定 | 目安 | `news-log.md` |
|:---:|:---|:---|
| **有** | 底稿公開後に**新事実・公式更新・別稿で論点が進んだ**一次／準一次が取れる | **`### note底稿の続報`** に**必ず1行以上**追記する |
| **薄** | **再掲・論評・同週の整理記事**のみで、主出典が既ログと同一事件に近い | **新規トピック／note底稿の続報には載せない**（必要なら `### メモ（任意）` に「note続報・薄: …」と短く） |
| **無** | 該当なし | `news-log.md` には**書かない**（調査したことは `### メモ（任意）` に「note続報: 該当範囲で無」と任意） |

### 手順（ウェブ調査の前後）

1. 対象となる底稿ファイルを列挙し、各ファイルの**主トピック**（見出し・一次リンク・日付）を把握する。
2. トピックごとに公開ウェブで**底稿の日付以降**の続報を検索する。
3. **「有」**のトピックについて次を実施する。
   - **続報の主出典 URL 1 本**（必要なら補助 URL は要約文に追加）を決める。
   - **重複排除:** 正規化後 URL が `seen-urls.json` に**既にある**場合は、そのトピックは **`### note底稿の続報` には載せない**（同一事件済み）。**薄**だけの場合も載せない。
   - **鮮度:** 続報の主出典が **`news-recency-policy.md`** のとおり、調査日 **D** に対して **JST で D−1 以降**に公開（または Last updated）であることを確認する。**D−2 より前**の続報のみでは `### note底稿の続報` に**入れない**（メモや別資料に回す）。
4. **採用 URL の確定:** `### note底稿の続報` に載せる**各続報の主出典 URL**（正規化後）を列挙する。**`seen-urls.json` への書き込みは、本章「調査手順」の手順8**で、新規トピックの主出典と**まとめて**マージする（重複除去・昇順推奨）。
5. **一覧表の更新（任意）:** 有／薄／無の全体表を維持する場合は `reports/note-articles-follow-up-*.md` を新規作成または更新してよい（本コマンドの必須成果物ではない）。

### `news-log.md` への追記位置・フォーマット

- 同一実行の **`## 調査 YYYY-MM-DD HH:MM （TZ）` ブロック**内で、`### 新規トピック` セクションの直後かつ `### メモ（任意）` より前に、次の見出しを**続報「有」が1件でもあるときのみ**追加する。**1件もないときは `### note底稿の続報` 見出し自体を書かない**。

```markdown
### note底稿の続報

- **〈続報の一行件名〉** — 要約（日本語）。**底稿:** `note/YYYYMMDD/ファイル名.md`。**続報出典:** [媒体名](正規化前の元URL)（ページ上の公開日または Last updated を括弧に明記できるなら明記）
```

- **複数底稿が同一続報 URL に収束する場合:** 行を1つにまとめ、底稿を「`…` / `…`」のように並記してよい。
- **底稿はリポジトリ相対パス**で書く（note.com の URL ではなく、編集・再実行のしやすさ優先）。

## 重複排除（必ず実行前に）

1. **`seen-urls.json` を読む**（`urls` 配列を取得）。
2. **URL の正規化**して比較する（比較用）:
   - `http` → `https` に揃える（可能なら）
   - **末尾 `/` を除去**
   - **フラグメント `#...` とクエリのうち追跡系**（`utm_*` 等）は除去してよい
   - ホスト名は**小文字**に揃える
3. 調査で採用する各トピックの**主たる出典URL 1本**を決め、正規化後の文字列が `urls` に**既に含まれる場合はそのトピックはスキップ**（別メディアの二次報道のみの場合は、ログにまだ無いURLでも**同一事件**なら既出扱いでスキップする）。
4. **残ったトピックだけ**を `news-log.md` に追記し、使った正規化後URLをすべて `seen-urls.json` の `urls` に**マージ**して保存する（重複は入れない）。

**`### 新規トピック` が0件**かつ **`### note底稿の続報` も0件**のときも、`news-log.md` に `## 調査 YYYY-MM-DD HH:MM TZ` 見出しを1つ追加し、本文に「新規出典なし（いずれも `seen-urls.json` と同一事件・同一URL済み）／note 続報も該当なし」と書く。

**`### 新規トピック` は0件だが `### note底稿の続報` に1件以上ある**ときは、見出しブロックに **`### note底稿の続報` のみ**（またはメモ併記）を書き、「新規トピックなし」と明記してよい。

## 掲載日（鮮度）— `news-recency-policy.md` 必須

1. 実行開始時に **`reports/overseas-it-news/news-recency-policy.md` を読む**。
2. 調査見出しの日付を **D**（JST の暦日）とする。`### 新規トピック` および **`### note底稿の続報`** に入れる**各主出典**について、**公式等に記載の公開日（または Last updated）**が **JST で `D − 1` 以降**であることを確認する。**`D − 2` より前**のソースは当該セクションに**採用しない**（当日 `D` 公開は可）。境界・日付不明の扱いはポリシー本文に従う。
3. 上記を満たさないが参照価値がある場合は **`### メモ（任意）`** や別資料に回し、新規トピックに混ぜない。

## 前提

- **実行日時**は、ユーザーのメッセージに含まれる場合はそれを優先する。なければ会話・システムで分かる「今日の日付」と、可能なら**タイムゾーン（例: JST）**を明記する。
- 情報源は**公開ウェブ**。**海外:** 公式ブログ、大手メディア、業界専門誌、ベンダアドバイザリなど。**日本国内:** 省庁・地方自治体・独立行政法人の公式ページ、経産省・デジタル庁・総務省・IPA・JPCERT/CC 系の公開、アクセスの取りやすい報道（日経・共同・時事など）、企業のプレスリリース／適時開示。**いずれも**推測で事実を補わない。

## 調査手順

1. **`seen-urls.json`・`news-recency-policy.md`・`news-log.md`（末尾数ブロック）を読む**（重複・鮮度・直近の書き方の参照）。
2. **`note/` 底稿の続報調査**（本章「note/ 底稿の続報調査」）を実施する。採用行と `seen-urls.json` 追加分を確定する（**この時点では `seen-urls.json` に書き込まず**、手順8 で新規トピック分とまとめてマージしてよい）。
3. **複数の検索**で**海外・日本**の IT／テック動向を網羅する（例: 次を**組み合わせ**て実行する）。
   - **海外（英語）:** `technology news` + 当日または直近の日付（年はユーザーコンテキストの「今日」に合わせる）
   - **海外:** `cybersecurity breach` / `CVE` + 直近
   - **海外:** `AI` / `cloud` / `semiconductor` + `news` + 直近
   - **海外:** 必要なら `EU regulation tech` / `SEC filing` など、ユーザーが関心を示した領域を追加
   - **日本（日本語）:** `セキュリティ サイバー攻撃` / `インシデント` + 当日または直近、`IPA 脆弱性` / `JPCERT` + 直近（公開日が鮮度内のもの）
   - **日本:** `デジタル庁` / `経産省` + `AI` または `サイバー` + 直近（プレス・告示ベース）
   - **日本:** `国内 クラウド` / `データセンター` + `ニュース` + 直近（大手経済紙・通信社の公開記事）
   - **日本:** ユーザーが関心を示した国内企業名・製品名 + `ニュース` / `発表`
4. 候補を**重複排除ルール**に通し、続けて **`news-recency-policy.md` の掲載日ルール**に通す（満たさないものは新規トピックから除外）。**手順2 で採用した続報 URL**がここで弾かれる場合は、`### note底稿の続報` からも**外す**。
5. 残りを重要度が高い順に並べる（セキュリティ実害、規制、大手M&A・資金調達、インフラ障害など）。
6. **各トピックに出典URL**を付ける（可能なら一次情報を優先）。
7. **注意書き**: 4月1日前後はエイプリルフールやジョーク記事が混ざりやすい。**単独の煽り見出し**は一次ソースで確認した旨を書く。
8. **`news-log.md` に1ブロック追記**する。フォーマットは下記「ログ追記フォーマット」。**手順2 の `### note底稿の続報`** を書く場合は **`### 新規トピック` の直後**に置く。続いて **`seen-urls.json` を保存**（新規トピックの主出典 URL と、手順2 で最終採用した note 続報 URL を**重複なく**マージ）。チャットへの返答も同じ構成でよい（新規分のみ要約すればよい）。


## ログ追記フォーマット（`news-log.md` の末尾に追加）

```markdown
## 調査 YYYY-MM-DD HH:MM （TZ）

### 新規トピック

- **件名** — 要約（日本語）。出典: [媒体名](正規化前の元URL)

（カテゴリ別にしたい場合は ### セキュリティ などに分割してよい）

### note底稿の続報

（**続報「有」が1件以上あるときのみ**この見出しを置く。0件なら見出しごと省略）

- **〈続報の一行件名〉** — 要約（日本語）。**底稿:** `note/YYYYMMDD/ファイル名.md`。**続報出典:** [媒体名](正規化前の元URL)

### メモ（任意）

- 省略した既知トピック数: N（任意）
- note 続報で「薄」のみだった底稿があれば短く列挙してよい（任意）
```

## チャットでの返答

- **新規トピック**の要約（ログに書いた内容と整合）
- **`### note底稿の続報`** を書いた場合は、その要約と**底稿パス**を1行ずつ示す
- **ファイル保存先**をパスで明示（少なくとも `reports/overseas-it-news/news-log.md` と `seen-urls.json`）
- **新規トピックも note 続報「有」もともに0件**ならその旨

## ツール

- **Web検索**を主に使う。ページ本文が必要なら取得ツールで要約・引用の根拠を補強する。
- **`seen-urls.json`・`news-log.md`・`news-recency-policy.md` の読み取りは必須**（重複排除と鮮度のため）。
- **`note/` 底稿**は Glob／Read で列挙・読取する（続報調査のため）。

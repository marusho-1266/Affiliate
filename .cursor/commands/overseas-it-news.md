# 海外ITニュース調査・報告（実行時点）

実行時点の最新の**海外（主に英語圏）IT・テックニュース**を調査し、日本語で報告する。結果は **Markdown ファイルに追記**し、**過去実行とURL単位で重複しない**ようにする。

## 出力ファイル（リポジトリ内）

| ファイル | 用途 |
|----------|------|
| `reports/overseas-it-news/news-log.md` | 本コマンドの**累積レポート**（時系列で追記） |
| `reports/overseas-it-news/seen-urls.json` | **既出の出典URL**（重複排除用。本コマンドと brief 共通） |
| `reports/overseas-it-news/news-recency-policy.md` | **掲載日（鮮度）ルール**（新規トピック採用時に**必ず遵守**。実行前に読む） |

初回のみ、上記が無ければ作成する。`seen-urls.json` は `{"urls": ["https://...", ...]}` 形式（配列はユニーク・昇順推奨）。

## 重複排除（必ず実行前に）

1. **`seen-urls.json` を読む**（`urls` 配列を取得）。
2. **URL の正規化**して比較する（比較用）:
   - `http` → `https` に揃える（可能なら）
   - **末尾 `/` を除去**
   - **フラグメント `#...` とクエリのうち追跡系**（`utm_*` 等）は除去してよい
   - ホスト名は**小文字**に揃える
3. 調査で採用する各トピックの**主たる出典URL 1本**を決め、正規化後の文字列が `urls` に**既に含まれる場合はそのトピックはスキップ**（別メディアの二次報道のみの場合は、ログにまだ無いURLでも**同一事件**なら既出扱いでスキップする）。
4. **残ったトピックだけ**を `news-log.md` に追記し、使った正規化後URLをすべて `seen-urls.json` の `urls` に**マージ**して保存する（重複は入れない）。

**新規トピックが0件**のときも、`news-log.md` に `## 調査 YYYY-MM-DD HH:MM TZ` 見出しを1つ追加し、本文に「新規出典なし（いずれも `seen-urls.json` と同一事件・同一URL済み）」と書く。

## 掲載日（鮮度）— `news-recency-policy.md` 必須

1. 実行開始時に **`reports/overseas-it-news/news-recency-policy.md` を読む**。
2. 調査見出しの日付を **D**（JST の暦日）とする。`### 新規トピック` に入れる各主出典について、**公式等に記載の公開日（または Last updated）**が **JST で `D − 1` 以降**であることを確認する。**`D − 2` より前**のソースは新規トピックに**採用しない**（当日 `D` 公開は可）。境界・日付不明の扱いはポリシー本文に従う。
3. 上記を満たさないが参照価値がある場合は **`### メモ（任意）`** や別資料に回し、新規トピックに混ぜない。

## 前提

- **実行日時**は、ユーザーのメッセージに含まれる場合はそれを優先する。なければ会話・システムで分かる「今日の日付」と、可能なら**タイムゾーン（例: JST）**を明記する。
- 情報源は**公開ウェブ**（公式ブログ、大手メディア、業界専門誌、ベンダアドバイザリなど）。推測で事実を補わない。

## 調査手順

1. **複数の検索**で網羅する（例: 次を組み合わせて実行する）。
   - `technology news` + 当日または直近の日付（年はユーザーコンテキストの「今日」に合わせる）
   - `cybersecurity breach` / `CVE` + 直近
   - `AI` / `cloud` / `semiconductor` + `news` + 直近
   - 必要なら `EU regulation tech` / `SEC filing` など、ユーザーが関心を示した領域を追加
2. 候補を**重複排除ルール**に通し、続けて **`news-recency-policy.md` の掲載日ルール**に通す（満たさないものは新規トピックから除外）。
3. 残りを重要度が高い順に並べる（セキュリティ実害、規制、大手M&A・資金調達、インフラ障害など）。
4. **各トピックに出典URL**を付ける（可能なら一次情報を優先）。
5. **注意書き**: 4月1日前後はエイプリルフールやジョーク記事が混ざりやすい。**単独の煽り見出し**は一次ソースで確認した旨を書く。
6. **`news-log.md` 追記ブロック**のフォーマットは下記「ログ追記フォーマット」。チャットへの返信も同じ構成でよい（新規分のみ要約すればよい）。

## ログ追記フォーマット（`news-log.md` の末尾に追加）

```markdown
## 調査 YYYY-MM-DD HH:MM （TZ）

### 新規トピック

- **件名** — 要約（日本語）。出典: [媒体名](正規化前の元URL)

（カテゴリ別にしたい場合は ### セキュリティ などに分割してよい）

### メモ（任意）

- 省略した既知トピック数: N（任意）
```

## チャットでの返答

- **新規トピック**の要約（ログに書いた内容と整合）
- **ファイル保存先**をパスで明示
- 新規0件ならその旨

## ツール

- **Web検索**を主に使う。ページ本文が必要なら取得ツールで要約・引用の根拠を補強する。
- **`seen-urls.json`・`news-log.md`・`news-recency-policy.md` の読み取りは必須**（重複排除と鮮度のため）。

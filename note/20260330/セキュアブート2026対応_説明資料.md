# 【2026年問題】セキュアブート証明書の期限切れに備える
「2026年6月にセキュアブート証明書が期限切れになるらしい」  
この話を聞いて、不安になった方も多いと思います。

結論から言うと、今すぐ全PCが起動不能になるわけではありません。  
ただし、放置すると将来的にセキュリティ更新が適用しづらくなる可能性があるため、**今のうちに状態確認しておく価値は高い**です。

この記事では、次の順番で整理します。

- そもそもセキュアブートとは何か
- 自分のPCが更新済みかを確認する方法
- エラーコード `1801` の意味と見方
- `True` なのに `1801` が出るケースの考え方
- 実際にやるべき対処の優先順

---

## そもそもセキュアブートとは？

### 1. セキュアブートを一言でいうと？
「PCの起動時に、怪しいプログラムが紛れ込んでいないかチェックする門番」です。  
OS（Windowsなど）が立ち上がる前の「一番無防備な瞬間」を狙うウイルス（ルートキット等）から守る仕組みです。

### 2. どんな仕組みで守っているのか？
**「信頼の鎖（チェーン・オブ・トラスト）」** で守っています。  
マザーボード（UEFI）が、読み込もうとするプログラムの「署名（印鑑）」をチェックします。

- OK: 信頼できる署名がある -> OSを起動
- NG: 署名がない、または改ざんされている -> 起動をブロック

### 3. なぜ「証明書」に期限があるのか？
技術の進歩により、古い署名技術ではセキュリティを突破されるリスクが出てきます。  
そのため、定期的に「より強固で新しい署名（証明書）」にバトンタッチする必要があります。  
今回の2026年6月問題は、その「バトンタッチの期限」が迫っているという話です。

### 4. ユーザーにとってのメリットと注意点
- メリット: PCの乗っ取りや、偽OSの起動を未然に防げる
- 注意点: 期限切れのまま放置すると、将来のセキュリティ更新プログラムが「偽物（署名が古い）」と判定され、適用できなくなる可能性がある

## 自分のPCが更新済みかを確認する方法

1. **PowerShellで確認する（推奨）**  
   スタートボタンを右クリックし、`ターミナル（管理者）` または `PowerShell（管理者）` を開いて、次のコマンドを実行します。

   ```powershell
   [System.Text.Encoding]::ASCII.GetString((Get-SecureBootUEFI db).bytes) -match "Windows UEFI CA 2023"
   ```

   判定の目安:
   - `True` -> 2023証明書が登録されている可能性が高い
   - `False` -> 更新未完了の可能性が高い
   - `True` でも `1801` が出る場合は、他の反映が未完了の可能性があるため次の章もあわせて確認してください。

2. **レジストリで確認する（補助）**  
   `Win + R` を押して `regedit` を実行し、次のパスを開きます。
   - パス: `HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Control\SecureBoot\Servicing`
   - 値: `UEFICA2023Status`
   - `Updated` なら更新済みの目安

3. **イベントビューアーで更新結果を確認する（間接確認）**  
   `イベントビューアー > Windows ログ > システム` を開き、イベントソース `SecureBoot-Update` や `TPM-WMI` を確認します。  
   直近でエラー `1801` が出ていなければ、更新処理が大きく詰まっていない目安になります。

4. **Windows Update履歴を確認する（間接確認）**  
   `設定 > Windows Update > 更新プログラムの履歴` から、セキュアブート/UEFI関連更新が適用済みかを確認します。  
   ただし、これだけで「Windows UEFI CA 2023 登録済み」とは断定できません。

5. **システム情報（msinfo32）でSecure Boot状態を確認する（間接確認）**  
   `Win + R` で `msinfo32` を起動し、`システム概要` の `セキュア ブートの状態` を確認します。  
   `有効` であればSecure Boot機能自体は有効ですが、証明書の世代までは判別できません。

6. **BIOS/UEFI設定画面で確認する（間接確認）**  
   BIOS/UEFI設定画面を開き、Secure Bootが `Enabled` かを確認します。  
   こちらも有効/無効の確認が中心で、証明書の詳細確認にはPowerShellまたはレジストリ確認が必要です。

---

## エラーコード `1801` はなぜ重要？

`1801` は、TPM-WMI のイベントとして記録される「Secure Boot の証明書更新は行われたが、デバイスのファームウェアへまだ適用（反映）されていない」ことを示すエラーです。  
証明書移行を進めるうえで、見逃せないサインであり、重要な理由は次の3点です。

- 再起動や反映の完了待ちで止まっている可能性があるため
- PowerShellで `True` が出ていても、他の Secure Boot 変数（例: DB/DBX/KEK 等）の反映が未完了なケースがあるため
- この状態を放置すると、2026年以降の早期起動（Secure Bootまわり）の保護が十分に適用されない可能性があるため

---

## `True` なのに `1801` が出るのはなぜ？

実は、これはあり得ます。

`PowerShell` の `True` は「`Windows UEFI CA 2023` が Secure Boot の `db` に含まれている可能性」を示します。  
一方で `1801` は「更新された証明書が、デバイスのファームウェアへ完全に適用されていない」ことを示します。  
そのため、次のような状況では `True` でも `1801` が残ることがあります。

- 他の Secure Boot 変数（例: `KEK` や `DBX`、Option ROM 用の証明書）が未反映のままになっている
- ブートマネージャ更新など、反映が完了するまでに再起動が必要な段階にある

また、更新の途中ではログが残ることもあるため、**再起動後の最新ログ**で「直近に `1801` が出ているか」を確認するのが重要です。

## 実際にやるべき対処（優先順）

1. **Windows Updateを最新化**
   Windows Update（特に月例更新や任意更新）で、セキュアブート関連の更新が配布・適用されることがあります。まずはPCを「最新状態」にし、必要な更新は適用→再起動まで行ってください（途中のままだと判定がブレます）。
2. **メーカー提供のBIOS/UEFIを最新化**
   証明書や拒否リスト（`db`/`dbx` 等）の反映挙動はファームウェア側の実装に依存します。そのため、BIOS/UEFIの更新で適用性が改善する場合があります。アップデートは必ずメーカー公式から取得し、リリースノートに `Secure Boot` や `UEFI` 変数関連の修正がないか確認すると安全です。
3. **再起動後にPowerShell判定とイベントログを再確認**
   PowerShellで `Windows UEFI CA 2023` が `True` かを再確認し、あわせてイベントログで `1801` が「直近で」出ていないかを確認します。ここで見るべきは“過去の記録”ではなく、更新手順のあとに新たに出ているかどうかです。
4. **1801が継続する場合は、メーカー情報を確認してSecure Bootキー再構成を検討**
   `1801` が再起動後も繰り返し発生する場合は、端末の機種/BIOSバージョンに固有の要因が考えられます。自分で Secure Boot キーを初期化・復元する前に、メーカーが案内する回避策や必要なファームウェア更新の有無を確認し、その手順に従ってください。
   
---

## 作業時の注意点

- BIOS/UEFI更新やSecure Boot関連の操作を行う前に、`BitLocker` 回復キーを必ず保管してください（失念すると復旧に時間がかかります）。
- Secure Bootキーの初期化/復元（工場出荷時のキー復元など）は、メーカーが手順として明示している場合に限って実施してください。やり方を誤ると起動リスクが出ます。
- レジストリ操作など、更新の制御に関わる変更は、挙動が端末や環境に依存するため、管理者・検証環境で慎重に行ってください（実運用前の事前確認推奨）。
- 企業貸与PCは、更新やBIOS設定がポリシーで制御されている可能性があります。気になる場合や操作権限がない場合は、独断で変更せず情シス（IT管理部門）へ確認してください。

---

## 古い端末は要注意

2017年以前の機種では、必要なBIOS更新が提供されない場合があります。  
対応不能な場合は、セキュリティ維持の観点から端末更改も選択肢に入ります。

---

## まとめ

今回のポイントはシンプルです。

- まずは `True/False` と `1801` の有無を確認する
- 次に、Windows UpdateとBIOS更新で土台を整える
- `True` でも `1801` が出るなら「中途半端な更新状態」を疑う

2026年6月はまだ先ですが、今確認しておけば、後から慌てる可能性を大きく減らせます。
まずは自分の端末の状態を一度チェックしてみてください。

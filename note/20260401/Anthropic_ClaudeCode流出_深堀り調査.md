# Anthropic と「Claude Code」ソース流出 — 深堀り調査メモ

**調査日:** 2026-04-01  
**テーマ:** Anthropic による Claude Code 一部ソースのリリースパッケージ人的ミスによる漏えい、および直近の「未公開モデル名が公開キャッシュに載っていた」件との関係。Firstpost 等の報道を手がかりに、セキュリティ情報整理の3問（背景・目的・効果）で整理。

---

## 1. 事象の整理（前提）

### 第1波（報道では概ね 2026-03-26 前後）

CMS 等の設定不備により、未公開コンテンツが公開キャッシュ／検索可能な状態になっていた、という報道。未公開モデル「Mythos」やティア名「Capybara」などが話題に。

- 出典例: [Fortune — Anthropic left details of unreleased AI model… (2026-03-26)](https://fortune.com/2026/03/26/anthropic-leaked-unreleased-model-exclusive-event-security-issues-cybersecurity-unsecured-data-store/)

### 第2波（2026-03-31 前後）

`claude-code` の npm パッケージにソースマップが同梱され、クライアント側の TypeScript 相当が読める状態になった、という報道。発見者が公表し拡散した流れが多くの記事で触れられている。

- 出典例: [WinBuzzer — Anthropic Leaks Claude Code Source via Npm Source Map](https://winbuzzer.com/2026/03/31/claude-code-source-leaked-npm-source-map-xcxwbn/)
- 2件目としてのまとめ: [Fortune — second security lapse…](https://fortune.com/2026/03/31/anthropic-source-code-claude-code-data-leak-second-security-lapse-days-after-accidentally-revealing-mythos/)

### 会社側の説明（報道経由で伝わる範囲）

人的ミスであり、**ユーザー・顧客データや認証情報は含まれない**旨の説明がある、という報道が複数ある（公式プレスの全文までは本メモ作成時点で未突合）。

「一週間で二度目」というフレーミングは、**外部脅威の急増**というより、**同じ組織の公開・配布プロセス上のミスが短期間に2種類表面化した**、という読み方が報道と整合する。

## 2. 問い1: なぜ「今」この情報を公開したのか？（背景）

「公開」は次の2層に分けると整理しやすい。

| 種類 | 内容 |
|------|------|
| **(A) 組織の自発的開示** | 方針・年次レポート型の「今だから話す」ではないケースが主。 |
| **(B) メディア・研究者の暴露 → 追って会社が認める／コメント** | 既に外に出た事実への**反応型**に近い。 |

### 観点別の当てはまり

| 観点 | 当てはまり方 |
|------|----------------|
| **脅威急増** | 主因ではない。中心は「社内の設定・ビルド／npm 公開のミス」であり、第三者の大規模攻撃というより**オペレーション事故**に近い。 |
| **法制度変更** | 直接のトリガーとしては弱い。ただし、大企業顧客・規制当局・投資家向けに「個人データ・認証情報は含まない」と言い切るのは、説明責任・契約上の安心材料になり得る。 |
| **大規模インシデント** | **信用・ブランド上のインシデント**としては大きい。「安全性・慎重さ」を売りにする企業ほど、既に外に出た後に沈黙すると推測と悪意のある解釈が先に回るため、**認定・範囲限定・再発防止の言及**が遅れにくい。 |
| **年次レポート等** | 該当しない。典型的には**報道・SNS・コミュニティで既に可視化された後の、事実関係の枠取りと影響の限定**に近い。 |

**要約:** 「今」話す理由は、**情報が既に流通しているタイミングで、何が漏れていて何が漏れていないかを組織側が定義し直し、二次被害（「全部流出した」誤解）を抑える**ため、という側面が強い。

## 3. 問い2: 誰に、何をしてほしいのか？（目的）

| 想定読者 | 組織側が望みそうなこと |
|----------|-------------------------|
| **既存・見込み顧客（特にエンタープライズ）** | 「モデル重みや顧客データの大量流出ではない」と理解してもらい、契約継続・新規商談の心理的ハードルを下げる。社内のリスク評価で「影響範囲はコード／マーケ素材側」と整理しやすくする。 |
| **開発者・CLI 利用者** | **パッケージの更新・入手経路の統一**、公式情報への追随。ミラー／フォーク依存はライセンス・サプライチェーンリスク、という論点も記事によっては含まれる。 |
| **規制・メディア・世論** | ナラティブを「二流のハッキング」ではなく**限定された人的ミス＋是正**に寄せる。 |
| **社内・取締役会** | 再発防止（npm 除外、ビルドパイプライン、CMS の公開範囲監査など）の**説明可能性**。 |

### 顧客が想定読者に含まれる場合の行動例（社内報告用）

- 利用中の `claude-code` / 関連クライアントの**バージョンと入手経路の棚卸し**
- **公式アナウンスと CVE 等の有無**の確認（該当すれば）
- 流出コードに**社名・内部 URL・認証まわりの記述が混ざっていないか**の確認（「顧客データは含まれない」とは別次元のリスク）
- ベンダーに対する**再発防止策の確認質問**（リリースゲート、シークレットスキャン、npm publish の承認フローなど）

## 4. 問い3: 広まると何が変わるのか？（効果）

### 短期

- Anthropic に対する**リリース工程・セキュリティ監査の外部からの注目**が一時的に強まる。
- 他社も **「npm にソースマップを載せない」「CMS のドラフトがインデックスされない」** をチェックリストに足しやすくなる。

### 中期

- **AI コーディング CLI／エージェント製品**について、「クライアントに何が入っているか」「未公開機能フラグがコードに残るリスク」が、調達・セキュリティレビューの定番論点になりうる。
- コミュニティ側では流出コードの**フォーク・解析**が続き、競合・研究者の「実装ヒント」としての二次利用が議論される（ライセンスと倫理は別問題）。

### 長期

- 「安全を掲げる企業もヒューマンエラーでは防ぎきれない」という認識が共有され、**文化としては「完璧なブランド」より「透明性・検証可能性・復旧力」**が重視されやすくなる、という読みもある。
- 一方で、**短期の二度手間**が続くと「運用品質」への不信がブランドに定着するリスクもある。

## 5. 来期に向けて検討しうること（提案）

1. サードパーティ AI 開発ツール導入時の契約・DPA・サブプロセッサーに加え、**「クライアント配布物に何が含まれるか」**の確認項目をテンプレ化する。
2. 自社の npm / パッケージ公開でも **`.npmignore`、publish CI、本番ビルドでの source map 禁止**をゲートにする（今回の類型の再発防止）。
3. **Web／CMS の公開ステージングと検索エンジン・キャッシュ**（noindex、認証、WAF）を、年1回ではなく**変更時に自動テスト**する。
4. インシデント時の**対外コミュニケーション**（何を言い切れるか／言い切れないか）を、情シス・広報・法務で事前にシナリオ化しておく。

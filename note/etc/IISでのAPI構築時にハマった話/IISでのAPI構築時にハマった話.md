# IISでのAPI構築時にハマった話

IIS（Internet Information Services）上で新しい ASP.NET Core の API を立てたときにかなり時間を溶かしたので、切り分けの覚書です。公式ドキュメントの抄訳ではなく、個人の障害対応メモとして書いています。

---

## はじめに

結論から言うと、原因はプログラムや接続文字列ではなく、**IIS のアプリケーションプールを既存 API と共有していたこと**でした。ここに気づくまで、設定ファイルを行き来していました。

---

## 最初は`.env` や `appsettings.json` を疑っていた

IIS にはもともと別の API が動いており、追加する新 API も「既存と同じで大丈夫だろう」と、**既存のアプリケーションプールをそのまま使う**形で進めていました。

しかし、いざセットアップが完了してもAPIが上手く疎通しませんでした。

接続文字列は `.env` や `appsettings.json` で何度も確認し、権限周りも一通り問題ないように見えました。それでも IIS 経由でAPIを叩くと接続まわりでこけ続ける。一体何が起こっているのか。

---

## ローカルだけ動くなら、疑うのは IIS 側

「そもそもアプリが壊れているのか」を切るため、IIS を通さず PowerShell から DLL を直接起動してみました。

```powershell
dotnet .\myApp.dll
```

※ DLL 名は汎用化しています。

起動後、ブラウザで `http://localhost:5000/api/health`（ヘルスチェック用）を叩くと、

- Result: `{"status": "OK"}`

という感じで問題なく返りました。**アプリ単体なら動く。なら原因は IIS 側に寄せてよい**と踏んで、そのあと IIS 設定とにらめっこが続きます。

---

## デベロッパーツールに出てきた「HTML」

設定をいじっている最中、ブラウザのデベロッパーツール（Console）に、次のようなログが紛れ込みました。

```text
page-06ca0c00168e631e.js:1 [login] non-JSON body: <!DOCTYPE html>
...
<title> HTTP Error 500.35 - ASP.NET Core does not support multiple apps in the same app pool </title>
```

JSON を期待しているのに HTML が返ってきた、という趣旨のメッセージです。

気になって本文を読むと、肝はこのあたりでした。

> **HTTP Error 500.35 - ASP.NET Core does not support multiple apps in the same app pool**

---

## 原因：ASP.NET Core は「1プール 1アプリ」が原則

調べてみると、IIS 上では **同じアプリケーションプールの中で ASP.NET Core アプリを複数同時に動かせない**、という制約がありました。

ASP.NET Core はアプリケーションプールのプロセス内で動くので、1 プロセス（1 プール）に載せられる ASP.NET Core アプリは 1 つ、という整理になります。既存 API がそのプールを先に占有していたため、あとから載せた API が正しく立ち上がらず、IIS が 500.35 の HTML ページを返していた、というのが結論です。

---

## 解決：専用のアプリケーションプールを作る

やること自体はシンプルで、**アプリごとに専用のプールを用意する**だけでした。

1. IIS マネージャーを開く。
2. 「アプリケーションプール」を右クリックし、**「アプリケーションプールの追加」** を選ぶ。
3. 名前（例：`MyNewApiPool`）を付けて作成する。

.NET CLR のバージョンは、調べると「マネージド コードなし」を勧める説明も出てきますが、こちらの環境ではバージョンを選んでも動作しました。

1. 対象アプリケーション（API）の「基本設定」から、**さきほど作ったプールを選ぶ**。

保存したところ、通信はさっぱり通るようになりました。

---

## まとめ

- **まず疑うのは設定ファイルだけにしない。** `dotnet` で単体起動できれば、次はリバースプロキシや IIS 周りを疑う足がかりになります。今回は、既存が上手く動いているせいで、盲目的にアプリ側は大丈夫と思っていたことがそもそも問題でした。
- IIS で ASP.NET Core を載せるときは、**アプリケーションプールの使い回しは危険**です。
- **1 アプリ 1 プール**は、自分にとっては「一度ハマったら忘れない」系の鉄則になりました。

同じエラーで時間を溶している方の参考になれば幸いです。

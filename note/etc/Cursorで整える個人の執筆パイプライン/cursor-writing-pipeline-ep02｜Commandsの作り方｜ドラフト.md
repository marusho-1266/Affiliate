# Cursorで作る「書き続ける仕組み」入門 ②｜Commandsの作り方

第1話で作った `/weekly-report` を1回でも動かしてみた方向けに、**「同じ形式の Command を5本・10本と量産できる書き方」**を渡します。**前半は無料**で、Command の骨格（依頼文／入力／出力の型／ルール）と、1本目の型を流用した2本目（読書メモ）の作り方まで。**後半（有料）**で、「型が守られない日をどう減らすか」の5つの勘所と、運用版の完全なコピペ素材3本が揃います。**Command が「便利ツール」から「自分の仕事の型」に変わる分岐点**が、この回です。

> **この連載について（第2話）**
> 本記事は有料想定の連載の第2話です。**前半（1〜4章）は無料**、**後半（5章以降）は note エディタで有料ブロックに指定**する想定で書いています。第1話を読んでいなくても読めるよう、冒頭で要点を再掲します。
> **読了目安:** 約12〜15分／**想定読者:** Cursor で最初の Command を触ってみた、またはこれから作ってみたい非開発者。

---

## 第1話の要点（1段落で）

第1話では、**Cursor を「資料とチャットがつながる作業場」**と捉え、最初の一歩として `/weekly-report` という Command を1本作りました。Command は `.cursor/commands/〈名前〉.md` に書かれた **「定型の依頼文」**で、Cursor のチャット欄で `/〈名前〉` と打つだけで呼び出せます。同じ指示を毎回書かなくても、**毎回同じ型で作業が立ち上がる**のが肝でした。第2話では、この Command を**再現性のある道具として育てる書き方**を扱います。

---

## この記事で得られるもの

- **無料パート（1〜4章）:** Command の最小構成、2本目の作り方（読書メモ）、置き場所と命名の緩い約束。
- **有料パート（5章以降）:** 「型が守られない」「書きにくい」「読み返せない」を減らす**5つの勘所**、自分の型を言語化する**3問のワークショップ**、**つまずきリスト深掘り**、**出荷前チェックリスト**、そのままコピペで使える**完全な Command 実例3本**。

## 前提（短く）

- Cursor のバージョン・UI は変わります。本稿は**ファイルの配置と中身**を主に扱うので、画面の変化には比較的強い構成です。
- すべての Command はローカルのフォルダ1つから始められます。GitHub は不要です。
- 完全自動は目指しません。**最後に人が整える**前提で書きます。

## 1. Command の最小構成

Command は Markdown ファイル1枚です。ばらして見ると、中身は次の**4つのパート**に分解できます。

| パート | 役割 | 例 |
|--------|------|-----|
| **依頼文（先頭）** | AI に何をしてほしいかを1〜2行で宣言 | 「社内共有向けの週報を作成してください」 |
| **入力** | ユーザーが何を貼り付けるかを説明 | 「今週やったこと・所感・来週の予定」を箇条書きで |
| **出力の型** | 見出し・順序・文量の指定 | 「1. ハイライト / 2. 進捗 / …」 |
| **ルール** | 書き方・禁止・仕上げ | 「数字は出典のあるものだけ」「末尾にチェックリスト」 |

この4つが揃っていれば、**Command として成立**します。逆に、どれかが抜けていると、**毎回出力がブレて「結局また手で直す」**という現象が起きます。第1話で配った `/weekly-report` も、見ればこの4パートがきちんと揃っています。

## 2. 2本目を作る — 読書メモ

1本目が動いたら、**2本目は型を流用するだけで作れます**。執筆以外の題材として、読書メモの Command を用意します。1冊読み終えた直後の記憶が鮮度を保っているうちに、翌朝の自分が思い出せる粒度で残すための道具です。

### `.cursor/commands/reading-note.md`

```markdown
# 読書メモを作る

読んだばかりの書籍について、**翌朝の自分でも思い出せるメモ** を Markdown で作成してください。

## 入力
ユーザーが「書名・著者・読んだ目的・印象に残った一節・所感」を自由記述で貼り付けます。

## 出力の型
1. **書誌情報** — 書名 / 著者 / 出版年（分かれば）
2. **読んだ目的**（1文）
3. **3行要約**（3点まで）
4. **印象に残った一節**（原文ママ引用＋ページ数または章。なければ章のみ）
5. **所感**（2〜4段落）
6. **この本を薦めたい相手 / 合わない相手**
7. **次に読みたい関連書**（任意）

## 書き方のルール
- 著者の主張と自分の意見は必ず分ける。「著者いわく」「私の解釈では」を使う。
- 誇張や曖昧な礼賛は書かない。「良かった」は**何が、どの観点で**まで分解する。
- 未読の関連書を断定的に紹介しない。

## 仕上げ
最後に、**1か月後に見返したときに分かる粒度か**を自問する1行を末尾に添える。
```

これで Command が**2本**になりました。1本目が動いたら、**2本目以降は見出しと題材を差し替えるだけで量産できる**手応えが掴めるはずです。

## 3. 置き場所・命名・呼び出しの緩い約束

Command が増えていくと、必ず「どこに置くか」「何と名付けるか」で迷います。最初から完璧は狙わず、**緩い約束だけ先に決めておく**と、後から揃えやすくなります。

### 置き場所

- すべて `.cursor/commands/` の直下に置く。
- サブフォルダは、**本数が10を超えてから**考える。早く分けると、自分が何を持っているか分からなくなる。

### 命名

- **英語・ハイフン区切り・小文字**で揃える。例: `weekly-report.md`、`reading-note.md`、`meeting-minutes.md`。
- 名前は**動詞または成果物名**にする。「book-summary」より「reading-note」、「format」より「format-meeting」のほうが、呼び出すときに意図を思い出しやすい。

### 呼び出しの約束

- 入力は**Command 呼び出しの直後**に貼る。
- 長文を貼るときは、**区切りとして `---` を挟む**と AI が入力範囲を誤解しにくい。
- 1回で完璧を狙わず、**返ってきたら「型に沿って書き直して」と追撃**する。最初はこの追撃回数が1〜2回あるのが普通です。

---

## 4-末. ここまでで出来ること / 出来ないこと

無料パートで扱った内容で、**以下は到達できる**はずです。

- Command を**2本以上**持ち、毎日の作業の立ち上がりを揃える。
- 2本目以降は**構造をコピー**して増やす。

一方で、**次のような壁**に近い将来ぶつかります。

- 出力の型が**守られない日**がある。
- 2本目が1本目と**似て非なる形**に育ってしまう。
- 禁止事項を書いたのに**余計な総括**が末尾に付く。
- Command を**3本以上持つと、自分でも何があるか忘れる**。

これらを減らすのが、**後半（有料）で扱う「5つの勘所」と「つまずきリスト」と「完全な実例3本」**です。ここから先は、**実際に Command を量産している人だけが踏み込む**領域の実務的な話になります。

## 5. 再現性を上げる5つの勘所（ここから有料想定）

前半で最小形は掴めたはずです。ここからは「**書いたのに守ってくれない Command**」を減らすための、実務的な書き方の勘所を5つに絞ります。

この有料パートで売っているのは、Command の完成形だけではありません。むしろ大事なのは、**なぜその書き方にするのか／どこが崩れる原因になるのかを自分で添削できる視点**です。テンプレートはAIにも作れますが、崩れたテンプレートを直す基準は、自分の中に持っていないと運用できません。

### 5-0. まず、失敗する Command の形を見る

次のような Command は、一見それっぽく見えても出力が安定しません。

```markdown
# 週報を書いて

今週の内容をいい感じに週報にしてください。
分かりやすく、読みやすく、必要なら補足してください。
```

弱い点は3つあります。

- **入力の形がない:** ユーザーが何を貼るのか決まっていない。
- **出力の型がない:** 見出し・順序・文量がAI任せになる。
- **書かないことがない:** 推定数字、余計な総括、定型の締めが混ざりやすい。

この状態で「AI が言うことを聞かない」と感じても、原因はAIだけではありません。Command 側が、守るべき型を渡せていないのです。

### 5-1. 入力欄を「1行の契約」にする

「ユーザーが X を貼り付けます」と書くとき、**X を2行以内で明確に定義**します。曖昧な定義は AI に「気を利かせて補完」させてしまい、再現性が下がります。

- 悪い例: 「今週の出来事を貼り付けます」
- 良い例: 「今週やったこと・所感・来週の予定の3ブロックを、各3〜10行の箇条書きで貼り付けます」

**狙い:** 入力の形が決まれば、出力の形も揃う。

### 5-2. 出力の型は「見出しの順序」で固定する

型は**見出しの数と順序**で固定します。各見出しの中身は自由度を残して構いません。「必ず1〜5の順序で、数字も見出しも省略しない」と明記すると効きます。「重要度に応じて順序を入れ替えて」のような指示を**入れると崩れます**。

**狙い:** 順序が揃うと、読者（や未来の自分）が読み流しやすい。

### 5-3. 「書かない」を明文化する

AI が勝手に足しがちな要素を、**禁止リスト**として本文に書きます。例:

- 「絵文字は使わない」
- 「『今後も頑張ります』のような定型締めを足さない」
- 「入力にない数字を推定で書かない」

禁止は**3〜5個まで**に絞ります。増やしすぎると結局守られません。「書くこと」より「書かないこと」のほうが、その人の型を表します。

**狙い:** 禁止は**個人の編集方針**そのもの。ここが他人の Command との差になります。

### 5-4. 「仕上げ」をルーティン化する

Command の末尾に**「仕上げ」節**を置き、最後の1手を指定します。例:

- 「出力末尾に**抜け漏れチェックリスト**を付ける」
- 「全体を声に出して読んで**2分以内**で終わるか自己評価する」
- 「**1か月後の自分**が読んで分かるかの1行を末尾に追加する」

この「ルーティンの押し込み」は、Command が**「便利ツール」から「自分の型」に変わる境界線**です。仕上げの1行があるかどうかで、出力を読み返したときの**納得感**が変わります。

### 5-5. 呼び出しやすい粒度に保つ

1つの Command に**詰め込みすぎない**。粒度の目安は次のとおり。

- **1つの成果物**に対して1本（議事録整形と議事録要約は分ける）。
- **1入力 → 1出力**（複数を束ねる Command は、第3話で扱う Skills に昇格させる判断へ）。

粒度が大きすぎると「何でもできるが何も決まらない」、小さすぎると「毎回 Command を探す」になります。迷ったら次を自問してください。

> **「この成果物が欲しいとき、直感でどのコマンド名を打つか？」**

この問いで名前が2秒以内に出てくる粒度が、自分にとっての正解です。

## 6. 自分の型を言語化する3問ワークショップ（有料）

ここは**読者自身の仕事の型を Command に落とすため**の、空欄テンプレです。紙でもテキストでもいいので、**3つの質問**に答えてから Command を書き始めると、ファイルが短く・強くなります。

```markdown
## 書き起こし前の3問

1. **この成果物を、毎回どんな状況で作っているか**（時間・場所・前後のタスク）
   →

2. **完成品に必ず入っている要素／絶対に入れない要素は何か**
   入っている:
   入れない:

3. **出来上がったものを、翌日の自分や同僚が見たとき、最初に目が行く1点は何か**
   →
```

書き終えたら、**この3問の答えをそのまま Command の「ルール」節に転記**します。1 は入力節の前提に、2 は出力の型と禁止リストに、3 は仕上げ節に落ちます。**自分の型が言語化できていれば、Command の7割は既に書けている**のです。

**コツ:** 3問に答えるのに10分以上かかるなら、それは Command 化が早い題材です。「まだ型が定まっていない仕事」を Command にしても、どのみち毎回 Command を書き換えることになります。**「何度も同じ手順で作っている」**仕事から Command 化してください。

## 7. つまずきリスト（深掘り・有料）

第1話で軽く触れた3つに加え、**Command を増やす段階でだけ発生する**問題を扱います。

| つまずき | 原因 | 対処 |
|---------|------|------|
| 2本目以降が1本目と**似て非なる**形になる | 見出し順を微妙にズラしている | 1本目の「出力の型」節を**コピーしてから差し替え**る |
| 出力の末尾に**余計な感想**が付く | 禁止事項が漏れている | 禁止リストに「総括コメント・締めの挨拶を付けない」を追加 |
| 入力量が多いと**途中から型が崩れる** | 入力の区切りが不明瞭 | `---` で区切る／入力節に「**合計◯◯字以内**」を書く |
| 呼び出し時に**直前のチャット履歴**に引っ張られる | 別トピックの直後に呼んだ | **新しいチャット**で呼ぶ／「この Command の入力は下の `---` 以降のみ」と明記 |
| 仕事用と個人用の Command が混ざる | すべて直下に置いている | 10本超のタイミングで `personal/` と `work/` に分ける |
| **守られない日**がある | モデル側の揺らぎ | 「型に沿って書き直して」で1回追撃。**3回以上守られないなら Command を直す合図** |
| 自分でも何の Command があるか忘れる | インデックスがない | `.cursor/commands/INDEX.md` を1枚作り、本数が増えたら追記する |

**「守られない日がある」は完全には消せません。** それを前提に、「追撃で直せる余地」と「Command を直す閾値」をあらかじめ決めておくと、**諦めずに続けられます**。

## 8. 出荷前の品質チェック（有料）

Command を作ったら、使い始める前に一度このチェックを通してください。再現性の体感が大きく変わります。

- [ ] **冒頭1〜2行**で「何をする Command か」が分かる
- [ ] **入力節**が2行以内で具体的に定義されている
- [ ] **出力の型**が見出しの順序で固定されている
- [ ] **禁止事項**が3〜5個に絞られている（増えすぎていない）
- [ ] **仕上げ**が1つ決まっている（末尾に必ず何かが付く）
- [ ] 自分で**実入力を2つ**流して、**両方で型が崩れない**
- [ ] **1か月後の自分**が、このファイルを読んで中身を思い出せる
- [ ] 公開するなら、**自分の仕事の秘密**が漏れていない

このうち **「実入力を2つ流して両方崩れない」** が最重要です。1回だけの成功は偶然、2回連続で型が守られれば Command として合格です。

## 9. そのままコピペで使える実例3本（有料）

最後に、**コピペで使える運用版の Command を3本**配布します。第1話の `/weekly-report` は**簡易版**でした。以下は**運用版**で、禁止事項・仕上げ節・入力の定義が厚めに入っています。

### 9-1. `.cursor/commands/weekly-report.md`（運用版）

**設計意図:** 第1話の簡易版に比べて、入力の境界、禁止事項、仕上げを厚くしています。週報は「それっぽい前向きな総括」が混ざりやすいため、定型締めと推定数字を明示的に禁止しています。

```markdown
# 週報ドラフトを作る（運用版）

以下の入力をもとに、**社内共有向けの週報**を Markdown で作成してください。

## 入力
ユーザーが以下の3ブロックを、各3〜10行の箇条書きで貼り付けます。
1. 今週やったこと（事実ベース、主語なし）
2. 所感・気づき
3. 来週の予定・依頼事項

入力の境界は、この Command 呼び出し直後の `---` 以降から、次の `---` または末尾までです。

## 出力の型（見出しの順序・数字を省略しない）
1. **今週のハイライト（3点まで）**
2. **進捗（プロジェクト別）** — 箇条書き
3. **気づき・学び（任意）** — 1〜3点
4. **来週の予定・依頼事項** — 箇条書き
5. **来週のリスク・ブロッカー（任意）**

## 書き方のルール
- 主語は「私」を使わない。事実ベース（「〇〇を完了」「△△を相談」）で書く。
- 推測や曖昧なものは「要確認」と明記する。
- 入力にない数字を推定で書かない。

## 書かないこと（禁止）
- 「今後も頑張ります」のような定型の締め
- 絵文字
- 総括コメント（「全体として良い週でした」など）

## 仕上げ
最後に **抜け漏れチェックリスト**（3項目まで）を付ける。例: 「リスクは共有したか」「依頼の期限は明記したか」「来週の最初のブロッカーは特定したか」。
```

### 9-2. `.cursor/commands/meeting-minutes.md`

会議の**走り書きメモ**を、**議事録＋決定事項リスト＋宿題リスト**の3点セットに整形する Command です。

**設計意図:** 議事録で一番危ないのは、議論中の発言を「決定事項」に見せてしまうことです。そのため、出力の型で決定事項と宿題を分け、書き方のルールで「合意が明記された場合だけ」と制限しています。

```markdown
# 議事録を整える

会議中の走り書きメモを、**社内配布に耐える議事録**に整形してください。

## 入力
会議名・日時（任意）・参加者（任意）・走り書きメモ本文。形式は自由。入力の境界は Command 呼び出し直後の `---` 以降。

## 出力の型（この順序で）
1. **会議情報** — 会議名 / 日時 / 参加者（分かる範囲）
2. **要旨**（3行以内）
3. **議題ごとの議論**（議題名を H3 で立て、箇条書き）
4. **決定事項**（番号付きリスト。各項に「誰が何をいつまでに」）
5. **宿題（アクションアイテム）**（番号付きリスト。担当者・期限・成果物を明記）
6. **次回の論点**（任意）

## 書き方のルール
- 発言者の固有名詞は**役職または匿名化**を優先。入力に役職がない場合は「発言A」「発言B」でよい。
- **「決定事項」と「議論中の発言」を混ぜない**。決定事項は合意が明記された場合だけ書く。
- 入力にない**日時・数値**を補完しない。

## 書かないこと
- 発言のトーン（「熱心に」など）
- 結論めいた総括

## 仕上げ
末尾に **「この議事録で抜け落ちやすい観点」**を1〜3行で注記する（例: 決定事項の期限未設定、担当の二重化など）。
```

### 9-3. `.cursor/commands/reading-note-deep.md`（読書メモ・深掘り版）

第1話で扱った読書メモに、**著者の主張への反証可能性**を自問する節を1つ足した版です。積読や速読より、1冊を深く読む習慣のある人向け。

**設計意図:** 読書メモは放っておくと「良かった」「勉強になった」に寄りがちです。反証可能性を1節入れることで、著者の主張をただ受け取るのではなく、自分の中で検証するメモに変えています。

```markdown
# 読書メモ・深掘り版

読み終えた書籍について、**著者の主張を自分の中で検証するためのメモ**を作ります。

## 入力
ユーザーが「書名・著者・読んだ目的・核となる主張（自分の言葉で）・印象に残った一節」を貼り付けます。

## 出力の型
1. **書誌情報**
2. **読んだ目的**（1文）
3. **著者の核となる主張**（2〜3行、自分の言葉で）
4. **その主張を支える根拠**（著者が提示したもの、箇条書き）
5. **反証可能性の自問** — どんな事実・データが出てきたら、この主張は揺らぐか？（2〜3点）
6. **自分の経験との接続**（同意／反対／未検証 を明記した上で2〜4段落）
7. **この本を薦めたい相手 / 合わない相手**
8. **次に読みたい関連書**（任意）

## 書き方のルール
- 著者の主張と自分の意見を必ず分ける。
- 5 の反証可能性は、**「反証不能なら主張ではない」**という姿勢で厳しめに書く。
- 未読の関連書を断定的に評価しない。

## 書かないこと
- 曖昧な礼賛（「とても勉強になった」だけ）
- 著者の動機の憶測（「著者はおそらく〜したかったのだろう」）

## 仕上げ
末尾に **「1か月後にこの本を誰かに紹介するとしたら、何を1行で言うか」**を書く。
```

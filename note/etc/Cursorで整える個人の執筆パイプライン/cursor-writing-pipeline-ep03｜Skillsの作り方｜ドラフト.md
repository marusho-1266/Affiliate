# Cursorで作る「書き続ける仕組み」入門 ③｜Skillsの作り方

Commands を触ってみて「ルールが重複している」「Command ごとに語尾や禁止が微妙にズレる」と感じた方へ。Skills は、**ユーザーが明示的に呼ばなくても AI が自動で読みに行く横断ルール集**。**前半は無料**で、Skills と Commands の違い／最小の SKILL.md ／Skill 化の判断基準まで。**後半（有料）**で、「AI に拾ってもらう `description` の書き方」と、**散らばったルールを既存 Command から引き剥がす3ステップ**を、実例付きで渡します。**Command ファイルが目に見えて短く・読み返しやすくなる**のが、この回の果実です。

> **この連載について（第3話）**
> 本記事は有料想定の連載の第3話です。**前半（1〜4章）は無料**、**後半（5章以降）は note エディタで有料ブロックに指定**する想定で書いています。第1話・第2話を読んでいなくても読めるよう、冒頭で要点を再掲します。
> **読了目安:** 約12〜15分／**想定読者:** Commands を3〜5本作ってみて、「同じルールをあちこちに書いている」ことに気づき始めた非開発者。

---

## 第1〜2話の要点（1段落で）

第1話では **Cursor を「資料とチャットがつながる作業場」**と捉え、`.cursor/commands/〈名前〉.md` に書いた定型の依頼文＝**Command** を `/〈名前〉` で呼ぶ仕組みを導入しました。第2話では Command の**4パート構成（依頼文／入力／出力の型／ルール）**と、**再現性を上げる5つの勘所**、そして運用版の実例3本（週報・議事録・深掘り読書メモ）まで踏み込みました。

第3話では、Command が**3〜5本まで育ったとき**に起きる「ルールが重複する」「Command 間で整合が取れない」問題に対応する道具＝**Skills** を扱います。

## 1. なぜ Skills が必要になるか — Commands 3〜5本後の症状

Commands を2〜3本作った段階では、Skills は**不要**です。第2話でもそう書きました。ところが、5本前後まで増えると、次のような症状が出始めます。

| 症状 | 具体例 |
|------|--------|
| **同じ禁止事項を何度も書いている** | 「絵文字は使わない」「定型の締めを足さない」が、`weekly-report.md`・`meeting-minutes.md`・`reading-note.md` の3箇所にコピペ状態で存在する |
| **語尾・文体が Command ごとに微妙に違う** | ある Command は「である調」、別の Command は「ですます調」が混ざり、読み返すとちぐはぐ |
| **ルールを直したいとき、3〜5ファイルを開き直している** | 「禁止事項を追加したい」だけなのに、全 Command を巡回して同じ追記 |
| **判断ルールが Command 本文に紛れて、呼び出しごとに目が滑る** | 1本の Command が長文化し、本当の「依頼文」が読めなくなる |

これは**Command を使っているなら誰もが通る成長痛**です。この段階で初めて、**「複数の Command にまたがる共通ルール」を、別ファイルに引き剥がす**価値が出てきます。これが Skills の出番です。

## 2. Skills と Commands の違い（使い分けの核）

第1話でも1行ずつの比較を書きましたが、ここではもう少し踏み込んで整理します。

| 観点 | Commands | Skills |
|------|----------|--------|
| **置き場所** | `.cursor/commands/〈名前〉.md`（1ファイル） | `.cursor/skills/〈名前〉/SKILL.md`（1フォルダ + SKILL.md） |
| **呼び出し方** | **ユーザー**が `/〈名前〉` で明示的に呼ぶ | **AI が自動で**「今これが必要そう」と判断して引き込む |
| **役割** | 「こう指示して」= **定型の依頼文** | 「こう考えて／こう書いて」= **判断ルール集・品質基準** |
| **何を書くか** | 入力・出力の型・その Command 固有のルール | **横断的に効くルール**・判断基準・禁止・品質チェック |
| **長さの目安** | 短い（30〜80行） | 中程度〜長め（80〜300行） |

Skills の一番の特徴は **「AI が自動で読みに行く」**ことです。だからこそ、Skill には**「いつ使うか」の情報そのものをファイル内に書く**必要があります。これが後述する `description` の話につながります。

### ざっくりの覚え方

- **Commands = レシピ**（材料と手順が書いてある／呼ばれたら動く）
- **Skills = 家の作法**（誰が料理しても守る暗黙ルール／常に背景で効いている）

この2つを同時に使うことで、「料理のたびに手順を思い出す必要がなく、かつ家の作法は勝手に守られる」状態が作れます。

## 3. 最小の Skill の骨格

Skill のファイルは `.cursor/skills/〈英語名〉/SKILL.md` です。中身は Markdown で、**先頭にフロントマター**（YAML形式の小さな設定欄）が付くのが Command との最大の違いです。

```markdown
---
name: editorial-voice
description: >-
  自分の文章全体に効く語尾・禁止事項・帰属ルールを固定する。
  週報・議事録・読書メモなど、長文を書くすべての Command の
  バックグラウンドで効くこと。「ですます調で書いて」「定型の締め
  を足さないで」と繰り返し指示している人向け。
---

# 執筆の編集方針（editorial-voice）

## 適用範囲

このスキルは、**私が書く長文すべて**（note 記事、議事録、週報、読書メモなど）に効かせる。

## 語尾・文体

- ですます調で統一する。
- 句点の直前に「〜ね」「〜よ」などの話し言葉を入れない。
- 1文は原則として80字以内。超える場合は2文に分ける。

## 禁止事項

- 絵文字は使わない。
- 「今後も頑張ります」「少しでも参考になれば幸いです」などの定型締めを足さない。
- 入力にない数字を推定で書かない。

## 帰属のルール

- 他者の主張を書くときは「誰が・いつ・どこで」のうち分かる情報を必ず添える。
- 自分の意見との境界は「私の解釈では」「個人的には」で明示する。

## 品質チェック（AI 側の自己確認）

出力を返す前に、次を内心で確認してから返す。

- [ ] 語尾は統一されているか
- [ ] 禁止事項に触れていないか
- [ ] 主張には帰属が付いているか
```

フロントマターとは、ファイル先頭にある `---` で挟まれた設定欄のこと。ここの **`description`** が、**AI が「今この Skill を参照すべきか」を判断する最重要フィールド**です（後半で深掘りします）。

## 4. Skill に昇格させる判断基準

Command と Skill の境界は、慣れるまで迷います。**最初は迷ったら Command のまま**で構いません。以下の**3つが2つ以上あてはまったら、Skill 化を考え始める**、という目安を覚えておいてください。

| 判断軸 | Skill 化のサイン |
|--------|------------------|
| **重複** | 同じルール／禁止が**3本以上**の Command に同じ内容で書かれている |
| **変更頻度** | その共通ルールだけを**単独で変えたい**場面が月に1回以上ある |
| **AI 側に持たせたい暗黙知** | 明示的に呼ばなくても、**背景で効いていてほしい**（= ユーザーが毎回 `/` で呼ぶべきでない）ものになっている |

逆に、**次のものは Skill 化しない**ほうが無難です。

- 1回しか使わない判断（その Command にだけ書けばよい）
- **入力と出力の型そのもの**（これは Command の本業）
- 「呼んだときだけ効いてほしい」もの（これは Command 向き）

---

## 4-末. ここまでで出来ること / 出来ないこと

無料パートの内容で、以下は到達できます。

- Commands が増えてルールの重複に気づき、「そろそろ Skill か？」と自問できる。
- 最小の SKILL.md を真似て、**1つだけ Skill を作ってみる**ことができる。

一方で、次のような壁がすぐに出ます。

- `description` の書き方が曖昧で、**AI が Skill を拾ってくれない**。
- 既存 Command に散らばった共通ルールを**どう引き剥がせばいいか**、手順が定まらない。
- Skill を作ったら**Command 側がどう軽くなる**のかイメージが湧かない。
- Skill が**重くなりすぎて**逆に守られない。

これらに実務で効く書き方・手順・実例が、**後半の有料パート**です。

## 5. SKILL.md の書き方の勘所（ここから有料想定）

Skill は AI が自動で読み込む点が肝です。**「呼ばれないと意味がない」**ので、**誘導情報＝フロントマター**の書き方が最大のコツになります。

第2話で Command を増やした人ほど、この回の価値が出ます。Command は「作る」だけなら簡単ですが、3〜5本に増えた瞬間、同じ禁止事項や語尾ルールがあちこちに散らばります。この有料パートでは、その散らばりを**管理できる形に畳み直す**ところまで扱います。

### 5-0. 失敗する Skill の形

まず、拾われにくい Skill の例です。

```markdown
---
name: writing-style
description: 文章を整える
---

# 文章ルール

読みやすくしてください。
必要なら自然に直してください。
```

この Skill は弱いです。理由は明確です。

- `description` が短すぎて、AI が**いつ使えばよいか判断できない**。
- 対象成果物（週報、議事録、note記事など）が書かれていない。
- 禁止事項や品質チェックがなく、出力前に何を守るべきか曖昧。

Skill は「良い感じにするファイル」ではありません。**いつ使うか、何を主管するか、何をしないか**まで書いて初めて、Command 群の背景で効くルールになります。

### 5-1. `description` は「呼ばれたい場面」を書く

`description` は、AI が**「今ユーザーがやっていることに、この Skill は関係あるか？」**と判断する手がかりです。ここに機能説明だけを書くと、AI は拾いません。次の3要素を**1〜3文の中に混ぜる**のがコツです。

1. **この Skill が何をするか**（機能説明）
2. **どんな出力・成果物と関係するか**（週報・議事録・note記事、などの具体名）
3. **いつ使うか（トリガー語）**（「〜を書いているとき」「〜の語尾が揃わないとき」など、状況とキーワードを入れる）

悪い例と良い例:

- 悪い例: `description: 文体を整えるスキル` ← 短すぎて拾われない
- 良い例: `description: 自分の文章全体に効く語尾・禁止事項・帰属ルールを固定する。週報・議事録・読書メモなど長文を書くすべての Command のバックグラウンドで効くこと。「ですます調で書いて」「定型の締めを足さないで」と繰り返し指示している人向け。`

**狙い:** AI がユーザーの指示・入力文の中に、Skill を呼び出す手がかりを発見できるようにする。

### 5-2. 本文は「見出し」で引ける構造にする

Skill 本文は長めになりがち。読み手（AI）が**見出しだけで目的の節に到達できる**ように構造化します。

- **適用範囲** — この Skill がいつ効くか
- **ルール本体** — 実質的な判断基準・禁止・書き方
- **例外** — 敢えてルールを外す状況がある場合
- **品質チェック** — 出力前の自己確認リスト
- **関連** — 他の Skill や Command との関係（参照）

節が多くなりそうなら、**ルール本体を2〜3節に分ける**のも有効です（例: 語尾／禁止／帰属）。

### 5-3. 「守らせる」ために命令形を使う

Skill の本文は**命令形で書く**ほうが守られます。「〜するとよい」ではなく「〜する」「〜しない」。

- 弱い: 「できるだけ絵文字は避けましょう。」
- 強い: 「**絵文字は使わない**。」

禁止は**太字**で書くと AI が拾いやすい、という経験則もあります。

### 5-4. 品質チェックは「チェックリスト形式」で末尾に置く

本文の最後に **「出力する前に AI が内心で確認するチェックリスト」**を置くと、出力の安定性が一段上がります。`- [ ]` 形式で書き、**5項目以内**に絞るのがコツ。増やしすぎると守られません。

### 5-5. 「他の Skill との境界」を明記する

Skill が増えると、「このルールはどの Skill が主管か」が曖昧になります。冒頭に「**このスキルは〇〇を主管する／〇〇については△△スキルに譲る**」と書くと、AI が複数 Skill を同時参照したときに競合しにくくなります。

## 6. 既存 Commands から共通ルールを引き剥がす3ステップ（有料）

既に Commands を3〜5本持っている人向けに、「散らばったルールを Skill に引き上げる」実践手順です。

### ステップ1: **重複の棚卸し**

手元の `.cursor/commands/` を全部開き、以下を1枚の表にメモします。

| Command | 禁止事項 | 語尾 | 帰属のルール | 仕上げ |
|---------|----------|------|---------------|--------|
| weekly-report | 絵文字／定型締め／推定数字 | ですます | — | チェックリスト |
| meeting-minutes | 結論的総括／発言トーン | ですます | 役職優先 | 抜け観点 |
| reading-note | 曖昧な礼賛／動機の憶測 | ですます | 著者と自分を分離 | 粒度自問 |

この表を作るだけで、**「3本とも書いてある項目」が一目瞭然**になります。

### ステップ2: **引き剥がす候補の絞り込み**

表を見て、**3本以上にまたがって同じ内容で書かれているもの**を候補にします。上の例だと:

- 語尾: ですます調（3本とも）
- 推定数字の禁止（入力にない数字）
- 絵文字・定型締めの禁止（複数本）

**「1〜2本だけにある固有ルール」は Skill に上げません**。その Command 固有の責任として残します。

### ステップ3: **Skill に書いて、Command 側から削る**

候補ルールを `.cursor/skills/editorial-voice/SKILL.md` にまとめ、**同時に各 Command からそのルールを削る**。ここが勇気のいる作業ですが、ここをやらないと「Skill も作ったけど Command にも残っている」二重管理状態が発生します。

削ってから、**各 Command の実入力を1〜2回流してみて**、出力の型が崩れないか（＝Skill が実際に効いているか）を確認します。これが**受け入れテスト**になります。

## 7. Skills のつまずきリスト（有料）

| つまずき | 原因 | 対処 |
|---------|------|------|
| **Skill が拾われない** | `description` が短い／機能説明のみ | §5-1 に沿って「機能＋成果物＋トリガー語」の3要素を入れ直す |
| Skill が**複数同時に拾われて競合する** | 守備範囲が被っている | §5-5 の「境界」明記／または片方を廃止し1つに統合 |
| Skill を作ったのに**Command と同じことを2箇所で書いている** | Command 側を削っていない | §6 のステップ3を徹底。削ったら1回テスト走行 |
| Skill が**長大化して守られない** | 1つの Skill に詰め込みすぎ | 200行を超えたら分割を検討（語尾系／帰属系／品質チェック系に分ける） |
| **フロントマターの書式エラーで読み込まれない** | 字下げ・クォートのミス、特に `description` 複数行のインデント崩れ | `>-` で始めて後続行を**2スペース字下げ**に揃える／エディタの YAML チェック機能を使う |
| Skill の**改訂履歴が追えない**、何を変えたか分からない | 単一ファイルで上書きしている | SKILL.md の末尾に `## 改訂メモ` 節を作り、変更時に1行ずつ追記する |
| Skill 同士の参照でループが起きる | Skill A → B → A と互いに参照 | 参照は**片方向**に限定する（上位／下位の階層を決める） |

## 8. 出荷前の品質チェック（有料）

Skill を作ったら、保存前に以下を確認してください。

- [ ] フロントマターの `description` が**機能＋成果物＋トリガー語**の3要素を含む（1〜3文）
- [ ] 本文は **見出し**で引ける構造（適用範囲／ルール／例外／品質チェック／関連）
- [ ] 主要ルールは**命令形**で書かれている
- [ ] 末尾に**5項目以内**のチェックリストがある
- [ ] 他の Skill との**境界**が冒頭に書かれている
- [ ] 関連する Command から**同じ内容が削除**されている（二重管理になっていない）
- [ ] **1か月後の自分**が読んで「なぜこの Skill を作ったか」思い出せる
- [ ] 実際に関連 Command を走らせて、**型と禁止が守られる**ことを2回確認した

このうち**最重要は「Command 側から削除した上で、実走行2回で守られている」**。これを経ない Skill は、実質機能していません。

## 9. そのまま使える完全な SKILL.md 実例2本（有料）

第2話で配った3本の Command と組み合わせて使える、実用 Skill を2本配布します。

### 9-1. `.cursor/skills/editorial-voice/SKILL.md`

文章全般の**語尾・禁止・帰属**の方針を固定する Skill。第2話で配った `weekly-report` / `meeting-minutes` / `reading-note-deep` と組み合わせて使うと、それぞれの Command から禁止事項や文体ルールを削れます。

**設計意図:** この Skill は「文体の好み」ではなく、複数の Command にまたがって毎回守りたい編集方針を主管します。Command 側には入力と出力の型だけを残し、語尾・禁止・帰属はここへ寄せることで、二重管理を減らします。

  週報・議事録・読書メモ・note 記事など、長文を書くすべての

# 執筆の編集方針（editorial-voice）

## このスキルの主管範囲

このスキルは、**語尾・禁止事項・帰属のルール**を主管する。
入力・出力の型そのものや、題材ごとの固有ルールは**各 Command に譲る**。

## 1. 語尾・文体

- **ですます調**で統一する。
- 1文は原則として**80字以内**。超える場合は2文に分ける。
- 句点の直前に「〜ね」「〜よ」「〜かな」などの話し言葉を入れない。

## 2. 禁止事項

- **絵文字**は使わない。
- **定型の締め**（「今後も頑張ります」「少しでも参考になれば幸いです」）を足さない。
- 入力にない**数字**を推定で書かない。必要なら「要確認」と明記する。
- 発言や行為の**動機の憶測**（「著者はおそらく〜したかったのだろう」）を書かない。

## 3. 帰属のルール

- 他者の主張を書くときは「**誰が・いつ・どこで**」のうち分かる情報を必ず添える。
- 自分の意見との境界は「**私の解釈では**」「**個人的には**」で明示する。
- 報道と一次情報が混ざる題材では、「〜と報じられている」「公式は〜と述べている」を使い分ける。

## 4. 例外

- 私的なメモや下書きで、**公開を想定しない**ものは本スキルの対象外。
- Command 側で明示的に「である調で」と指定した場合は、その Command に従う。

## 品質チェック（出力前の自己確認）

- [ ] 語尾は統一されているか
- [ ] 禁止事項に触れていないか
- [ ] 他者の主張に帰属が付いているか
- [ ] 自分の意見との境界が明示されているか
- [ ] 公開前提なら、動機の憶測を含んでいないか

# 出力の最終ゲート（output-quality-gate）

## このスキルの主管範囲

このスキルは、**出力を返す直前の最終確認**を主管する。
文体・禁止・帰属は `editorial-voice` スキルに譲る。

## 必ず通す3つのゲート

### ゲート1: 見出しの順序

- Command 側で指定された**見出しの順序と数**が、出力に反映されているか。
- 勝手に節を追加・削除していないか。
- 「その他」「補足」を呼ばれていないのに足していないか。

### ゲート2: 入力にない情報の混入

- 入力に含まれていない**固有名詞・日付・数値**を、推定で書いていないか。
- 一般常識の範囲で補った箇所があるなら、「要確認」または「一般には〜とされる」を添えているか。

### ゲート3: 曖昧さのチェック

- **主語が分からない**文がないか（「〜される」「〜されている」の多用）。
- **期限・担当・成果物**のうち、業務系の出力で欠けているものがないか。
- 読者（翌日の自分・同僚）が**1回読みで意味を取れる**文になっているか。

- [ ] ゲート3: 期限・担当・成果物が揃っている（該当する場合）

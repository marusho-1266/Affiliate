2026年6月に期限を迎えるセキュアブート（Secure Boot）の電子証明書更新問題について、現在の状況と対応方法をまとめました。

結論から申し上げますと、「OSが起動しなくなる」といったパニックに陥る必要はありませんが、セキュリティを維持するためにはWindows UpdateやBIOS（UEFI）の更新が必要です。

1. なぜ対応が必要なのか？
セキュアブートは、PCの起動時に「信頼された署名（証明書）」があるプログラムだけを実行する仕組みです。

期限切れの影響: 現在広く使われている「2011年版」の証明書が2026年6月に有効期限を迎えます。

リスク: 更新しないまま期限が過ぎると、将来的に脆弱性が修正された新しい「ブートマネージャー」が、古い証明書では「信頼できない」と判断され、最悪の場合OSが起動できなくなったり、最新のセキュリティ更新が適用できなくなったりする可能性があります。

2. 対応が必要かの確認方法
まず、ご自身のPCがすでに新しい証明書（2023年版）に更新されているかを確認します。

PowerShellでの確認（最も確実）
スタートボタンを右クリックし、**「ターミナル（管理者）」または「PowerShell（管理者）」**を開きます。

以下のコマンドをコピーして貼り付け、Enterキーを押します。

PowerShell
[System.Text.Encoding]::ASCII.GetString((Get-SecureBootUEFI db).bytes) -match "Windows UEFI CA 2023"
結果が「True」: 対応済みです。

結果が「False」: まだ更新されていないか、準備が整っていません。

レジストリでの確認
Win + R キーを押し、regedit と入力して実行します。

以下のパスを確認します。
HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Control\SecureBoot\Servicing

UEFICA2023Status という値が 「Updated」 になっていれば完了しています。

3. 対応が必要な場合の手順
多くの個人向けPCでは、Windows Updateを通じて段階的に自動適用されますが、以下の3ステップを意識してください。

ステップ1：最新のWindows Updateを適用する
Microsoftは月例更新プログラムを通じて、新しい証明書を配布しています。まずは最新の状態に保つことが第一歩です。

ステップ2：BIOS（UEFI）を更新する（重要）
証明書の情報はPC本体のマザーボード（BIOS/UEFI）に保存されます。

Dell、HP、富士通、NECなどのメーカー公式サイトから、お使いの機種の最新BIOSアップデートが出ていないか確認し、適用してください。

注意: BIOS更新を行う際は、念のため「BitLockerの回復キー」をメモしておくことを強く推奨します。

ステップ3：手動で更新を促す（上級者・企業向け）
自動更新を待てない場合やIT管理者の場合は、以下のレジストリを設定することで、システムに更新を促すことができます。

レジストリパス: HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Control\SecureBoot

名前: AvailableUpdates

値（DWORD）: 0x5944

設定後、タスクスケジューラから \Microsoft\Windows\PI\Secure-Boot-Update を手動実行し、再起動します。

4. 注意点：2017年以前の古いPCについて
2017年以前のモデルや、Windows 11の要件を満たしていないPCの場合、メーカーからのBIOSアップデートが提供されず、新しい証明書に対応できないケースがあります。その場合、2026年6月以降はセキュリティ維持のためにPCの買い替えを検討するタイミングとなります。

Gemini の回答
結論から申し上げますと、エラーコード「1801」の確認は非常に重要です。

このエラーは、PCの起動時（POST画面）やWindowsのイベントビューアーに表示されるもので、**「セキュアブートの署名リスト（DB/DBX）を更新しようとしたが、マザーボードの空き容量が足りなくて書き込めなかった」**ことを意味します。

これが解決できないと、2026年6月の期限に向けた更新が完了しません。

1. なぜ「1801」が問題なのか？
セキュアブートの新しい証明書（2023年版）を適用するには、マザーボード上の「NVRAM」という非常に小さな保存領域にデータを書き込む必要があります。

原因: 古いPCや一部の設計のモデルでは、この領域がいっぱいで、新しい証明書を追加するスペースがない場合があります。

結果: Windowsが更新を試みてもエラー「1801」で失敗し、古い証明書のまま取り残されます。

2. 1801エラーが出ているかの確認方法
Windows上で、実際に更新が失敗していないかログを確認するのが最も確実です。

イベントビューアーで確認する手順
スタートボタンを右クリックし、**「イベントビューアー」**を選択します。

左側のツリーから [Windows ログ] ＞ [システム] を開きます。

右側の「現在のログをフィルター」をクリックし、イベントソースに 「TPM-WMI」 または 「SecureBoot-Update」 を指定します（あるいはイベントID 1801 で検索します）。

内容を確認:

「Secure Boot 変数の更新に失敗しました。エラーコード: 1801」といったメッセージが出ていれば、容量不足で更新が止まっています。

3. エラー「1801」が出た場合の対処法
もしエラーが出ていた場合、手動でどうにかするのは難しく、以下のメーカー公式の手段を頼ることになります。

BIOS（UEFI）のアップデート（最優先）
メーカーは「1801」エラーを回避するために、NVRAMの整理や領域拡張を行うBIOSアップデートを配布しています。これを適用することで、書き込みスペースが確保されます。

不要なセキュアブート変数の削除
一部の高級マザーボードでは、BIOSメニューから「Restore Factory Keys」や「Clear Secure Boot Keys」を実行してリセットすることで、ゴミデータが消えて更新可能になる場合があります（※設定を間違えると起動しなくなるため、メーカーの指示書がある場合のみ行ってください）。

True なら: 1801エラーは発生せず、無事に更新されています。

Gemini の回答
PowerShellの結果が True（「Windows UEFI CA 2023」が存在する）にもかかわらず、イベントログに 1801エラー が記録されているという状況は、一見矛盾しているように見えますが、技術的に起こりうる「移行途中のトラブル」です。

主な原因は、**「新しい証明書の書き込みには成功したが、付随するDBX（拒否リスト）の更新や、NVRAMの整理プロセスで容量不足が発生した」**というケースがほとんどです。

1. なぜ「True」なのに「1801」が出るのか？
セキュアブートの更新は、単一の作業ではなく、複数の変数（db, dbx, KEKなど）を書き換える一連のプロセスです。

一部の成功: db（許可リスト）への「Windows UEFI CA 2023」の書き込みはギリギリ成功した。

一部の失敗: しかし、その直後に実行される dbx（脆弱性のある古いソフトをブロックするリスト）の更新時に、NVRAM（マザーボードのメモリ）の空き容量が完全に底を突き、1801エラー（EFI_OUT_OF_RESOURCES） を吐いた。

つまり、**「新しい証明書は入ったけれど、セットで行うべきセキュリティ情報の更新が不完全な状態」**である可能性が高いです。

2. 考えられる具体的な原因
NVRAMの断片化（ゴミデータの蓄積）
長年の使用や過去のOSアップデートにより、マザーボード上の小さな保存領域（NVRAM）に不要なログや古い変数が溜まっており、新しい大きなデータ（今回の更新は従来よりサイズが大きいです）が入りきらない。

OEM（メーカー）固有の予約領域不足
PCメーカーが独自の診断ツールや設定をNVRAMに大量に保存している場合、Windows Updateが使える領域が極端に少なくなっていることがあります。

書き込みプロセスの競合
Windowsがバックグラウンドで更新を何度もリトライしており、過去の失敗ログが残っているだけ（最新の状態では成功している）というパターン。

3. 今後の対処法
「True」が出ているので、2026年6月にいきなり起動しなくなるリスクは低いですが、エラーが出続けているのは健全ではありません。以下の手順を試してください。

① 再起動して再度ログを確認
一度PCを完全にシャットダウンし、再起動した直後のイベントログを確認してください。もし「True」かつ「新しいエラーが出ていない」のであれば、過去の失敗記録が残っているだけなので無視して構いません。

② BIOSの「Secure Boot Keys」をリセットする（推奨）
これが最も効果的です。

BIOS（UEFI）設定画面に入ります。

「Secure Boot」の項目を探し、「Restore Factory Keys（工場出荷時のキーを復元）」 を実行します。

これによりNVRAMが整理され、Windowsが改めて「2023年版証明書」を正しく、かつ完全な形で書き込み直せるようになります。

注意: BitLockerを使用している場合は、必ず回復キーを手元に用意してから行ってください。

③ メーカー製ファームウェア更新ツールを使う
DELLなら「SupportAssist」、HPなら「HP Support Assistant」など、メーカー公式のツールを使ってBIOSが最新か再度確認してください。1801エラー対策として、**「NVRAMのガベージコレクション（ゴミ清掃）機能」**を強化したBIOSが配布されている場合があります。

追加セクション：セキュアブートの基礎知識
1. セキュアブートを一言でいうと？
「PCの起動時に、怪しいプログラムが紛れ込んでいないかチェックする門番」

OS（Windowsなど）が立ち上がる前の「一番無防備な瞬間」を狙うウイルス（ルートキット等）から守る仕組み。

2. どんな仕組みで守っているのか？
**「信頼の鎖（チェーン・オブ・トラスト）」**の解説

マザーボード（UEFI）が、読み込もうとするプログラムの「署名（印鑑）」をチェック。

OK： 信頼できる署名がある → OSを起動。

NG： 署名がない、または改ざんされている → 起動をブロック！

3. なぜ「証明書」に期限があるのか？
技術の進歩により、古い署名技術ではセキュリティを突破されるリスクが出てくる。

定期的に「より強固で新しい署名（証明書）」にバトンタッチする必要がある。

今回の2026年6月問題は、その「バトンタッチの期限」が迫っているという話。

4. ユーザーにとってのメリットと注意点
メリット： PCの乗っ取りや、偽OSの起動を未然に防げる。

注意点： 期限切れのまま放置すると、将来のセキュリティ更新プログラムが「偽物（署名が古い）」と判定されて適用できなくなる可能性がある。

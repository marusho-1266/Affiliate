## 1. 派生とは何か — 「本稿を何度も書かない」ための発想

1本の記事を書き切ったあとに陥りがちな失敗があります。**Twitter（X）投稿・ラジオ台本・サムネ文言を、それぞれ「また1から」書き直す**ことです。ここで時間が無限に溶けます。

**派生**とは、**本稿を「原材料」と見て、そこから別フォーマットに加工する**発想です。

| 本稿 | → | ラジオ台本 | → | SNS連投 | → | サムネ文言 |
|------|----|-----------|----|--------|----|-----------|
| 論旨とエビデンス | | 声で伝わる語順 | | 1行で刺さる断片 | | 表紙1枚で伝わる核 |

**肝は、本稿の中に「派生の種」がすべて入っていること**です。本稿がしっかり書けていれば、派生は**抜き出し・並び替え・語尾直し**でほぼ成立します。逆に、本稿がフワッとしていると、派生するたびに「言い直し」が発生して時間が倍々に増えます。

**派生で初めてアイデアが出る、という状況は避ける**。アイデアは本稿に戻して追記する。これが派生の一番大事なルールです。

---

## 2. 派生の基本形 — 「本稿がブレなければ派生もブレない」

派生Commandは、**本稿ファイルを入力として受け取る**形が基本になります。第2話で学んだ Commandの4パート構成（依頼文／入力／出力の型／ルール）を、派生の文脈に当てはめると次のようになります。

| パート | 派生Commandでの中身 |
|--------|---------------------|
| **依頼文** | 「本稿から〇〇（ラジオ台本／SNS連投など）を生成」 |
| **入力** | 本稿ファイルのパス、またはチャットに貼られた本稿テキスト |
| **出力の型** | 派生フォーマットの構造（ラジオなら「フック／本題／まとめ」など） |
| **ルール** | **本稿にない情報を足さない**／語尾・媒体の作法／禁止事項 |

**特に重要なのが「本稿にない情報を足さない」**ルールです。派生する過程で AI が気を利かせて補完すると、**本稿と派生で事実がズレる**事故が起きます。ここは強めに書いておきます。

そして、第3話で作った **editorial-voice**（語尾・禁止・帰属の Skill）は、派生Commandにもそのまま効きます。Skill の一番の果実が見える瞬間です。

## 3. 最小の派生Command — SNS1枚の文案

理屈より動かした方が早いので、最小の派生Commandを1本作ります。題材は **SNS1枚の文案**（X / Threads 想定）にします。

### やること

1. `.cursor/commands/social-post.md` を作る。
2. 本稿を書いたら、チャット欄で `/social-post` と打ち、続けて本稿の URL またはファイルパスを貼る。
3. 140〜500字の投稿文案が返ってくる。

### `.cursor/commands/social-post.md`

```markdown
# 本稿からSNS1枚の文案を作る

以下の入力（本稿）をもとに、**SNS 1投稿分の文案**を1〜2案、Markdown で出力してください。

## 入力
本稿の本文（Markdown または URL）。入力の境界は Command 呼び出し直後の `---` 以降。

## 出力の型
案ごとに次の3ブロックで出す。

1. **1行フック**(30字以内) — スクロールを止める1文
2. **本文**(200〜400字) — 何が起きたか／なぜ重要か／1行の示唆
3. **ハッシュタグ案**(2〜3個) — 題材固有のもの。汎用タグ（#AI など単独）は避ける

## 書き方のルール
- 本稿にない事実・数字・固有名詞を**絶対に足さない**。迷ったら本稿の表現をそのまま使う。
- 「すごい」「神」のような強調語は使わない。
- リンクの明記が必要な媒体なら、末尾に「→ 詳細（本稿リンク）」の1行を足す。
- 筆者の意見を書く場合は「私の解釈では」と明示する。

## 仕上げ
末尾に **「この投稿が伸びなかったときに、どう書き直すかの1行」**を出力する（自己検証用メモ）。
```

### 使い方

```
/social-post
---
本稿の本文をここに貼る（または URL を1行で貼る）
```

**これだけ**で、本稿の語尾と禁止事項を守った SNS 文案が返ります。もし editorial-voice Skill を入れていれば、他の派生Commandを足したときも**同じ語尾と禁止**が自動で効きます。「一度作れば他にも効く」という Skill の価値が、派生で一番わかりやすく実感できます。

## 4. 題材が尽きない仕組みの考え方（候補ログのミニ版）

派生で1本を広げても、**次の1本**が書けなければ連載は続きません。ここで効くのが **候補ログ**という仕組みです。

### 候補ログとは

「**書きたい題材・書くかもしれない題材を、1ファイルに溜めておく場所**」のことです。思いついたタイミングで1行ずつ足していき、書く直前に「今日はこれ」と選ぶ。**毎回ネタを絞り出さない**ための道具です。

### 最小の候補ログ（コピペで使える）

手元のフォルダに `reports/candidates/backlog.md` のようなファイルを1枚作り、次の形で埋めていきます。

```markdown
# 執筆候補ログ

## 書きたい（優先度: 高）
- [ ] 〇〇の新サービス発表 — 公式プレス: URL — 気づいた日: 2026-04-21
- [ ] 〇〇業界の規制動向 — 初出: URL — 気づいた日: 2026-04-19

## 書くかも(優先度: 中)
- [ ] 読んだ本『〇〇』の所感

## 温めておく（優先度: 低・続報待ち）
- [ ] 先月の〇〇発表の3か月後レビュー — 観測窓: 2026-07ごろ
```

### これだけで何が変わるか

- **「今日は何を書こう」の10分**がほぼゼロになる。
- 「3か月後に続報を書こう」と思っていたものが**勝手に出てくる**（温めておく欄）。
- **重複**（先週書いた題材とほぼ同じもの）に気付ける。

ここまでの仕組みがあれば、**書き切った直後の達成感が、次の執筆の重さを下げる**方向に働きます。書くことが続く状態を作るのが、候補ログの役目です。

**この章は「仕組みの入口」までです。** 鮮度の判定・重複の検知・優先度の入れ替え、といった運用の勘所は有料パートで扱います。

## 4-末. ここまでで出来ること / 出来ないこと

無料パートの内容で、以下は到達できます。

- **1本の本稿から SNS 文案を派生できる**（最小Command 1本）。
- **候補ログ1枚**で「次に何を書くか迷う10分」を圧縮できる。
- editorial-voice などの Skill が、**Commandをまたいで効く**ことを体感できる。

一方で、次の壁が残ります。

- ラジオ台本・SNS連投・サムネ文言を**まとめて派生**する設計が具体化できない。
- 候補ログが**溜まりすぎて**「結局どれも書かない」状態になる。
- **連載自体が腐る**（Cursor の仕様が変わる／題材が古くなる／自分が飽きる）ときの手当てが分からない。

ここから先の有料パートで、**連載を止めずに走らせる仕組み**まで一気に引き上げます。

## 5. 本格的な派生の設計図 — Command実例3本（ここから有料想定）

最小版の `/social-post` を手に入れたら、派生は**あと3本**で基本形が揃います。ラジオ台本・SNS連投・サムネ文言。**この3本を持つと、1本の本稿が「1回の発信」から「1週間の発信」に変わります**。

### 5-1. `.cursor/commands/radio-script.md`（ラジオ／ポッドキャスト台本）

**用途:** 本稿を声に出して読む台本に変換。文字媒体にはない「語順」「息継ぎ」「聞き返せない前提」の制約を入れる。

```markdown
# 本稿からラジオ／ポッドキャスト台本を作る

以下の入力（本稿）をもとに、**1回8〜12分の音声台本**を Markdown で作成してください。

## 入力
本稿の本文（Markdown）。入力の境界は Command 呼び出し直後の `---` 以降。

## 出力の型（見出しの順序を守る）
1. **オープニング（30秒）** — 本日の主題を1文で宣言
2. **今日扱うトピックの位置づけ（1分）** — なぜ今この話題か
3. **事実の整理（3〜4分）** — 誰が・いつ・何をしたか。数字と日付は本稿から引用のみ
4. **読み方の筋道（3〜5分）** — 本稿の観察ベースの部分を、声で辿れる順に並べ替え
5. **持ち帰り（1分）** — リスナーがこの後1つだけするなら何か
6. **クロージング（30秒）** — 次回予告または関連本稿の案内

## 書き方のルール
- **本稿にない事実を補わない**。迷ったら本稿の該当節を引用。
- **1文は40字以内**（聞き取りやすさの目安）。接続詞で文を長くしない。
- 固有名詞・略語は**初出時にフルネーム**で読み上げるよう `（フルネーム）` を併記。
- 数値は**桁を必ず読み下す**（「10億ドル＝10億ドル、およそ1500億円」等、本稿に換算がある場合のみ）。
- 聞き手への問いかけは**1回まで**。多用すると間延びする。

## 書かないこと
- 音響効果の指示（BGM、ジングル、SE）
- 「今日もよろしくお願いします」のような定型挨拶
- 出演者の架空名

## 仕上げ
末尾に **「この回の発信タイトル候補（3案）」**と **「本編から派生するSNS投稿1枚の案」**を添える。
```

### 5-2. `.cursor/commands/social-thread.md`（SNS連投 — X / Threads など）

**用途:** 本稿を **3〜7投稿の連投**に分解。1枚Tweetの `/social-post` とは**別の粒度**で使い分ける。

```markdown
# 本稿からSNS連投を作る

以下の入力（本稿）をもとに、**3〜7投稿の連投**を Markdown で作成してください。

## 入力
本稿の本文（Markdown または URL）。入力の境界は Command 呼び出し直後の `---` 以降。

## 出力の型（番号付きで、各投稿140〜300字）
1. **投稿1: フック** — スクロールを止める1文＋「この後◯投稿で〜」の予告
2. **投稿2〜n-1: 本論** — 本稿の観察ベースの節を1節＝1投稿で要約
3. **最終投稿: まとめ＋案内** — 持ち帰り1行＋本稿リンク＋ハッシュタグ2〜3

## 書き方のルール
- **本稿にない事実を足さない**。要約でも数字は原文のまま。
- 1投稿1論点。複数論点を詰めると連投の意味がなくなる。
- 「ここからが本題」などの**つなぎ語は最小限**。連投は順序自体が文脈。
- **判断の文**（筆者メモ系）は連投に入れない。観察ベースで完結させ、判断は本稿の有料側に閉じる。

## 書かないこと
- スレッドへの自己リプ誘導（「続きはリプで」など媒体固有の作法以外の煽り）
- 「いいねリポストお願いします」の類

## 仕上げ
末尾に **「この連投の最終投稿から、どの本稿節にリンクさせるか」**の指示（本稿内の節見出しを記載）。
```

### 5-3. `.cursor/commands/thumbnail-copy.md`（サムネ文言・画像1枚用）

**用途:** 本稿1本に対して、サムネ画像（note のアイキャッチ／SNSのカード画像など）に**載せる文言**を作る。画像生成AIへのプロンプトと区別する。

```markdown
# 本稿からサムネ文言を作る

以下の入力（本稿）をもとに、**サムネ画像に載せる文言**を案出ししてください。

## 入力
本稿の本文（Markdown）。入力の境界は Command 呼び出し直後の `---` 以降。

## 出力の型
3案を次の形式で出す。

1. **主題コピー**（10〜18字、1〜2行）
2. **副題**(任意、15字以内)
3. **タグ帯**(記事ジャンルを示す2〜4字、例: 企業経営／AI動向／セキュリティ)

## 書き方のルール
- **本稿のタイトルと重複しない**。サムネは「本稿を知らない人を止める」役割。
- **数字や固有名詞**を1つ以上入れる（ただし本稿にあるものに限る）。
- **疑問形・煽り形**（「本当に?」「衝撃」など）は使わない。
- 縦書きは想定しない（横書きを前提とする）。

## 書かないこと
- 具体的な色・フォント・レイアウトの指定（ここはデザイン側に譲る）
- 画像生成AI向けのプロンプト

## 仕上げ
3案それぞれに **「どの読者層に刺さる想定か」**を1行で付記。
```

**この3本と、無料パートの `/social-post` を合わせると、1本の本稿から「ラジオ／連投／1枚SNS／サムネ」の**4フォーマットが派生**できます。**本稿を書き切った直後の1時間以内**にこれらを動かすのが理想。日が経つと、本稿の記憶が薄れて派生の精度が落ちます。

## 6. 候補ログの運用 — 鮮度ポリシー・重複排除・粒度（有料）

無料パートで作った `backlog.md` を、**腐らないログ**に育てるための運用規約です。

### 6-1. 鮮度ポリシー（古くなった題材を自動で剥がす）

候補は溜めるだけだと**「書かない理由」が蓄積**されていきます。鮮度のルールを最初に決めておきます。

| 優先度 | 保存期間の目安 | 超えたらどうするか |
|--------|---------------|-------------------|
| **高（次に書く）** | 2週間 | 書けなければ「中」に降格 |
| **中（書くかも）** | 1か月 | 書けなければ「温めておく」に降格 or 削除 |
| **温めておく** | 観測窓に紐づく（例: 3か月後） | 観測窓を過ぎたら削除 or 別ログへアーカイブ |

**肝は「降格ルール」を明記すること**です。「削除するかどうか毎回迷う」状態を避けます。

### 6-2. 重複排除 — 「先週書いたのと同じ題材」を防ぐ

候補ログには、**書き終わった題材のインデックス**も別節で持ちます。

```markdown
## 書き終わった題材（直近90日）
- 2026-04-14: 〇〇社の規制対応 → note/20260414/...
- 2026-04-17: 〇〇AIの大型更新 → note/20260417/...
```

新しい候補を足すときに、**ここを1回だけ目で走る**。同じ題材を重複で拾っていないか、続報として出すなら**「前回との差分」が明確か**、の2点を確認します。

### 6-3. 粒度 — 「1行で書ける題材」は候補にしない

候補は **「1本の本稿になる題材」**に絞ります。目安は次。

- 本稿で **800字以上**書けるか（薄いとSNS1投稿で終わる）
- **一次情報の出典**が最低1つあるか（ない題材は候補以前の「メモ」に残す）
- **観察と判断の両方**を書ける題材か（片方だけなら無料記事、両方あれば有料も射程）

粒度の合わない題材を候補に残すと、**「結局書けない」山**ができ、ログを見るのが嫌になります。粒度ゲートが、候補ログを使い続けるための隠れた要石です。

## 7. 連載を腐らせない更新の回し方（有料）

仕組みを作ると、必ず**「仕組み自体の賞味期限」**が来ます。Cursor の UI が変わる、モデルが更新される、自分の編集方針が進化する。これに耐える運用の勘所を4つ。

### 7-1. **「連載そのもののバージョン」を書く**

本連載の場合、各話の末尾に `*作成: 2026-04-21（第◯話ドラフト）*` のように日付を入れています。これだけで、**半年後の読者に「いつの情報か」**が伝わります。大きな仕様変更があったら、**各話の冒頭に「改訂ノート」節**を追記する形を最初に決めておきます。

### 7-2. **UI 依存の記述を最小化する**

「〇〇ボタンを押す」のような画面依存の表現は、**UI変更で一斉に陳腐化**します。本連載が**ファイル配置と中身**を主戦場に据えているのは、この陳腐化を遅らせるためです。自分の記事でも、**手順は画面ではなくファイル操作**に落とすと腐りにくくなります。

### 7-3. **Skill の改訂メモを必ず残す**

第3話で扱った Skill の末尾には `## 改訂メモ` 節を作り、**変更のたびに1行追記**します。

```markdown
## 改訂メモ
- 2026-04-21: 初版。Command 側から禁止・語尾・帰属ルールを引き上げ。
- 2026-05-15: 「動機の憶測を書かない」を追加（読書メモ3本で同じ修正が必要になったため）。
- 2026-07-02: 1文の最大字数を80→70に調整。
```

**「なぜ変えたか」を1行で書く**のが要点です。これを残さないと、半年後の自分が Skill を直せなくなります。

### 7-4. **「走らせ直す日」を週に一度決める**

仕組みが腐る最大の原因は、**「使い続けていない」**ことです。週に1回（例: 日曜夜の30分）、

- 候補ログの降格・削除
- Command の出力を1本流して、型崩れがないか確認
- Skill に改訂が必要か1分だけ考える

この30分を確保すると、**連載の寿命が1年単位で伸びます**。逆に、ここをサボると、気づいたときには「もう使えないフォルダ」になっています。

## 10. 連載運用の空欄テンプレ（有料）

自分の連載／長期運用のための**設計用ワークシート**です。これを埋めてから運用に入ると、3か月後に**仕組み自体が残っている**確率が大きく上がります。

```markdown
## 個人の執筆パイプライン運用シート

### 0. 連載の骨格
- 連載名:
- 1本の想定読了時間:
- 想定する更新頻度（週◯本／月◯本）:
- 連載を止める条件（先に決めておく）:

### 1. 持っている Commands / Skills
- 本稿系 Commands:
- 派生系 Commands（ラジオ／SNS連投／1枚SNS／サムネ など）:
- 横断 Skills（editorial-voice / output-quality-gate など）:

### 2. 候補ログ設計
- ログの置き場所（ファイルパス）:
- 優先度の段階と降格ルール:
- 書き終わった題材の保存期間（例: 直近90日を見える化）:

### 3. 派生の回し方
- 本稿を書いた直後に回す派生（例: 連投＋サムネ）:
- 本稿公開から24時間以内に回す派生（例: 1枚SNS）:
- 週次で回す派生（例: 過去1週間の振り返りラジオ）:

### 4. 更新の回し方
- 週1回の「走らせ直し」の曜日と時間:
- 月1回の Skill 改訂レビュー日:
- 四半期ごとの連載インデックス棚卸し日:

### 5. 腐敗の早期シグナル
- 「これが起きたら仕組みを見直す」リスト（例: 同じ追撃指示を3週続けて書いている／週次走らせ直しを2週続けて飛ばした 等）:
```

このシートを埋めた時点で、**連載運用の7割は形になっています**。残りは回し続けるだけ。回し続けるための仕組みが、このシートそのものです。

## まとめ — 第5話

- **派生**は、本稿を原材料に「抜き出し・並び替え・語尾直し」で別フォーマットを作る発想。**本稿がブレなければ派生もブレない**。
- 派生Commandは **「本稿にない事実を足さない」**を強めに書く。editorial-voice などの Skill が派生にも自動で効くのが、Skill の一番の果実。
- **候補ログ**は「書きたい／書くかも／温めておく」の3段＋**降格ルール**＋**書き終わった題材の直近インデックス**で腐らない。
- 連載を腐らせない運用は、**バージョン明記・UI依存の最小化・Skill改訂メモ・週1の走らせ直し**の4つ。
- **連載運用シート**を埋めてから回すと、3か月後に仕組みが残っている確率が上がる。

## 連載全体の締め — 5話を通じて得たもの

本連載の最初に書いた**「明日の自分が1分でも早く『本題に入れる』仕組み」**まで、ここで一通りの輪が閉じました。振り返ると、5話で積み上げてきたのは次の4層です。

1. **作業場の捉え直し**（第1話）— Cursor を「資料とチャットがつながる作業場」として使う。
2. **段取りの文字化**（第2話）— Commands で「毎回同じ型で立ち上がる」を作る。
3. **ルールの横断化**（第3話）— Skills で「共通の作法」を Command から引き剥がす。
4. **深さの設計**（第4話）— サブエージェントと観察／判断の切り分けで、**有料に値する筆者メモ**を書く。
5. **派生と運用**（本話）— 1本を複数フォーマットに広げ、連載として**止めずに走らせる**。

**ここまで揃っても、「最後に自分の言葉で整える作業」は残ります。** 本連載は、その手前まで**速く・ブレなく・判断の深さ付きで**到達するための仕組み作りでした。完全自動を目指さなかったのは、**最後の一手が人間に残るからこそ、書き続ける意味がある**と考えているからです。

あなたの題材は、私の題材とは違います。**Commands と Skills と候補ログの「置き場所」は真似できても、「中身」は真似できません**。ここから先は、あなた自身の仕事の型を、1日1行ずつファイルに預けていく作業です。半年後、1年後、同じ仕組みで走り続けているあなたが、**「書くのが少し軽くなった」**と感じていれば、本連載の役目は終わりです。

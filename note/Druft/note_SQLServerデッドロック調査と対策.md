# SQL Server でデッドロックが多発したときに見るべきこと（調査の進め方と対策の優先順位）

ある朝から業務システムが **妙に重い**。操作が **途中で止まる**。DB まわりのログには **エラー 1205（デッドロック）** が散見される——という場面は、オンプレの SQL Server を抱える現場では珍しくありません。

この記事は、**実際にそうした事象を追った調査**をもとに、note 向けに **「何があったか」「何をしたか」「何を提案したか」** を書き起こしつつ、特定の会社名やテーブル名に依存しない **汎用の学び** として整理し直したものです。  
現場では **業務が止まるほどの事態** に対し、まず **DB の緊急再起動** で復旧を優先し、**再起動後に動作が安定してから** 本記事に書くような調査・分析を行いました。  
同じ症状が出たときの **手順の型** と **打ち手の優先順位** の参考になれば幸いです。

---

## 何が起きていたか（現象）

次のような報告がありました。

- **時間帯**：平日の業務時間帯の **約30〜40分** に集中  
- **体感**：画面の保存や一覧更新が **遅延** したり、**応答なしに近い** 状態になる  
- **深刻度**：状況によっては **業務が実質止まる** と判断されるレベルまで悪化  
- **エラー**：アプリやログに **デッドロック（1205）** が出る  
- **環境**：クライアントは **複数拠点・複数端末** から同時利用。DB は **SQL Server**、アプリは **.NET（SqlClient）** からストアドを呼ぶ構成  

「たまに1件」ではなく、**短時間に何十〜何百件** とグラフが溜まるような勢いであれば、**単発の不運**ではなく **設計・負荷・分離レベル** が組み合わさった **再現性のある問題** と見てよい水準です。

### まず取った応急処置（復旧優先）

事象が **業務継続の障害** になったため、**緊急で DB サーバー（SQL Server）の再起動** を実施しました。  
再起動は **ロックやセッション状態をリセットし、一時的に詰まりを解消する** ための対応であり、**根本原因の修正ではありません**。あくまで **ユーザーが作業できる状態に戻す** ことを優先した、現場として妥当な **応急手** です。

### 調査は「安定してから」

再起動直後は再接続の集中やバッチの再実行などで **まだ揺れる** ことがあります。  
そのため **再起動後、しばらく運用して動作が安定したことを確認してから**、落ち着いた環境で本記事に書く **デッドロックの本調査（ログの整理・ストアド精読・対策案の整理）** に着手しました。  
分析に使ったデッドロックグラフは、**事象発生時に拡張イベント等で収集・エクスポートしておいたもの** を用いています（再起動でメモリ上の状態は変わっても、取っておいたファイルやテーブル上のログは残せます）。

---

## 何をしたか（調査の進め方）

**インシデント対応の順序**は次のとおりです。  
**① 応急復旧（緊急再起動）** → **② 安定稼働の確認** → **③ 本項以下の調査・分析**。

再起動後、業務が通常どおり回せる状態になったことを確認したうえで、現場では次の順で手を動かしました。

1. **デッドロックグラフ（XML）を手元に集める**  
   今回は **事象発生中に拡張イベント等で収集し、エクスポートしておいたログ** を使用した。平常時や再発監視用には、`xml_deadlock_report` 相当のイベントを一定期間回し、**1イベント＝1件のデッドロック** としてファイルやテーブルに溜める運用がよい（TSV でも可）。

2. **件数と時間帯を押さえる**  
   「いつから」「どのくらいの密度か」を見る。インシデント報告と突合する。

3. **XML から機械的に集計する**  
   手で1件ずつ追うのは現実的でないので、  
   - どの **テーブル／インデックス** が `resource-list` に多く出るか  
   - どの **ストアド／バッチ** が `executionStack` に多く出るか  
   - **分離レベル** が Serializable と Read Committed のどちらが多いか  
   をざっくり数える。ここで **ホットスポット** が見える。

4. **上位のストアドをソースで読む**  
   頻出上位から **全文を読み**、  
   - 広い条件の **SELECT 集計**  
   - **カーソル** や **長いトランザクション**  
   - **採番テーブルの読み→別処理→更新**  
   がないか確認する。

5. **パターンに分類する**  
   「集計と更新のぶつかり」「採番競合」「同一クエリの並列デッドロック」などに **ラベルを付ける**。対策の打ち分けがしやすくなる。

6. **誤解しやすい打ち手を検証する**  
   例：ストアド先頭の `SET TRANSACTION ISOLATION LEVEL READ COMMITTED` だけで、**外側が Serializable のトランザクション**を上書きできるか——は **SQL Server の仕様上そう単純ではない**ので、ドキュメントで確認する。

この流れまで踏むと、「感覚でインデックスを足す」より **筋の通った優先順位** を説明できるようになります。

---

## 調べて分かったこと（要約）

今回のログを整理すると、おおむね次の像でした。

- **分離レベル**：関与セッションの **大半が Serializable**。レンジロック（`RangeS-S` / `RangeI-N` など）が多く、**読みと書きの順序が逆**になりやすい条件がそろっていた。  
- **ホットテーブル**：**明細系の1テーブル**が突出して多く、**主キーインデックス**まわりの競合が目立った。  
- **処理の型**：  
  - 得意先×期間などで **複数テーブルを跨いだ集計**をするストアド  
  - 明細を **カーソルで読み**、発注・仕入まわりを **削除して作り直す** ようなストアド  
  - **連番マスタ**を `SELECT` してから `UPDATE` する採番  
  が同時多発で重なっていた。  
- **別件**：クライアントツールからの **重いアドホック SELECT** が、**同一セッション内の並列プラン**で自己デッドロックしている例も少数あった。

つまり **「Serializable の常用 × ホットテーブル × 重い／長いトランザクション」** が重なった典型に近い、という結論になりました。

---

## デッドロックとは（超要約）

2 つ以上のセッションが、**互いに相手が持っているロックの解放を待っている**状態で、どちらも先に進めないことです。  
SQL Server はそのうち **片方（victim）を選んでロールバック** し、もう片方を進めます。  
被害側のアプリでは **一時的な失敗** や **リトライ** が必要になり、発生が続くと **体感の遅さ・不安定さ** に直結します。

---

## 調査データの取り方（技術メモ）

運用では次のような方法が一般的です。

1. **拡張イベント（Extended Events）** のセッションで `xml_deadlock_report`（または同等のデッドロックイベント）を収集する  
2. 一定期間の XML をファイルやテーブルに溜め、**1 イベント = 1 デッドロック** として分析する  

グラフ XML では、だいたい次のブロックが重要です。

| ブロック | 読み取る内容 |
|----------|----------------|
| `victim-list` | ロールバックされたセッション（アプリ側で失敗した側） |
| `process-list` | 各プロセスの **待ちリソース**、**ロックモード**、**分離レベル**、**実行スタック**、クライアント情報 |
| `resource-list` | **どのテーブル／インデックス** が競合したか（objectname, indexname） |

**分離レベル** が `serializable (4)` のように出ていれば、後述の **レンジロック** を疑う材料になります。  
**ロックモード** に `RangeS-S` や `RangeI-N` が並んでいれば、多くの場合 **Serializable 系の読み取りと、挿入の順序** が絡んだ典型的パターンです。

---

## よくあるデッドロックの型（汎用パターン）

各型の末尾に、**今回の調査で当てはまったか** を一言添えます。

### 型A：重い「読み（集計）」と「書き」が同じホットテーブルでぶつかる

- 一方：得意先・期間などで **広い範囲を SELECT して集計**（ストアドやバッチ）  
- 他方：同じテーブル／インデックスに対する **INSERT / UPDATE**  

**Serializable** だと、読み取り側に **キー範囲ロック** が付きやすく、挿入と順序が逆になるとデッドロックしやすいです。  
待ちリソースが **PAGE** になっている場合は、スキャン範囲が広く **ページ単位の共有ロック** が絡んでいることもあります。

**【今回】** 最多パターン。集計ストアドと明細の更新／発注まわりが同じ明細テーブルで交差していた。

### 型B：同じ処理が同時多発し、同じインデックス範囲で互いに待つ

同じストアドを **複数拠点・複数ユーザーが同時** に叩き、**同じテーブルの PK 付近**でレンジロック同士が交差するパターンです。  
**カーソル** で順に読みながら長くトランザクションを持つと、ロック保持時間が伸び、この型と相性が悪いです。

**【今回】** 同一業務ストアド同士の競合として複数件確認。

### 型C：採番テーブル（連番マスタ）の「読んでから更新」

次のような書き方は、高負荷・高並列だとデッドロックの温床になりがちです。

1. `SELECT MAX(連番)+1 ...` や `SELECT 連番 FROM ...`  
2. 別の処理を挟む  
3. `UPDATE ... SET 連番 = @採番`  

**Serializable** 下では、SELECT 時点のロックと UPDATE 時点のロックの組み合わせが強くなり、**別プロセス同士で順序が逆**になるとデッドロックします。

**【今回】** 発注番号・入金番号など、別種の伝票採番同士のデッドロックとして現れた。

### 型D：1 クエリの並列プランによる「自己デッドロック」

**同一セッション（同一 SPID）** の並列ワーカー同士がロックで待ち合うケースです。  
クライアントツールからの **重いアドホック SELECT** で起きやすく、グラフ上はプロセス数が異常に多く見えます。  
対策は **MAXDOP の抑制**（クエリヒント、サーバー設定のコストしきい値の見直しなど）が典型です。

**【今回】** 少数だが、1セッション内のワーカー数が非常に多いグラフとして検知。

### 型E：ロックエスカレーション（テーブル／オブジェクト単位のロック）

行・ページのロックが増えすぎると **テーブル（OBJECT）へのエスカレーション** が起き、別セッションの **同じテーブルへの更新** と正面衝突します。  
グラフで `OBJECT:` と **SIX** などが出る場合は、この疑いがあります。

**【今回】** 同一明細テーブルの更新同士で、オブジェクトレベルの競合として確認。

---

## 対策を「効き方」と「注意点」で整理する

### 1. 分離レベルを Read Committed に寄せる（根本に近い）

**Serializable** はファントム読み取りを防ぐため **レンジロック** を多用します。  
業務上それが不要な処理まで Serializable にすると、**読みと書きの衝突** が増えます。

**.NET の `TransactionScope` は既定が Serializable** である点に注意してください。  
運用上は **`IsolationLevel.ReadCommitted` を明示** することが安全策になります。

#### ストアド先頭の `SET TRANSACTION ISOLATION LEVEL READ COMMITTED` だけに頼らない理由

SQL Server では、**すでにトランザクションが開始されている状態**で `SET TRANSACTION ISOLATION LEVEL` を実行すると、**新しいレベルは「現在のトランザクション終了後」から有効**になります。  
そのため、**外側が Serializable のトランザクションで包んだまま** ストアドを呼ぶ場合、ストアド内の `SET` だけでは **期待どおりに下がらない**ことがあります。

確実に効かせるには、**トランザクション開始前／アプリ側で分離レベルを決める**ことが重要です。

**【今回の整理】** 「ストアドに1行足すだけ」より **TransactionScope の既定見直し** を最優先メッセージにした。

### 2. READ_COMMITTED_SNAPSHOT（RCSI）を有効にする

データベースで `READ_COMMITTED_SNAPSHOT ON` にすると、**READ COMMITTED の読み取り**が **行バージョン（tempdb）** ベースになり、**従来の共有ロックに依存しにくく**なります。

- **読み**が **書き**をブロックしにくい  
- **書き**が **読み**の S ロックに阻まれにくい  

ため、型A のような **集計と更新の同時アクセス** に効きやすいです。

注意点は次のとおりです。

- 効くのは **READ COMMITTED として動いているセッション**。**Serializable のまま**では恩恵は限定的です。  
- **tempdb（バージョンストア）の使用量・I/O** が増えるため、本番適用前に負荷試験や監視設計が必要です。  
- 有効化には **単一ユーザーモードなどのメンテナンス時間** が必要になることがあります。

**【今回の整理】** アプリ側を Read Committed に寄せたうえでの **セット案** として位置付けた。

### 3. ホットテーブルまわりのクエリとインデックス

広いスキャンは **ページロック** や長い待ちを招きます。  
集計条件（例：得意先＋日付）に合った **複合インデックス** があるかを確認し、**読み込み量とロック範囲を減らす**のは定石です。

**【今回の整理】** 集計ストアドが同じ条件で複数回テーブルを読んでいたため、**インデックス＋クエリ集約**の両面を候補にした。

### 4. 採番を「取ってから更新」から「ロック付きで一発更新」へ

採番行を `UPDATE ... WITH (UPDLOCK, HOLDLOCK)` で直列化する、**SEQUENCE** オブジェクトを使う、`sp_getapplock` でアプリロックを取る、など **競合を設計で潰す**方法があります。

**【今回の整理】** 連番マスタのパターンを **コードレビューで特定**し、改修チケットの候補にした。

### 5. カーソルや「削除して全部入れ直し」の見直し

長いトランザクションほど、他セッションとの **ロックの重なり方** が悪化します。  
可能なら **セットベースの1本の INSERT…SELECT** や、**差分更新** に寄せると、保持時間が短くなります。

**【今回の整理】** 発注書作成まわりの **カーソル＋長い処理** を、中期的な改善候補にした。

### 6. 並列プラン起因（型D）

問題のクエリに **`OPTION (MAXDOP 1)`** を付ける、サーバーの **並列化のコストしきい値** を見直す、などで **ワーカー同士の待ち** を防ぎます。

**【今回の整理】** 該当クライアント／クエリは少数だったが、**再発時の切り分け手順**に含めた。

### 7. ロックエスカレーション（型E）

更新件数や統計の状態によっては `ALTER TABLE ... SET (LOCK_ESCALATION = DISABLE)` を検討しますが、**行ロックが大量に残る**副作用があるため、監視とセットで判断します。

**【今回の整理】** 再現条件がはっきりするまで **様子見＋上位対策後に再評価** とした。

## では何から手を付けるか（今回の優先順位）

「何があったか」を踏まえ、**変更の小ささ対効果**で並べ直すと、次の順が現実的でした。

1. **アプリのトランザクション分離レベル**を見直す（`TransactionScope` の **`IsolationLevel.ReadCommitted` 明示** など）  
2. **READ_COMMITTED_SNAPSHOT** の可否を検証し、問題なければ有効化（**tempdb** の監視設計とセット）  
3. **ホットテーブルのインデックス**と **集計クエリの統合**（I/O とロック時間の削減）  
4. **採番ロジック**の直列化・SEQUENCE 化  
5. **カーソル／全削除再作成**の縮小（中期的な改修）  
6. **MAXDOP** や **エスカレーション制御**（個別事象に応じて）

ここまで決めておくと、会議で **「まず何をやるか」** を説明しやすくなります。  
「ストアドの先頭に1行足すだけ」は **状況によっては効かない**ので、**誰がトランザクションを開始しているか**をセットで確認するのがコツです。

---

## まとめ

- **何があったか**：業務時間帯に **デッドロック多発** → **遅延・不安定化**、**業務が止まる** 状況まで悪化。**緊急で DB 再起動** し復旧を優先。ログ上は **Serializable 比率が高く**、**特定明細テーブル**に競合が集中。  
- **何をしたか**：**再起動後、動作が安定してから**、事象発生時に取っておいたグラフを材料に **件数・時間帯・オブジェクト・ストアドを集計** → **上位ストアドを全文読んで**採番・カーソル・集計の重なりを特定 → **型に分類**（調査中に新規に拡張イベントを回す場合は、その手順に沿う）。  
- **何を提案したか**：**アプリの分離レベル見直し**を最優先、**RCSI** は Read Committed 前提のセット、あわせて **インデックス・採番・クエリ構造** を段階的に改善。  

同様の調査を社内用に残す場合は、**期間・件数・上位オブジェクト・上位ストアド・代表グラフ1件** を添えておくと、次のインシデントで再利用しやすくなります。

読者の環境でも、まずは **「1205 が出たらグラフを1件保存する」** ところから始めると、後から **何があったか／何をしたか** を文章にしやすくなります。

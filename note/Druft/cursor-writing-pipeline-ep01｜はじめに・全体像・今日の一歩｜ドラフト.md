# 「Cursorで整える個人の執筆パイプライン」①｜はじめに・全体像・今日動かせる一歩

「Cursor を開いたけれど、何ができるのか・どこから触ればいいのかで止まっている」という方へ。本連載では、毎日の執筆（記事・メモ・週報・読書ノート）を**毎回同じ品質で・迷わず**立ち上げるための仕組みを、非開発者でも触れる粒度で**5話かけて**解説します。**第1話は全文無料**。読み終える頃には、`/weekly-report` という**最初の Command が1本**、あなたの手元で動いているはずです。

> **この連載について（第1話）**
> 本記事は**有料想定の連載**の第1話です。第1話は**無料**で、連載の地図と「今日動かせる最小の一歩」までを届けます。第2話以降の「Commands の作り方」「Skills の作り方」などは、各話の**後半を有料ブロック**にする予定です。
> **読了目安:** 約7〜10分／**想定読者:** AI に興味があり Cursor を使い始めたばかりで、「どう効率化すればいいか分からない」非開発者。

---

## はじめに — なぜこの連載を書くか

冒頭で触れたように、「何ができるか」で止まっている方は多いと思います。チャットに質問できるのは分かる。けれど、それだけなら他の AI サービスと何が違うのか。使いこなしている人の SNS 投稿は華やかだけれど、自分の日常にどう落ちるのかが見えない。私自身も、最初はそうでした。

本連載の主役は、Cursor の **Commands** と **Skills** と呼ばれる小さな機能です。これらを組み合わせると、**「書くたびに毎回考え直している段取り」を、1つのファイルに固定**できます。1回書いておけば、次からは同じ型が**スラッシュひとつ**で立ち上がります。

難しいコードは書きません。エンジニアでなくても読める粒度に寄せます。ただし、**「完全自動で記事が仕上がる魔法」は約束しません**。最後に自分の言葉に直す作業は残る前提で、**その手前までを徹底して速く・ブレなくする**、というのが本連載の立場です。

## この連載で扱うこと / 扱わないこと

| 扱うこと | 扱わないこと |
|----------|--------------|
| Cursor の **Commands / Skills** を使った執筆フローの組み立て | Cursor でのコード開発の話（主役は執筆） |
| **非開発者でも触れる**粒度でのサンプルと設計 | GitHub を前提とした高度な運用（GitHub は必須ではありません） |
| **無料と有料の境界の引き方**（自分の note に応用できる視点） | 特定ジャンル（海外ITニュース等）に縛られた話 |

**海外ITニュースは、あくまで「この仕組みで書いた具体例の1つ」**として登場します。読書メモ・週報・議事録整理など、他の題材にも同じ仕組みが使えます。

## 前提（短く）

- **Cursor のバージョン・UI は変わります。** 本連載は執筆時点の仕様を前提に書きます。画面より**ファイルの配置と中身**を主に扱うので、UI が多少変わっても読み替えが効きます。
- **GitHub は必須ではありません。** 手元のフォルダ1つから始められます。
- **完全自動は約束しません。** 仕組みは「速さ」だけでなく、**「毎回同じ品質」「迷わない」**ために作ります。最後に人が整える前提で設計します。

## 1. Before / After — 何が変わるか

仕組みを入れる前と後で、執筆時間の使い方は次のように変わります（筆者の体感ベースの例です）。

| 段階 | Before（仕組みなし） | After（Commands + Skills あり） |
|------|----------------------|----------------------------------|
| ① 題材を決める | 毎回ネタ探し、気分に左右される | **候補ログ**に日々溜めておいた中から選ぶ |
| ② 構成を作る | 白紙から見出しを考える | **型が呼び出される**（「まずこの見出しから」が出る） |
| ③ 本文を書く | 用語説明・帰属（誰が言ったか）を都度迷う | ルールがSkillに書いてあるので**揃う** |
| ④ 仕上げ | 見出しの整合・注意書き・リンク整理で時間を使う | **チェックリスト**が呼ばれ、抜け漏れが減る |
| ⑤ 派生（SNS・音声など） | また白紙から書き直す | **同じ本文から台本やSNS文案を派生**できる |

ポイントは**②〜④**です。Cursor を「資料とチャットがつながる作業場」として使うと、この3段階の**「毎回考え直しているはずの判断」**を、ファイルに預けられるようになります。

---

## 2. Cursor を「作業場」と言い換える

Cursor は「コードエディタ」と紹介されがちですが、執筆用途で使うなら **「資料とチャットがつながる作業場」**と捉えるのが正確です。

- **資料**: 手元のフォルダに置いた Markdown ファイル（下書き・メモ・テンプレ・ルール）。
- **チャット**: その資料を見ながら、AIに指示を出す会話欄。
- **つながる**: チャットがフォルダ内のファイルを**読み取れる・書き換えられる**ので、「さっきのメモを参考に、この型で書き直して」が一発で通る。

この「資料を横に置いてAIと話す」感覚が、**ChatGPT や他のチャットUIとの一番大きな違い**です。そして**Commands / Skills** は、この「資料」の中に、**「段取り」や「判断ルール」自体を文字で置いておく**仕掛けです。

## 3. 全体像（1枚の図）

本連載で組み立てる仕組みは、次の5段で表せます。

```
[1] 調査（候補ログを溜める）
       │   （ネタ切れ防止・重複排除）
       ▼
[2] ブリーフ（執筆前の短い企画書を作る）
       │   （背景・目的・効果を整理）
       ▼
[3] 本稿（note 記事などの本文を書く）
       │   （無料と有料の境界をここで引く）
       ▼
[4] 派生（ラジオ台本・SNS文案・サムネ）
       │   （同じ本稿から別フォーマットを生成）
       ▼
[5] 運用（更新・棚卸し・題材補充）
```

- **[1]〜[2] は Commands と Skills の合わせ技**で形にします。
- **[3] が主戦場**。ここで「無料で読めるところ／有料で読めるところ」を分けます。
- **[4] は [3] の派生**。本稿がぶれなければ派生もぶれません。
- **[5] は仕組みの寿命を伸ばす**段階。ここを軽く書くと、毎日続きます。

第1話では、**[1] と [3] に登場する最小の Command を1つ**だけ取り上げます。

## 4. Commands と Skills の違い（1行ずつ）

連載全体で繰り返し出てくる2つの言葉を、ここで整理します。

| 名前 | 置き場所の例 | ざっくり何者か | 使い時 |
|------|--------------|----------------|--------|
| **Command** | `.cursor/commands/〈名前〉.md` | **「こう指示して」と書いた定型の依頼文**。スラッシュで呼び出す（`/〈名前〉`）。 | 同じ指示を繰り返したいとき。1枚で完結する作業。 |
| **Skill** | `.cursor/skills/〈名前〉/SKILL.md` | **「こう考える／こう書く」と書いた判断ルール集**。AIが自動で参照する。 | 複数の Command にまたがる共通ルール、品質チェック、禁止事項。 |

第一印象としては、こう覚えておくと十分です。

- **Commands = 呼び出せる段取り**（レシピ）
- **Skills = 自動で効くルール**（家の作法）

### 迷ったら Command から

最初は**Command 1本**から始めるのが現実的です。Skill は**「Command がいくつか育ってきて、共通のルールを持たせたくなったとき」**に分離するのが自然で、第1話の段階では不要です。

## 5. 今日動かせる最小の一歩 — 週報を書く Command

理屈より動かした方が早いので、最小の Command を1つ作ります。題材は**週報**にします。毎週書く人が多く、形が揃っていればいるほど助かる題材です。

### やること

1. 手元のフォルダに `.cursor/commands/` という空のフォルダを作る。
2. その中に `weekly-report.md` という名前で、次の中身をコピペする。
3. Cursor のチャット欄で `/weekly-report` と打って、今週の出来事を数行貼り付けるだけで、定型の週報ドラフトが返ってくる。

### コピペするファイル（`.cursor/commands/weekly-report.md`）

```markdown
# 週報ドラフトを作る

以下の入力をもとに、**社内共有向けの週報**を Markdown で作成してください。

## 入力
ユーザーが箇条書きで「今週やったこと・所感・来週の予定」を貼り付けます。

## 出力の型
以下の見出しをこの順で必ず使う。箇条書きは1項目1〜2行。

1. **今週のハイライト（3点まで）**
2. **進捗（プロジェクト別）** — 箇条書き
3. **気づき・学び（任意）** — 1〜3点
4. **来週の予定・依頼事項** — 箇条書き

## 書き方のルール
- 主語は「私」にせず、事実ベース（「〇〇を完了」「△△を相談」）で書く。
- 推測や曖昧なものは「要確認」と明記。
- 数字は出典や一次情報があるものだけ書く。なければ書かない。
- 150〜400字の圧縮感を目安にする。

## 仕上げ
最後に、**抜けそうな観点のチェックリスト**を出力の末尾に付ける（例: リスクの共有／来週のブロッカー／他部署への依頼）。
```

### 使い方

チャット欄で次のように打つだけです。

```
/weekly-report
今週やったこと：
- 新人面談を3件実施
- 見積もり提出を2件（A社・B社）
- 社内勉強会の司会
所感：
- A社は来週発注の可能性
- 勉強会の出席率が前回比で上がった
来週：
- B社からの返答待ち
- 月次レビュー資料
```

**これだけ**で、見出しの揃った週報ドラフトが返ります。整えるのは人間の仕事ですが、**「毎回同じ型で立ち上がる」**ことが実感できるはずです。

### この小さな一歩で見えてくること

- **チャットに毎回長々と指示を書かなくていい**ことの価値。
- 「この見出しは要らない」「この項目も足したい」と**ファイルを数行書き換えるだけで**明日から反映される気楽さ。
- **自分の仕事の型を、文字でファイルに預けられる**という感覚。

ここまで来たら、あとは題材を**読書メモ**や**議事録整形**に差し替えるだけで、**Command の数が自然に増えていきます**。

## 6. つまずきやすい3つと、向き合い方

最初の数日で出会いやすい壁を先に書いておきます。

### 6-1. `.cursor/commands/` フォルダが見つからない

Cursor の画面でフォルダが見えないだけで、**手元のフォルダを開けば普通に作れます**（隠しフォルダではありません）。「ファイル＞フォルダを開く」で執筆用フォルダを開き、その中に `.cursor/commands/` を作ります。

### 6-2. スラッシュで呼んでも反応しない

- ファイル名が `.md` になっているか。
- **Cursor を一度再読み込み**すると拾う場合があります。
- 念のため `.cursor/commands/〈ファイル名〉.md` が正しい階層にあるかを確認。

### 6-3. AIが「型」を守ってくれない

- Command のファイル内で**「必ず」「使わない」「〜しない」**といった言葉を使うと、守られやすくなります。
- それでも守ってくれないときは、**出力に対して「型に沿って書き直して」ともう一度言う**のが、今の現実解です。「1回で完璧」を期待しすぎないのが、続けるコツです。

**各話は単独で読めるように書いています。** 第1話を読まずに第3話から入っても、そこに第1〜2話の要点が1段落で再掲される形にしています。

## 8. よくある疑問（短め FAQ）

**Q. コードを書かないのですが、Cursor を使う意味はありますか？**  
A. あります。本連載は**執筆の段取り**を主戦場に置きます。コードを書かない方でも、**「資料が横にある作業場」**としての Cursor は、ChatGPT 単体の会話とは別物の便利さがあります。

**Q. GitHub のアカウントは必要ですか？**  
A. **必要ありません。** 手元のフォルダ1つあれば始められます。共有や履歴管理に興味が出てきたら、後から足せます。

**Q. 他のAIチャット（ChatGPT など）でも同じことができますか？**  
A. 似たことは可能ですが、**「フォルダの中のファイルを横に置いて、AIが読みながら進む」**体験は、Cursor のような**IDE型のAIクライアント**が圧倒的にやりやすい、というのが実感です。

**Q. Cursor の UI が変わったら、この連載は使えなくなりませんか？**  
A. 画面依存の箇所は最小限に抑え、**フォルダ配置とファイルの中身**を主に扱う構成を採っています。UI の変化には比較的強いつくりです。**大きな仕様変更があった場合は、各話冒頭に改訂ノートを追記**します。

**Q. 「Skills」と「Rules（.cursor/rules）」は違うものですか？**  
A. 近い機能ですが、本連載では**Skills のほうに寄せて解説**します。Skills は「ひとまとまりの手続き」としてAIが自動で呼び出せて、第3話以降で扱う「共通ルールの引き剥がし」と相性が良いためです。Rules については、本連載では踏み込まず、必要に応じて Cursor の公式ドキュメントを参照してください。

## まとめ

- Cursor は**コードエディタではなく「作業場」**として捉えると、非開発者の執筆に効いてくる。
- 主役は **Commands（段取りを文字にする）**と **Skills（ルールを文字にする）**。まずは **Commands 1本**から。
- 第1話で作った `/weekly-report` は、**「毎回同じ型で立ち上がる」**体験の最短ルート。題材を差し替えればそのまま増えていく。
- 次回（第2話）は、**Commands の書き方そのもの**を解説します。4パート構成（依頼文／入力／出力の型／ルール）、再現性を上げる5つの勘所、そして **そのままコピペで使える運用版の実例3本**（週報・議事録・深掘り読書メモ）までを渡します。

このまま、**明日の自分が1分でも早く「本題に入れる」仕組み**を、一緒に組み立てていきましょう。

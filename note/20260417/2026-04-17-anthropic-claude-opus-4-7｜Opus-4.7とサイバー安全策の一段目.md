## はじめに

2026年4月16日、Anthropic は公式ブログ **[Introducing Claude Opus 4.7](https://www.anthropic.com/news/claude-opus-4-7)** で、**Claude Opus 4.7** の **一般提供（generally available）** を発表しました。高度なソフトウェア工学・視覚・長時間タスクの改善に加え、**直前に公表した Project Glasswing／Claude Mythos Preview** の文脈と**明示的につなげ**、次の三点を同じ投稿で説明しています。

1. **Opus 4.7 は Mythos Preview より「広く」は強くない** — サイバー能力は抑えた上で、**まずこのモデルで安全策を試す**のが第一段だ、という位置づけ。  
2. **禁止・高リスクのサイバー用途**は、**自動検知してブロック**する安全策を載せた。  
3. **正当なセキュリティ用途**（脆弱性研究、ペネトレーション、レッドチーム等）の利用者には、新設の **Cyber Verification Program** への参加を案内する。

料金は **Opus 4.6 と同水準**（入力 **$5**／百万トークン、出力 **$25**／百万トークン）と記載されています。開発者向けの API 名は **`claude-opus-4-7`**。提供は Claude 各製品に加え、**API、Amazon Bedrock、Google Cloud Vertex AI、Microsoft Foundry** と列挙されています。

本稿の読みどころは次の2点です。

1. **Glasswing で語った「段階的に広げる」**と、**Opus 4.7 GA** が**同じストーリー線**でどう接続されているか。  
2. **「ブロック」と「検証プログラム」**の二層が、現場の**依頼・契約・ログ**にどう落ちるか（一次が言える範囲と、まだ分からない範囲の切り分け）。

## 1. 公表の中身（公式で言えること）

### 1-1. Opus 4.7（2026-04-16）

**[Introducing Claude Opus 4.7](https://www.anthropic.com/news/claude-opus-4-7)**（Apr 16, 2026）の範囲で、少なくとも次が述べられています。

- **Opus 4.6 からの改善**として、高度なソフトウェア工学、**より高解像度の視覚**、長時間タスクでの一貫性・指示追従・自己検証に触れている。  
- **Mythos Preview との関係**: 「**less broadly capable than our most powerful model, Claude Mythos Preview**」と明記したうえで、**Opus 4.6 よりベンチマーク上は良い**と主張。  
- **Glasswing との接続**: 「**Last week we announced Project Glasswing**」とし、Mythos Preview の提供を限定したうえで、**「より能力の低いモデルで先にサイバー安全策を試す」**方針の**最初のモデルが Opus 4.7**だと説明。  
- **学習時の扱い**: サイバー関連能力について **「differentially reduce these capabilities」** する実験をした、と記載（外部検証は別問題）。  
- **安全策**: **禁止または高リスクのサイバー利用**を示す依頼を**自動検知してブロック**する、と説明。  
- **Cyber Verification Program**: 正当なセキュリティ用途で使いたい人向けに **[申請フォーム](https://claude.com/form/cyber-use-case)** へ誘導。  
- **提供・価格・API 名**: 上記はじめにのとおり。

### 1-2. Project Glasswing（前段・2026-04-07 前後）

**[Project Glasswing](https://www.anthropic.com/glasswing)** は、Mythos Preview とパートナー連携、クリティカルソフトの防御、クレジット・寄付のコミットなどを説明する**前提稿**です。Opus 4.7 投稿はこれを**「先週」**として参照しています。

## 4. 短期・中期・長期の整理（定義と併記）

**時間軸の定義（本稿内）**

- **短期**: 0〜3か月  
- **中期**: 3か月〜1年  
- **長期**: 1年以上  

**短期** — Opus 4.7 への切り替え、ツール連携の追従、Cyber Verification の**申請増**などが起きやすい、という読み方ができます。**不確実性**は、自動ブロックの**偽陽性・偽陰性**や、検証プログラムの**審査能力**は実データがないと読みにくい点。**効果が薄い条件**は、現場が **4.6 固定**のまま、または**別モデルへ評価だけ移す**こと。

**中期** — 公式が述べる **「Mythos-class models のより広いリリース」**に向け、安全策の知見を**ポリシーと実装に反映**していく段階、と読めます。**不確実性**は規制・訴訟・政治文脈。**効果が薄い条件**は、ブロックが**正当利用まで阻害**し、プログラムが**信頼されない**と判断されること。

**長期** — 「強いモデル＝サイバー用途は**ゲート付き**」が業界の**デフォルト設計**として並ぶプレッシャー（他社の TAC／サイバー向け派生などと**並行**して語られることがある。推測成分あり）。**不確実性**はオープンウェイト・オンプレ回帰・地政学。**効果が薄い条件**は、攻撃側の**回避技術**が先行し、ガバナンスコストだけが増えるパターン。

以下は **Anthropic 公式等の公開情報を超える推論**を含みます。**投資判断・ベンダ選定・契約条項の解釈の代わりにはなりません。** 必要なら社内の**法務・コンプライアンス・調達**に持ち込んでください。

---

### 7-1. 含意 — 「ブロック」と「検証プログラム」が組織に落ちるとき

公式の二層構造は、組織ではしばしば次のように**別オーナー**に割れます。

1. **プロダクト／開発** — ブロックで止まった依頼の**再現手順**、代替モデル、プロンプトの言い換え。  
2. **セキュリティ／GRC** — Cyber Verification の**申請者・範囲・証跡**（誰が、どの契約の下で、どの環境で）。  
3. **法務** — ペネトレーション等の**許可文書**と、クラウド経路の**禁止行為の定義**の整合。

「モデルが賢くなった」話と「**誰が鍵を持つか**」の話が同時に来る局面だと捉えると、後工程の混乱が減ります。

---

### 7-2. 他社との並べ方（参照のみ）

OpenAI 側では **Trusted Access for Cyber（TAC）** や **GPT-5.4-Cyber** など、**身元・ティア・段階配布**の語彙で防衛向けを説明しています（別日・別公式投稿）。**「どちらが強いか」より先に**、自社が触るチャネルで **誰が使えてログがどう残るか**を表にするのが安全です。

### 7-3. 反論・限界（筆者メモ）

- **偽陽性**: 正当なリファクタやセキュリティ教育まで**止める**と、現場は別経路に逃げる。  
- **偽陰性**: ブロックが**見せかけ**に見えると、コンプラは「モデル任せ」を信じない。  
- **形骸化**: Verification が**紙だけ**になると、後から監査で詰まる。  
- **ジェイルブレイク・蒸留**: ガバナンスは**入口**に過ぎず、**出口**（ログ・人間の承認・データ持ち出し）を捨てると意味が薄い。

---

### 7-4. 現場打ち手の優先順位（筆者メモ）

| 優先 | 打ち手 |
|:----:|--------|
| 1 | **禁止行為の定義**（社内規程・顧客契約・クラウド利用規約）を、**「ブロックされる依頼」**の例と突き合わせる。 |
| 2 | **Cyber Verification を誰が申請するか**（個人アカウントか法人か、証跡の保管場所）を決める。 |
| 3 | **切替日**（Opus 4.6→4.7）と**ロールバック条件**（品質・コスト・ブロック率）をテンプレ化する。 |

## はじめに

**2026年4月16日**、OpenAI は公式ブログ **[Codex for (almost) everything](https://openai.com/index/codex-for-almost-everything/)** で、開発者向け **Codex** の大型アップデートを公表しました。同日の記事では **「Starting today」** とし、**Codex デスクトップアプリ**に **ChatGPT** でサインインしているユーザーへ、更新の**段階的な提供（rolling out）**が始まる、と説明されています。

## 1. 何が公表されたか（公式の範囲で）

以下は、**同一の公式記事**に照らして整理した機能・メッセージです。

### 1-1. コーディングの外側へ（computer use・ブラウザ・画像・プラグイン）

- **バックグラウンドの computer use:** エージェントが画面を見て **クリック・入力**し、**複数のエージェントを Mac 上で並行**させつつ、利用者の他アプリの作業と干渉しない、と説明。**macOS で先行**し、**EU・UK** 向けは **soon** と記載。  
- **アプリ内ブラウザ:** ページに**コメント**してエージェントへ精密な指示を渡せる。**フロントエンドやゲーム開発**での反復に有用とし、**localhost 以外の Web を含めブラウザを完全に指揮する**展開は **over time（今後）** に広げる計画、と記載。  
- **gpt-image-1.5:** 画像の生成と反復。スクリーンショットやコードと組み合わせ、プロダクト案・UI・モック・ゲーム素材など**同じワークフロー内**での利用を想定。  
- **プラグイン90件超の追加:** skills・アプリ連携・**MCP サーバー**を束ねる、との説明。うち公式ブログで具体名が挙がっているのは **Atlassian Rovo（JIRA 管理向け）、CircleCI、CodeRabbit、GitLab Issues、Microsoft Suite、Neon by Databricks、Remotion、Render、Superpowers** の9件で、全体90件超のうちの代表例として紹介されています。

### 1-2. 開発ライフサイクル全体（PR・ターミナル・SSH・ファイル）

- **GitHub の PR レビューコメント**への対応。  
- **複数ターミナルタブ**。  
- **SSH 経由のリモート devbox 接続**は **alpha**。  
- サイドバーで **PDF・表計算・スライド・文書**などのリッチプレビュー、**要約ペイン**でエージェントの計画・ソース・アーティファクトを追跡、と記載。

### 1-3. 時間をまたぐ作業とメモリ

- **自動化:** 既存の会話スレッドを**再利用**し文脈を引き継ぐ。**将来の時刻に実行をスケジュール**し、**長期タスク**のために自動で再開しうる、と説明（**日や週単位**の表現あり）。Slack・Gmail・Notion などの例が挙げられています。  
- **メモリ（preview）:** 過去のやり取りから有用な文脈（好み・訂正・時間のかかった情報）を保持し、以降のタスクを速く高品質に、と説明。  
- **文脈に応じた提案:** プロジェクト・接続プラグイン・メモリから、**作業の再開や1日の始め方**を提案しうる、と記載（Google Docs の未対応コメントなどの例）。

### 6-1. ロックインと「職場 OS」への寄り

公式は **デスクトップアプリ**と **ChatGPT サインイン**を前提にロールアウトを説明しています。技術的には自然な統合ですが、組織から見ると **認証・課金・データの出口**が **チャット製品と開発エージェント**に束ねられやすい、という**設計上の含意**は読み取れます（公式が「ロックイン」を謳ったわけではありません）。

**筆者の整理:** 「便利さ」と引き換えに、**退出コスト（カスタム指示・メモリ・プラグイン設定）**が積み上がるタイプのプロダクトです。評価軸は **「モデル品質」単体**より **ID・ログ・データ領域・サブプロセッサ**のセットで持つほうが、後からの差し替えで迷子になりにくいです。

さらに一歩踏み込むと、**職場 OS**という言い方は比喩に過ぎませんが、実務では「**どの画面にエージェントのカーソルが入っていいか**」「**どのリポジトリとどの SaaS が同一の会話文脈に載るか**」が、従来の **IDE＋Git** より広い範囲で組織ルールの対象になります。すでに **SSO や MDM で ChatGPT を配っている**企業ほど、**「チャットのポリシー」と「開発エージェントのポリシー」を分けたまま運用できるか**が、導入後の摩擦点になりやすい、というのは現場観察に近いメモです。

---

### 6-2. 攻撃面が増えるポイント（チェック観点だけ）

**画面操作・バックグラウンド・スケジュール実行**が揃うと、従来の「コード補完」より **誤操作・過剰権限・サプライチェーン（悪性プラグイン）**の影響半径が大きくなります。公式記事は機能説明が中心で、**権限モデルやログの粒度**は別ドキュメントでの確認が必要、と読むのが素直です。

| 観点 | 筆者メモ（問いの例） |
|------|----------------------|
| **画面・入力** | どの画面を見せてよいか、**業務データのマスキング**は誰が決めるか |
| **MCP・プラグイン** | **許可リスト**、バージョン固定、障害時の切り離し |
| **自動化の再開** | **無人実行**の範囲、失敗時の通知先、ロールバック |
| **メモリ・提案** | **誰の文脈が誰の端末に残るか**（共有マシン・職務代理・引き継ぎ時） |

**シナリオで頭を固める用のメモ（断定ではない）:**（1）**設定ミス**で広すぎる画面共有やトークンがエージェント側に載る、（2）**悪意あるプラグイン**が正規の MCP 経路を踏んで横移動する、（3）**スケジュール実行**が意図しない時刻に外部へ書き込む、の三つは、従来の「補完」より**再現実験の設計**が必要な類いです。ここは「禁止」より **再現手順とログ ID が取れるか**が、インシデント後の説明責任に直結します。

### 6-3. メモリと「覚えさせる」ことのコスト

公式は **メモリを preview** とし、**好み・訂正・時間のかかった情報**を将来タスクに活かす、と説明しています。利用者体験としては強力ですが、組織では **「一度覚えた誤り」**や **「古い前提」**が固定化しやすいという逆説もあります。個人の生産性と、**チームで共有する正の情報源（ドキュメント・チケット）**のどちらを優先するかが曖昧だと、あとから **「エージェントがそう言った」**が口実化しうる、というのは運用の話です。

**筆者の整理:** メモリまわりは、技術より先に **保持期間・削除権者・オフのデフォルト**を決めると会議が短くなることが多いです。特に **Enterprise／Edu／EU・UK が「soon」**とある帯域は、**提供開始と社内規程の更新**がズレやすいので、**「公式が出した日」と「社内で許可した日」を別記録**にしておくと、後追いが楽です。

### 6-4. Spud／法人シフトと Codex を同じ棚に並べない

**同じ週**に OpenAI 関連の話題が複数出ても、**本編が扱う Codex 更新**は **製品機能の公表（一次）**が主役です。**CFO インタビューや社内メモ**で語られる **法人比率や次世代モデル（Spud）**は、**情報の出どころと問いが違う**、と切るのが安全です（前日稿: [「ITニュース」｜OpenAI「Spud」と法人寄りシフト](../20260416/2026-04-16-openai-spud-enterprise-pivot｜OpenAI「Spud」と法人寄りシフト.md)）。

**編集・社内共有での切り分け話法（筆者案）:** スライド一枚目に **「今回の一次は何か」**だけを置く。Codex 本件なら **公式ブログの URL と日付**。その下に **「同じ週に出た別レイヤ」**を脚注扱いにし、**因果の矢印を引かない**。混ぜると起きやすい事故は、**「法人が増えるから computer use が安全」**のような **根拠のない接続**と、**「次世代モデルだから開発体験がこう変わる」**という **未公表スペックへの期待の輸入**です。どちらも読者には優しくないので、**見出しを分ける**のが一番安いです。

### 6-5. macOS 先行・法域と、端末標準のすれ違い

**computer use が macOS 先行**であることは、本件の一次から明確です。Windows 中心の企業や、**VDI 上の仮想デスクトップ**だけを開発に使う現場では、「記事の熱量」と **手元の端末**がすぐには一致しません。**rolling out** もあるため、**パイロット班だけ先に触る**段階で **運用ルールとセキュリティレビュー**が未整備のまま広がると、あとから **全面停止**になりがちです。

**EU・UK・Enterprise 向けの「soon」**は、機能ごとに速度が違いうる、と読んでおくほうが安全です。法務・情報システムでは **「公式の一文」と「社内の許可範囲」**を同義にしないことが、トラブル予防になります。

### 6-6. 画像・ブラウザ・スクショをまたぐ法務メモ（短文）

**gpt-image-1.5** や **スクリーンショット**を同一ワークフローに載せると、**第三者の UI や個人情報が画像に混ざる**ことがあります。著作権・商標・**顧客データの二次利用**の観点は、プロダクトの仕様書より **自社の契約テンプレと顧客 NDA** が先に効く場面が多いです。

**筆者の整理:** 「生成物の帰属」「学習に回るか」は **利用規約と管理者設定の追跡**が本体で、記事の感想より **法務チェックリストの一行**に落とすのが現実的です。ブラウザについて公式は **localhost 以外への完全操作は今後**と書いているので、**いま何ができて何ができないか**を **デモ動画ではなく一次の文面**で固定してから、社内説明に回すほうが誤爆が少ないです。

### 6-7. 比較表を乱立させないための三問

ベンダ横並びの表は気持ちよいですが、**定義が揃う前に作ると会議だけが増える**ことがあります。Codex 周りで先に自問自答しておくとよいのは、例えば次の三つです（**答えは組織依存**なので、ここでは列挙にとどめます）。

1. **「横断自動化」で削るのは、何の工数か**（レビュー待ち、環境構築、回帰テスト、ドキュメント同期、など）。削る対象が曖昧だと、導入成功の定義も曖昧になる。  
2. **失敗したときに、誰が止められるか**（個人の設定だけで止まるのか、管理者一括で止まるのか）。  
3. **「公式が言っている範囲」と「社内が許している範囲」**を、誰がいつ同期するか。

この三問に答えが出たあとで **Cursor／GitHub Copilot／他社エージェント**と並べると、比較表が **意思決定ツール**に近づきます。先に表を作ると、**スペックの穴埋めゲーム**になりやすい、というのが筆者の偏見です。

**本稿の主な用語（ごく短く）**

| 語 | この記事での意味 |
|----|------------------|
| **Responses API** | OpenAI の開発者向け API の一つ。エージェント的なやり取り（ツール呼び出しなど）を想定した系統。 |
| **WebSocket** | ブラウザやアプリとサーバの間を**つなぎっぱなし**にして、都度つなぎ直す HTTP より往復を減らす仕組み。 |
| **HTTP（都度接続）** | 本稿では、**リクエストごとに接続を開き直す従来型**の API 呼び出しを指す省略表現。 |
| **TTFT（Time To First Token）** | 最初のトークンが返るまでの時間。 |
| **TPS（Tokens Per Second）** | 1 秒あたりの生成トークン数。 |
| **接続スコープのキャッシュ** | 1 つの WebSocket 接続の間だけ**会話履歴・設定**を保持して、**毎リクエストの再処理**を避ける仕組み。 |

## 1. 何が公表されたか（公式の範囲で）

### 1-1. 背景 — 「推論が速くなった、次はどこが詰まるか」

同記事では、Responses API 発表後の経過を時系列で整理しています。

- **2025 年 3 月** — 同社は **Responses API** をローンチ（本記事内の記述。一次は本記事）。
- **2025 年 11 月前後** — Responses API 向けの**単一リクエストのクリティカルパス**の最適化を段階的に投入。**レンダー済みトークン・設定のキャッシュ**、**中間サービスの短絡**、**安全分類器の高速化**など。この結果、**TTFT** は「**約 45% 改善**」に近い水準に到達した、と記述。
- **推論の高速化**（例）— 従来の旗艦系は **約 65 TPS**、**GPT-5.3-Codex-Spark** は Cerebras 向け最適化で**目標 1,000 TPS 超**、バーストで **4,000 TPS まで**観測、等の数値が本文で例示されています。
- その結果、**推論（GPU）**が速くなるほど、**API 側 CPU の待ち**が相対的に目立つようになった、と説明されています。
- 構造的には、**各 Codex リクエストを独立として扱い**、フォローアップごとに**会話／再利用可能文脈の再処理**を重ねる設計の限界、が指摘されています。

### 1-2. WebSocket モードの設計方針

- **WebSocket** による**永続接続**と、**接続スコープでのキャッシュ**などで**往復を減らす**方針。
- **開発者体験の互換性** — 記事内では、**`response.create` ＋ `previous_response_id`** の形状を崩さない、といった**既存の入出力を大きく変えない**方針が示されています。HTTP からの移行コストを抑える設計思想、と読めます。
- **開発の経緯** — **プロトタイプから約 2 か月**の集中開発。**主要コーディングエージェント系スタートアップ向けアルファ**を経て、本番展開。**Codex** は **Responses API のトラフィックの多く**を短期間で WebSocket に移行した、と記述。

### 1-3. 効果の数値 — 本文と X 引用を分けて読む

| 出典の強さ | 内容 |
|---|---|
| **OpenAI 本文（一次）** | **エージェント的ロールアウト**が**エンドツーエンドでおおよそ 40% 速く**なった、等と説明。**TTFT 約 45% 改善**は WebSocket 前の取り組みに紐づく記述。 |
| **X（旧 Twitter）経由の第三者例** | **Vercel（AI SDK）**で「最大**約 40%** 減」／**Cline**で**約 39% 速い**／**Cursor** 上の OpenAI モデルは最大**約 30% 速い**、等が記事内 X リンク先で紹介。 |

## 4. 短期・中期・長期の含意（観察レーン）

**時間軸:** 短期＝0〜3か月、中期＝3か月〜1年、長期＝1年以上（本稿内の整理用の目安）。この節は**予想／不確実性／効きにくい条件**までの**観察**にとどめ、**どれが現実的か／どの順に起きやすいか**は筆者メモへ。

| 時間軸 | 予想されうること | 不確実性・効きにくい条件 |
|--------|------------------|---------------------------|
| **短期** | 既に **Codex の大部分トラフィック**で WebSocket 利用（公式）。**SDK・サンプルの追従**、エージェント系ツール利用者の**待ち時間**短縮（第三者例としては Vercel／Cline／Cursor 等） | 第三者数値の**ベンチワークロード・モデル・地域**は本文単体では未特定。**自環境では要計測** |
| **中期** | 「Responses API 上でエージェント的運用」する実装の**移行先パターン**が揃い、**HTTP のみ**実装のメンテ負債化の可能性 | 他社が**同様の常時接続＋セッション状態**を**別プロトコル**で標準化するか、**エッジ**や**オンプレ推論**でボトルネックが再配置されるかは不確定 |
| **長期** | エージェント的ワークフローを「**一連の独立 HTTP 会話**」で実装する設計の**相対的コスト上昇**（業界トレンドとの接続は留保） | モデル推論の**更なる高速化**、**新しい API 抽象**（例: 別のセッション API）への再統合で、**WebSocket 固有**の価値が相対的に下がる可能性 |

**効果が出ない条件（短期全般に効く一般則）:**

- **大量の小さな会話**で接続の恩恵が小さい
- **ツール実行やローカル I/O** が主体で、API 再処理の割合が相対的に小さい
- **接続未対応**のクライアントのまま
- 組織の**セキュリティ・コンプラ**で**長寿命 WebSocket** が採択しにくい
- **接続数スパイク**の運用課題でロールアウトが遅い

## 5. 混同しやすい点

| 論点 | 注意 |
|------|------|
| **「推論を速くすれば全部速くなる」** | 半分正しい。**残り半分は API の往復と毎回の会話再処理**、というのが本記事の問題提起。**ボトルネックの分解**から入るのが筋。 |
| **「40% 高速化は全員で出る」** | 公式本文の**エージェント的ロールアウト**への記述と、**X 引用の第三者例**は**強度が違う**。**出典と条件**を分けて読む。自社 SLO に直結させる前に**計測が必要**。 |
| **「HTTP は捨てるべき」** | **大量の小さな会話**や**ツール実行主体**のワークロードでは、WebSocket の恩恵は相対的に小さい。**急いで切り替えない**判断がありえる。 |
| **「1,000 TPS／4,000 TPS バースト」** | これは **GPT-5.3-Codex-Spark + Cerebras** 最適化の例。**汎用モデル**の一般的な数値ではない。 |
| **「TTFT 約 45% 改善は WebSocket の効果」** | **WebSocket 以前**の単一リクエスト最適化による数値、という位置づけ。**WebSocket の効果**と**以前の最適化**は別のブロックとして語られている。 |
| **「AI SDK で統合済みなら自動で速くなる」** | SDK のバージョン・設定・アプリ側の呼び出し方次第。**サンプル差し替えだけで済む前提**は持たない。 |

### 7-2. 比較の評価軸（筆者が選ぶ軸）

**軸の取り方自体が主張**です。WebSocket 対応の可否を**「対応した／していない」の二値**で比較しても意思決定の役に立ちにくい、というのが筆者の偏見。次の三軸が優先です。

1. **ボトルネック分解**（推論／API 層 CPU／ネットワーク往復／ツール実行／ローカル I/O）  
   公式本文のキーメッセージはまさにこれです。**自社ワークロードで**どこが遅いかを**分解計測**しないまま WebSocket 化しても、**恩恵のない変更**になりうる。**「何 % 速くなるか」より「何が遅かったか」を先に見える化**するほうが、意思決定が進みやすい。
2. **互換性と移行コスト**（SDK 対応・既存コードの書き換え量・ロールバック容易性）  
   公式が **`response.create` ＋ `previous_response_id` 形状を崩さない**と明記したのは、**互換性が最大の採用摩擦**だと理解しているからだと筆者は読みます。他社 API 比較では**「既存コードから小さな差分で乗れるか」**を筆頭に。
3. **長寿命接続の運用コスト**（ロードバランサ・タイムアウト・再接続・接続数上限・セキュリティ方針）  
   WebSocket は**輸送層の勝ちパターン**ですが、**運用では新しい論点を増やします**。**L7 ロードバランサ**の設定、**アイドルタイムアウト**、**再接続時の状態復元**、**企業のセキュリティ方針**（長寿命接続の認可・監査）の整備コストを**評価軸に組み込む**必要がある。

**避けたい軸の扱い:** **「TPS」「% 速い」の数値**は、**ベンチ条件が揃っていない段階では**主軸にしない（§1-3・§5 の注意と整合）。**数値は「自社ベンチ」に流し込んでから表に載せる**のが、会議の再現性を保つ近道。

### 7-3. 反論・限界（セルフ反証）

- **「HTTP のメンテ負債化」は一般論として強すぎる可能性。** **小さな会話中心**・**単発のバッチ推論**ではむしろ**HTTP のほうが運用が軽い**ケースが多い。**「全員が乗り換えるべき」という読み方は本文が支持していない**。
- **Codex の先行移行は「内製だから早い」面がある。** 自社製品の都合で**接続型に寄せやすい**環境と、**外部企業の本番運用**は条件が違う。**OpenAI 自身の移行速度**を**一般企業の目標値**にしない。
- **第三者数値（Vercel・Cline・Cursor）は X 経由。** 同一条件での再現は難しく、**本文以上の強度を持たせない**のが安全。**自社 SLO の根拠**としては**自社ベンチの数値だけ**を使う。
- **「輸送を速くすれば体感も速くなる」は半分。** **ツール実行（外部 API、DB、ブラウザ操作）**が主な遅延源なら、**WebSocket 化の効果は限定的**。**§7-2 の軸 1**（ボトルネック分解）が欠けると、**投資対効果の見積もりが過大**になる。
- **長寿命接続のセキュリティ負荷**が見落とされがち。 **監査ログの ID 単位**、**切断検知**、**認可トークンの更新**、**DDoS 面での接続数制御**など、**HTTP では相対的に軽かった論点**が上がってくる。
- **見立てを更新する材料の明示:** **GA（リサーチプレビュー相当表記の終了）**、**AI SDK の正式対応バージョン**、**公式ベンチ**が**同一ワークロード・同一地域**で再現可能な形で出るか、が揃うと、**§7-1 の注視順**の点検が必要になります。

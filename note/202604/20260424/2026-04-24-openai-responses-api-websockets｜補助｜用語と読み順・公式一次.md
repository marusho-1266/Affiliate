## 1. 用語ミニ辞典

| 語 | ひとこと |
|----|-----------|
| **Responses API** | OpenAI の開発者向け API の一つ。エージェント的なやり取り（ツール呼び出しなど）を想定した系統。**2025年3月**に同社がローンチ（本記事内記述）。 |
| **WebSocket** | ブラウザやアプリとサーバのあいだを**つなぎっぱなし**にする通信方式。**双方向・低オーバーヘッド**が特徴で、**都度つなぎ直す HTTP より往復を減らせる**。 |
| **HTTP（都度接続）** | 本稿では、**リクエストごとに接続を開き直す従来型**の API 呼び出しを指す省略表現。 |
| **TTFT（Time To First Token）** | 最初のトークンが返るまでの時間。WebSocket 記事中では、**WebSocket 以前**の単一リクエスト最適化で**約 45% 改善**に触れられている。 |
| **TPS（Tokens Per Second）** | 1秒あたりの生成トークン数。本文では従来旗艦系が**約 65 TPS**、**GPT-5.3-Codex-Spark** は Cerebras 向け最適化で**目標 1,000 TPS 超**、バーストで **4,000 TPS まで**観測、等の数値が登場。 |
| **接続スコープのキャッシュ** | 1 つの WebSocket 接続の間だけ**会話履歴・設定**を保持して、**毎リクエストの再処理**を避ける仕組み。 |
| **previous_response_id** | Responses API で**前の応答を引き継ぐ**ための ID。公式は WebSocket モードでも **`response.create` ＋ `previous_response_id` の形状**を崩さない方針、と記載。 |
| **response.create** | Responses API の応答生成エンドポイント呼び出し。**HTTP でも WebSocket でも同じ形**で書けるのがポイント。 |
| **Codex** | OpenAI のコーディング／エージェント製品群。**Responses API のトラフィックの多くを短期間で WebSocket に移行**した自社事例として本記事に登場。 |
| **Codex-Spark（GPT-5.3-Codex-Spark）** | Codex 系モデルの一つ。**Cerebras 向け最適化**での高速推論が本記事で例示。 |
| **エージェント的ロールアウト／agentic rollout** | ツール呼び出しや会話を**多段に重ねる**タイプのワークフロー。公式本文では**エンドツーエンド約 40% 高速化**の文脈で登場。 |
| **永続接続（persistent connection）** | 一度つないだあと**しばらく維持される接続**。WebSocket の特徴で、**再接続・再処理の回避**と引き換えに**接続数管理**などの運用論点が増える。 |
| **AI SDK（Vercel）** | Vercel が提供する AI 開発向け SDK。本記事が**X 経由で**引用する第三者数値の一つの出どころ。 |
| **Cline・Cursor** | 開発者向けツール。本記事が **X 経由で**改善率の数値を引用している例として登場。 |

| **二次報道・解説記事** | 各メディアの要約・ベンチ記事 | **追加の数値・競合比較**は帰属を確認。本件の核は公式本文で足りる前提。 |

**よくある誤読:**

- **「推論を速くすれば全部速くなる」** → 半分。**残り半分は API の往復と毎回の会話再処理**、というのが本記事の問題提起。
- **「40% 高速化は全ワークロードで出る」** → 公式本文の**エージェント的ロールアウト**への記述と、**X 引用の第三者例**は強度が違う。**自社 SLO に直結させる前に計測**が必要。
- **「TTFT 約 45% 改善は WebSocket の効果」** → **WebSocket 以前**の単一リクエスト最適化による数値。WebSocket の効果とは別ブロック。
- **「HTTP は捨てるべき」** → **大量の小さな会話・ツール実行主体**では WebSocket の恩恵は小さい。**ワークロード別の判断**が必要。
- **「1,000 TPS／4,000 TPS」** → **GPT-5.3-Codex-Spark ＋ Cerebras** 最適化の例。**汎用モデルの一般値ではない**。

## 4. 本編に入る前のチェックリスト（任意）

**前提・計測**

- [ ] 自社のエージェント実装が**ほんとうに「HTTP 往復＋会話再処理」で詰まっているか**を**自社ワークロードで計測**できているか（推論／API 層／ネットワーク／ツール実行／I/O の分解）
- [ ] **基準線の数値**（現行 HTTP での TTFT・往復回数・待ち時間）を取ってあるか
- [ ] **ワークロードの性格**（会話の長さ・ツール実行比率・並列度）を一枚で描けているか

**設計・互換**

- [ ] 既存コードが **`response.create` ＋ `previous_response_id`** を前提にしているか（互換移行の前提）
- [ ] 使用 SDK の**WebSocket 対応バージョン**とその**リリース方針**を確認したか
- [ ] **HTTP と WebSocket の並走**（ワークロード別の切り替え）ができる構造か
- [ ] **ロールバック**（WebSocket → HTTP）の手順がある か

**運用**

- [ ] **ロードバランサ**のアイドルタイムアウト・切断挙動を WebSocket 向けに確認したか
- [ ] **再接続時の状態復元**（`previous_response_id` ベース）を設計したか
- [ ] **接続数の監視**（しきい値・アラート・ダッシュボード）を準備したか
- [ ] **デプロイ時**の接続切断・順次切替戦略を決めたか

**セキュリティ・契約**

- [ ] **長寿命接続**を許容する**認可モデル**と**監査ログの粒度**を紙で書いたか
- [ ] **接続スコープのキャッシュに載るデータ**を契約書と突き合わせたか
- [ ] **地域・規制**（EU 等）での長寿命接続の扱いを整理したか
- [ ] **プロンプトインジェクション**・**意図しないツール実行**のシナリオを長寿命接続でも試したか

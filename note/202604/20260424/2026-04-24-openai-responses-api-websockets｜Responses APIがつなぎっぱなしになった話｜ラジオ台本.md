【本編パート1｜公表内容と設計の骨格（約4分）】

〔転換・読み上げ用〕

では本題に入ります。
まず、公式ブログ「Speeding up agentic workflows with WebSockets in the Responses API」で、何が語られているかを順に押さえます。

同記事では、ResponsesAPI発表後の経過を時系列で整理しています。

2025年3月、同社はResponsesAPIをローンチした、と記事内に書かれています。
2025年11月前後には、単一リクエストのクリティカルパス最適化が段階的に入り、レンダー済みトークンや設定のキャッシュ、中間サービスの短絡、安全分類器の高速化などが挙げられています。
その結果、TTFT、Time To First Token、最初のトークンが返るまでの時間は、おおよそ45パーセント改善に近い水準に到達した、と記述されています。

推論の高速化の例として、従来の旗艦系はおおよそ65TPS、1秒あたりの生成トークン数です。
GPT5.3CodexSparkはCerebras向け最適化で目標1000TPS超、バーストで4000TPSまで観測した、等の数値が本文で例示されています。

その結果、推論、GPUが速くなるほど、API側CPUの待ちが相対的に目立つようになった、と説明されています。
構造的には、各Codexリクエストを独立として扱い、フォローアップごとに会話や再利用可能文脈の再処理を重ねる設計の限界が指摘されています。

ウェブソケットモードの設計方針は次のとおりです。
ウェブソケットによる永続接続と、接続スコープでのキャッシュなどで往復を減らす。
開発者体験の互換性として、response.createとprevious_response_idの形状を崩さない、といった既存の入出力を大きく変えない方針が示されています。
HTTPからの移行コストを抑える設計思想、と私は読み取っています。

開発の経緯として、プロトタイプから約2か月の集中開発、主要コーディングエージェント系スタートアップ向けアルファを経て本番展開、と書かれています。
Codexは、ResponsesAPIのトラフィックの多くを短期間でウェブソケットに移行した、とも記述されています。

【本編パート4｜短期・中期・長期の観察（約3分）】

〔転換・読み上げ用〕

時間軸の置き方です。
この話では、短期を0から3か月、中期を3か月から1年、長期を1年以上の目安として整理します。
予想や不確実性、効きにくい条件まで含む観察にとどめ、どれが現実的かの優先順位づけまでは踏み込みません。

短期に想定されうるのは、すでにCodexの大部分トラフィックでウェブソケット利用、と公式に書かれている点、SDKやサンプルの追従、エージェント系ツール利用者の待ち時間短縮、第三者例としてVercelやCline、Cursor等が挙がることです。
不確実性として、第三者数値のベンチ条件は本文単体では未特定で、自環境では要計測です。

中期は、ResponsesAPI上でエージェント的運用する実装の移行先パターンが揃い、HTTPのみ実装のメンテ負債化の可能性が論点になりうる、という整理です。
他社が同様の常時接続とセッション状態を別プロトコルで標準化するか、エッジやオンプレ推論でボトルネックが再配置されるかは不確定です。

長期は、エージェント的ワークフローを一連の独立HTTP会話で実装する設計の、相対的コスト上昇が語られうる、と留めます。
業界トレンドとの接続は断定しません。
モデル推論の更なる高速化や、新しいAPI抽象への再統合で、ウェブソケット固有の価値が相対的に下がる可能性も、不確実性として並べておきます。

効果が出ない条件として、大量の小さな会話で接続の恩恵が小さい、ツール実行やローカルI/Oが主体でAPI再処理の割合が小さい、接続未対応のクライアントのまま、組織のセキュリティやコンプライアンスで長寿命ウェブソケットが採択しにくい、接続数スパイクの運用課題でロールアウトが遅い、といった一般則も挙げておきます。

【本編パート5｜混同点とまとめ（約4分）】

〔転換・読み上げ用〕

混同しやすい点を六つ、短く区切ります。

第一に、推論を速くすれば全部速くなる、と決めつけないこと。
半分正しく、残り半分はAPIの往復と毎回の会話再処理、というのが本記事の問題提起です。
ボトルネックの分解から入るのが筋です。

第二に、40パーセント高速化は全員で出る、と短絡しないこと。
OpenAI本文のエージェント的ロールアウトへの記述と、X引用の第三者例は強度が違います。
出典と条件を分けて読み、自社SLOに直結させる前に計測が必要です。

第三に、HTTPは捨てるべき、と急がないこと。
大量の小さな会話やツール実行主体のワークロードでは、ウェブソケットの恩恵は相対的に小さいです。
急いで切り替えない判断がありえます。

第四に、1000TPSや4000TPSバーストは、GPT5.3CodexSparkとCerebras最適化の例であり、汎用モデルの一般的な数値ではないこと。

第五に、TTFTおおよそ45パーセント改善はウェブソケットの効果と混同しないこと。
ウェブソケット以前の単一リクエスト最適化による数値、という位置づけです。
ウェブソケットの効果と以前の最適化は、別ブロックとして語られています。

第六に、AISDKで統合済みなら自動で速くなる、と仮定しないこと。
SDKのバージョン、設定、アプリ側の呼び出し方次第です。
サンプル差し替えだけで済む前提は持たないほうがよいです。

まとめです。
2026年4月22日、OpenAIはResponsesAPIにウェブソケットモードを追加したと公表しました。
背景は、推論が速くなった結果、HTTPの都度接続と会話再処理が相対的に重くなった、という問題提起で、永続接続と接続スコープのキャッシュで往復を削る設計です。

数値は、OpenAI本文の説明と、X引用の第三者例を、情報の強さで分けて扱うのが安全です。
自環境では要計測です。

HTTPを捨てる判断は急がないのが素直です。
大量の小さな会話やツール実行主体のワークロードでは恩恵が小さく、ウェブソケット化の運用コスト、ロードバランサ、タイムアウト、再接続、接続数監視も新たに積み上がります。

同日の法人向け発表は別レイヤとして切り離して読むほうが安全で、必要なら別途追ってください。

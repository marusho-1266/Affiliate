## はじめに

**2026年4月22日**、OpenAI は公式ブログ **[Introducing workspace agents in ChatGPT](https://openai.com/index/introducing-workspace-agents-in-chatgpt/)** で、法人向け ChatGPT の新機能 **workspace agents（ワークスペース・エージェント）** のリサーチプレビュー開始を公表しました。対象プランは **ChatGPT Business／Enterprise／Edu／Teachers**、背後のエンジンは **Codex**、位置づけは **カスタム GPT の進化形**、と公式本文に明記されています。**2026年5月6日までは無料**で、**同日からクレジット制の課金**に移行する、とも書かれています。

本稿の読みどころは次の2点です。

1. **何が「オン／オフ／未対応／Soon」に分かれているか** — 公式とヘルプセンターで**明確に条件が書かれている部分**を先に固定し、過大評価を避ける。
2. **統制**（Compliance API、既定オフ、EKM 未対応など）を**モデル性能の話と同列に**置く理由 — 法人導入で効く条件は、機能よりも運用側に寄っていることが多い。

**本稿の主な用語（ごく短く）**

## 1. 何が公表されたか（公式の範囲で）

### 1-1. 製品の位置づけと提供範囲

- **対象プラン:** ChatGPT **Business、Enterprise、Edu、Teachers**（公式本文）。
- **段階:** **リサーチプレビュー**。仕様や提供条件は変わりうる前提。
- **位置づけ:** **カスタム GPT（GPTs）の進化形**と明記。違いは**「個人のチャットに閉じた文脈」**から**「チームで共有する文脈・引き継ぎ・複数ツール横断」**へ広がる点、と説明されています（原文: *workflows that depend on shared context, handoffs, and decisions across teams*）。
- **基盤:** **Codex で駆動**（公式本文に明記）。直前の [Codex for (almost) everything](https://openai.com/index/codex-for-almost-everything/)（April 16, 2026）の延長線上に、**法人 UI へ同じ Codex ストーリー**を伸ばした格好です。
- **顧客事例（OpenAI が紹介）:** **Rippling** の顧客コメントとして、**週 5〜6 時間**かかっていた作業がバックグラウンドで回る例が引用されています。**事例紹介**であり、一般化された効果測定ではない点に注意。

### 1-2. 価格

- **2026年5月6日までは無料**、**同日からクレジット制の課金**に移行、と公式本文に記載。
- **クレジット単価の細目**は執筆時点の同記事では **「credit-based」** 相当の記述にとどまります。**単価・枠・超過ルール**は別アナウンス・契約書の追跡対象です。

### 1-3. 統制（ヘルプセンター）

[**ChatGPT Workspace Agents for Enterprise and Business**](https://help.openai.com/en/articles/20001143) に、運用の実装事実がまとまっています。

- **Enterprise はローンチ時に既定でオフ**。管理者が有効化する前提。
- **EKM（Enterprise 顧客管理鍵）利用ワークスペースには未提供**。該当組織は**現時点の対象外**と読むのが素直です。
- **管理者によるツール／アクション権限**、**Compliance API**、**エージェント停止**など、**エンタープライズ向けの統制**が同時に打ち出されています。
- 「共有接続＋Slack」のような構成では、**権限範囲**と**他者がトリガーした際の影響**をレビューするよう警告する旨もヘルプ側に記述。

**ページの「Updated」表記は相対的**な場合があるため、利用前に**最新版**を開き直すのが安全です。

### 1-4. GPTs との関係

- 公式は GPTs は**当面維持**、**近く workspace agents への変換を容易にする**と述べる段階です。
- **互換の細目・変換 UI・移行ツール**は **「Soon」** に委ねられる記述がありうるため、**既存の GPTs 資産は急いで移行せず**、**変換導線の登場を待つ**前提で読むのが安全です。

## 4. 短期・中期・長期の含意（観察レーン）

**時間軸:** 短期＝0〜3か月、中期＝3か月〜1年、長期＝1年以上（本稿内の整理用の目安）。この節は**予想／不確実性／効きにくい条件**までの**観察**にとどめ、**どれが現実的か／どの順に起きやすいか**は筆者メモへ。

| 時間軸 | 予想されうること | 不確実性・効きにくい条件 |
|--------|------------------|---------------------------|
| **短期** | 対象プラン組織での**パイロット**、営業・CS・運用チームの定型レポート／ルーティング業務の一部自動化、Slack 上での問い合わせ一次対応の試行 | **リサーチプレビュー**のため仕様変更・制限追加がありうる。**5/6 以降の課金**が利用パターンを変える。**EKM 非対応**の自社では展開が限られる |
| **中期** | 部門横断の**再利用可能なエージェント**の蓄積、GPTs からの移行・テンプレ標準化、Compliance API を前提にした**監査対応**の型 | 競合製品との機能・価格比較で**乗り換え・併用**が進む可能性。クレジット課金後の**TCO**がボトルネックになるかは公式情報が限定的 |

**効果が出ない条件（短期全般に効く一般則）:** 管理者が有効化しない／コネクタ認証が整備されない／**プロンプトインジェクション**や誤った書き込み承認設定で運用が止まる／ガバナンスが追いつかず乱立、または統制過多で現場が使わない。

### 7-1. シナリオ分岐（筆者の読み・**注視順**）

**注:** 以下は「起こる確率」ではなく「**筆者が現時点で注視したい順**」です。確度ではなく、**次の数週間で材料が揃ったとき、見立てが崩れるかどうかを点検する優先順位**として読んでください。

1. **分岐 B（法人で緩やかに定着、GPTs は並存したまま）** — 最も地合いが強いと筆者が見るのは**「5/6 の課金切替までに試したい組織が触り、触ってから本格導入を判断する」**というパターン。**Enterprise は既定オフ**なので、広がり方は**管理者の決裁速度**に規定されます。**GPTs を一気に捨てる判断**は、変換 UI（Soon）が出るまで**合理的ではない**と筆者は読みます。
2. **分岐 A（業務部門の突き上げで急速にパイロット化）** — Slack・Gmail・Notion 等の**接続が揃っている業務部門**が、情シスに先行してテンプレ運用を始めるパターン。**便利さが先行しガバナンスが追いかける**、という順序になりやすい。この場合、**5/6 以降の課金切替**で一度ブレーキがかかる可能性。
3. **分岐 C（EKM 等の制約で対象外のまま据え置き）** — 金融・医療・規制業種など **EKM 利用ワークスペース**が中心の組織は、**ローンチ時点で対象外**のため、**しばらく評価だけで終わる**シナリオ。将来 EKM 対応が出るまで、**内製または別ベンダ**で代替する判断が合理的になる場面もある。
4. **分岐 D（他社の法人エージェントに乗り換え）** — **Microsoft Copilot Agents**、**Google Workspace のエージェント**、**Anthropic Claude 系**が**同じ軸**（共有コンテキスト＋統制）で厚い発表を重ねると、**「どれか一つを社内標準」に決めたい組織**にとって本件は待ち筋の一つに降格しうる。**§7-2 の評価軸**で比較するときの分岐点。

**筆者が現時点で最も現実的と見る起点は B。** A／C は**業種・組織規模による条件依存**、D は**他社の打ち手次第**で本稿の主題から外に出る分岐です。

### 7-2. 比較の評価軸（筆者が選ぶ軸）

**軸の取り方自体が主張**です。他社比較で**モデル品質（スコア）**を先頭に置く表を並べても、会議が増えるだけで決まりにくい、というのが筆者の偏見。workspace agents を他社に対置するなら、次の三軸が優先です。

1. **状態の共有単位**（個人／セッション／チーム／組織）  
   workspace agents は**チーム**を持続単位に宣言した点が固有です。**誰の文脈が誰の端末に残るか**（共有マシン・職務代理・引き継ぎ時）が、従来の個人 ChatGPT より広く組織ルールの対象になる。他社のエージェントを見るときも**「状態の共有単位」**を最初に聞くと、製品の輪郭がつかみやすい。
2. **統制の粒度**（既定オン／オフ、EKM 対応、Compliance API 相当の有無、ログの ID 単位）  
   **既定オフ／EKM 未対応**は workspace agents の**現時点の事実**で、他社比較の際は**同格の項目**を並べる（他社は「既定オン」で始めるケースもある）。**勝ち筋は「制御がすぐ効く」側にある**、と筆者は読みますが、ここは組織のリスク選好で逆もあり得ます。
3. **移行コスト**（既存の GPTs／他社カスタム資産からの引き継ぎコスト）  
   公式が「Soon」とした **GPTs 変換**の具体仕様が出るまでは、**「移行コストを見える化できない」**のが正直な現状。**「今あるものを持ち越せるか」**の一点を比較表の筆頭に置くと、**スペック競争ではなく資産運用の視点**で議論できます。

**避けたい軸の扱い:** **「何ができる」の機能チェックリスト**だけの比較は、**定義が揃わない段階では意思決定の役に立ちにくい**です。**「自社の業務で何が削れるか」**を**1件あたりの往復回数・待ち時間・承認数**などの**尺度**に落としてから比較表を組むと、表の寿命が伸びます。

### 7-3. 反論・限界（セルフ反証）

- **「チームで育てるエージェント」は本当に新しいか。** 公式のフレーズとしては新しいが、**Slack bot／社内自動化基盤／他社のエージェントビルダー**は以前から同じ問題意識を持っていた。**「カスタム GPT の進化形」**という位置づけは、**OpenAI の製品線に閉じた表現**で、**業界全体を見たときの新規性**は別評価が要る。
- **Rippling の「週 5〜6 時間」に引きずられない。** 事例紹介であり、**業務・規模・既存自動化の整備度**が違えば効果の幅は大きく変わる。**自社 ROI の根拠**としては弱い材料なので、**自分の業務の尺度**に置き直す作業が必要。
- **「既定オフ」は安全装置だが、運用の難しさでもある。** オフから始める設計は**安全側**ですが、**「誰が有効化判断をするか」**が決まっていない組織では、**触ってみたい現場と、判断を保留する情シス**のあいだで**時間だけが過ぎる**こともある。
- **「Codex 駆動」の意味を過大に読まない。** 公式は「Codex で駆動」と書いているが、**workspace agents が Codex 本体の更新と同じタイミング・同じ機能を持つ**とは書いていない。**Codex アプリ側の対応**は「What's next」段階で、**今日時点の機能差**を前提に評価する。
- **見立てを更新する材料の明示:** **クレジット単価の具体提示で分岐 A/C の傾きが見える**、**他社の同軸の発表**が出たら分岐 D を引き寄せる。**GPTs 変換 UI**の登場は、**移行コスト議論の質を大きく変える**ので、§7-2 の三軸も書き直しが必要になります。

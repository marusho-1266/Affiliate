# 報告資料: OpenAI、Responses API に WebSocket モード — エージェント的ワークフロー高速化

- **調査日**: 2026-04-23
- **主出典**: [OpenAI — Speeding up agentic workflows with WebSockets in the Responses API](https://openai.com/index/speeding-up-agentic-workflows-with-websockets/)（**April 22, 2026**。本文・著者: Brian Yu, Ashwin Nathan）
- **要約（1文）**: 推論が高速化する一方で、HTTP の都度接続＋会話再処理がボトルネックになっていたエージェント用 Responses API について、**永続接続（WebSocket）**と接続スコープのキャッシュで往復を抑え、同社の説明では**エージェント的ロールアウトをエンドツーエンドで約 40% 速く**し、**Codex の本番トラフィックの大半**が WebSocket へ乗るまでになった、というエンジニアリング解説の公式記事。

---

## 1. 背景 — なぜ今この情報が出ているか

- **事実（時系列）**:
  - **2025 年 3 月** — 同社は Responses API 自体をローンチ（記事内記述。一次は本記事）。
  - **2025 年 11 月前後** — Responses API 向け**単一リクエストのクリティカルパス**の最適化（レンダー済みトークン・設定のキャッシュ、中間サービス短絡、安全分類器の速さ改善など）を段階的に投入。
  - その結果、**初トークンまでの時間（TTFT）**は「約 45% 改善」に近い水準（出典: 上記同記事）だが、**推論（GPU）**が速くなる新モデル前提では、**API 側 CPU の待ち**が相対的に目立つようになった、と説明されている。
  - **推論の例示**（同記事）— 従来の旗艦系は **約 65 TPS**、**GPT‑5.3‑Codex‑Spark** は専用 Cerebras 向け最適化で**目標 1,000 TPS 超**、バーストで **4,000 TPS まで**観測、など。
  - 構造的には、**各 Codex リクエストを独立**として扱い、フォローアップ毎に**会話／再利用可能文脈の再処理**を重ねる設計の限界を指摘。
  - **WebSocket プロトタイプ**以降、**約 2 か月**の集中開発、**主要コーディングエージェント系スタートアップ向けアルファ**、本番展開。Codex は **Responses API トラフィックの多く**を短時間で WebSocket に移行、と記述。
  - **2026 年 4 月 22 日** — 上記技術的経緯と、WebSocket 安定版（使い馴れた `response.create` ＋ `previous_response_id` 形状）の設計、効果測定の例を**公式ブログで公開**。

- **公開情報から読み取れる動機**:
  - **技術的成熟・製品ライン** — 「推論が速いほど、周辺（API サービス、クライアント、ツール）がその速度差を**ボトルネック化**しやすい」という課題の表明と、**輸送プロトコル（永続接続＋接続上キャッシュ）**での解消。
  - **利用実績の示唆** — 自社（Codex）とパートナー（Vercel / AI SDK、Cline、Cursor 等）名での**遅延改善の幅**（下記 3. の数値）を併記し、開発者向け導入の便益を主張。

- **補足・推測（根拠付き）**:
  - 競合他社（別ベンダ）との**価格・機能の直接比較**は、本主出典では主眼ではない（推測: 自社 API／Codex 体験の**体感速度**と、高速推論投資の**ユーザーへの到達**が説明上の主目的と読める）。
  - 記事は **X（旧 Twitter）**を引用して**第三者（Vercel、Cline、Cursor）**の数字を述べるため、**再現性・測定条件**は本編より一次が薄い（検証点に回す）。

## 3. 効果 — 短期・中期・長期の予想

**時間軸の定義（本ブリーフ内）**  
- **短期**: **0〜3 か月** — SDK・サンプル追従、既存利用者の段階的切替、計測レポートの出そろい。  
- **中期**: **3 か月〜1 年** — 本番採用の広がり、他クラウド／リバプロとの運用知見の一般化。  
- **長期**: **1 年以上** — 「エージェント＝長寿命接続＋状態の局所保持」が**事実上の定石**化するかどうか。

- **短期（0〜3 か月）**:
  - **予想される効果**: 既に **Codex の大部分トラフィック**で WebSocket 利用（**主出典**）。エージェント系ツール利用者の**待ち時間**が短くなる、と説明。第三者名として **Vercel（AI SDK）**で「最大**約 40%** 減」、**Cline**で**約 39%** 速い、**Cursor** 上の OpenAI モデルは最大**約 30%** 速い、など X 引用の例（**出典: 本記事内の X リンク先。一次は主に OpenAI 本文**）。
  - **不確実性**: 第三者数値の**ベンチワークロード、モデル、地域**は本文単体では未特定。自環境では**要計測**。
  - **効果が出ない条件**: **大量の小さな会話**で接続の恩恵が小さい、**ツール実行やローカル I/O**が主体で API 再処理が割合的に小さい、**接続未対応**のクライアントのまま、等。

- **中期（3 か月〜1 年）**:
  - **予想される効果**: 「Responses API 上でエージェント的運用」する実装の**移行先パターン**が揃い、**HTTP のみ**実装のメンテ負債化の可能性。高速推論ロードマップ（ハード含む）と**API 層**の**セット最適**が、ベンダ文脈で語られやすくなる（推測。業界一般トレンドとの接続）。
  - **不確実性**: 他社が**同様の常時接続＋セッション状態**を**別プロトコル**で標準化するか、**エッジ**や**オンプレ推論**でボトルネックが再配置されるかは不確定。
  - **効果が出ない条件**: 組織の**セキュリティ・コンプラ**で長寿命 WebSocket が採択しにくい、**接続数スパイク**の運用課題でロールアウトが遅い、等。

- **長期（1 年以上）**:
  - **予想される効果**: エージェント的ワークフローを「**一連の独立 HTTP 会話**」で実装する設計の相対的コスト上昇、と読める（推測。標準化が進むと仮定）。
  - **不確実性**: モデル推論の**更なる高速化**、**新しい API 抽象**（例: 別のセッション API）への再統合で、**WebSocket 固有**の価値が相対的に下がる可能性。
  - **効果が出ない条件**: **ステートフル**の長期管理が**クライアント側**に移り、サーバ側永続化の要請が下がる設計に舵を切る場合。

# ターゲットペルソナ（固定）：プロダクト／事業企画・AI活用PM

海外・日本 ITニュースの **note 記事**および **`/overseas-it-news` 系の整理**で、**主たる読者像をブレさせない**ための参照資料。  
業種・会社固有の数値は書かず、**役割と情報ニーズ**で固定する。

---

## 1. 一言定義

**事業会社で、生成AI・デジタル施策を「機能」だけでなく契約・稟議・情シス／法務連携まで含めて前に進める、事業側の企画／プロダクトPM。**  
技術専門家ではないが、**公式一次情報で「経営・上司に説明できる粒度」**まで落とす責任がある。

---

## 2. 役割・組織での位置

| 項目 | 内容 |
|------|------|
| よくある所属 | 事業開発、DX推進、事業部内デジタル／AI推進、新規事業 |
| 仕事の中心 | ロードマップ案、PoC設計、ベンダ／社内開発との要件調整、稟議資料、週次レビュー |
| 上司 | 事業部長／本部長（売上・コスト・リスク） |
| 横串 | 情シス（セキュリティ・契約）、法務、調達、広報、CS など |
| 英語 | リリース・記事は翻訳で追う。**意思決定に効く一行**を優先し、細部は専門部署に委ねる |

---

## 3. 週次の現実（時間・成果物）

- **海外IT情報に使える時間:** 週あたりおおむね **60〜120分**（深読みは週1テーマ程度）。
- **欲しい成果物のイメージ:** 社内メモ1〜2枚、短いスライド、稟議の **根拠リンク集**。
- **「今週の成功」:**  
  - 上司向けに **30秒で So what** が言える。  
  - **公式が言っている／いない**が分かれている。  
  - 情シス・法務に **具体の質問**が投げられる（「全部確認してください」で終わらない）。

---

## 4. 記事に求めるもの（優先順）

1. **So what（1〜3行）** — 自社の選択肢・制約・タイミングにどう効くか。  
2. **一次情報の位置づけ** — 公式／当局／大手報道のどれを主にしたか。  
3. **不確実性** — 推測と事実の区切り。  
4. **社内アクション** — 情シス・法務・調達への **質問例**または **確認チェック**（箇条書きで可）。  
5. **日本文脈（短く）** — 多くても **3点**（調達・法規・競合など）。無理に国内ワードを盛らない。

---

## 5. 離脱・信用を落とすパターン

- 煽り見出し＋ **出所のない数字**、ブログのみ根拠。  
- CVE・バージョンの羅列が主で、**誰が何を決める話か**がない。  
- 「革命／破壊」など感情語だけで、**提供開始・対象顧客・契約・価格**がない。

---

## 6. 文体・トーン

- **冷静・箇条書き・比較表歓迎。**  
- 記事先頭に **「この記事で決まること」1行**があるとミスマッチが減る。

---

## 7. 題材の寄せ方（迷ったとき）

| 題材の塊 | このペルソナ向けの切り口 |
|----------|---------------------------|
| 大手AI・クラウドの発表 | ロードマップ、提供形態、**契約・ガバナンス**、競合比較 |
| M&A・大型調達 | 誰が顧客・GPU・パートナーを取りに行っているか、**我が社の交渉材料** |
| 標準化・財団・決済プロトコル等 | 中長期の **依存先・ロックイン／脱却** |
| 半導体・インフラマクロ | PoCから本番スケールまでの **供給・コスト感** |
| 規制・競争政策（DMA 等） | プロダクト設計・データ・第三者連携の **長期制約** |
| セキュリティ速報 | PMは深追いしない。**情シス依頼の1ブロック**に圧縮 |

**車載・交通安全（NTSB／NHTSA 等）**は読者期待がズレやすい。**書く場合はタイトルと冒頭で層を明示**するか、一般化して「Level2・データ記録・規制トレンド」に寄せる。

## 8. 改定ルール

- ターゲットを変える場合は **本ファイルを更新し、日付を冒頭に追記**する。  
- 「業種・KPI を具体化したい」場合は、**別紙**や note 固定記事に書き、本資料は **役割ベースの共通項のみ**に留める。
